# 03 — 长期记忆系统

> **1 分钟叙事**：Agent 每次会话都是「无状态」的——上下文窗口一关，上一轮聊过什么就丢了。Memoh 的长期记忆系统要解决的是「跨会话的连续认知」：把用户偏好、身份事实、计划意图沉淀下来，下次对话时悄悄注入。它的做法不是单一向量库，而是**多 Provider 混合检索**——用 LLM 从对话里抽取结构化事实（Extract），用 Mem0 风格的 ADD/UPDATE/DELETE 决策去重合并（Decide），再用 Qdrant 稀疏向量（BM25 关键词）或稠密向量（embedding 语义）做检索，最后用带「防中段遗忘」重排的打包器塞进有限 token 预算。底层还有一层关键设计：**文件是真相源，Qdrant 只是派生索引**，随时可从 markdown 重建。这让记忆既人类可读可编辑，又能享受向量检索的速度。

## 核心结论

1. **统一 `Provider` 接口**（`internal/memory/adapters/provider.go:14-42`）抽象三种实现：`builtin`（自研，三模式）、`mem0`（SaaS 委托）、`openviking`（自托管/SaaS 委托）。Bot 通过 `bots.memory_provider_id` 外键挂一个 provider。

2. **builtin 三模式**（`internal/memory/adapters/builtin/factory.go:17-21`）：`off`（纯文件，无检索）、`sparse`（Qdrant 稀疏向量 + Python BM25 编码服务）、`dense`（Qdrant 稠密向量 + embedding API）。由 `memory_providers.config.memory_mode` 决定，渐进式能力——桌面端单机可只开 off，生产环境开 sparse/dense。

3. **文件是 source of truth，Qdrant 是 derived index**。所有记忆条目以 canonical markdown（`## Entry mem_xxx` + YAML frontmatter）存在 bot 工作空间的 `memory/` 目录；Qdrant 只是检索加速层，`Rebuild()` 可从文件全量重建（`sparse_runtime.go:402`、`dense_runtime.go:490`）。

4. **记忆形成走 LLM 管线**（`builtin/formation.go:34`）：`Extract`（抽事实）→ `gatherCandidates`（查候选）→ `Decide`（决策 ADD/UPDATE/DELETE/NOOP）→ `applyActions`（执行 CRUD）。借鉴 Mem0 的更新决策模式。

5. **记忆注入走 `OnBeforeChat` hook**（`builtin/builtin.go:172`）：overfetch（默认 3x）→ 去重排序 → `packContext` 贪心打包（字符预算 + 防中段遗忘重排）→ 包成 `<memory-context>` 注入 user message。

6. **配置存 `memory_providers` 表**（`db/postgres/migrations/0001_init.up.sql:148`、`0020_memory_providers.up.sql:5`），`config` 是 JSONB，承载 `memory_mode`/`embedding_model_id`/`qdrant_collection`/`context_target_items` 等。运行时用 `Registry` 缓存实例（`registry.go`）。

## 多 provider 架构（Qdrant/BM25/LLM 抽取）

### Provider 接口

`internal/memory/adapters/provider.go:14-42` 定义了统一的 `Provider` 接口，分四组方法：

- **会话钩子**：`OnBeforeChat`（检索注入）、`OnAfterChat`（写入抽取）
- **MCP 工具**：`ListTools`/`CallTool`——只暴露 `search_memory` 一个工具给 Agent（`provider.go:9`）
- **CRUD**：`Add`/`Search`/`GetAll`/`Update`/`Delete`/`DeleteBatch`/`DeleteAll`
- **生命周期**：`Compact`（压缩）、`Usage`（用量统计）

另有 `SourceSyncProvider`（`Status`/`Rebuild`）和 `SemanticCompactProvider`（能力声明）两个扩展接口（`provider.go:46-56`）。

### 三种 ProviderType

`internal/memory/adapters/types.go:251-255`：

| Provider | 实现位置 | 定位 |
|---|---|---|
| `builtin` | `adapters/builtin/` | 自研，文件 + Qdrant，三模式可切 |
| `mem0` | `adapters/mem0/` | 委托 Mem0 SaaS API，本地仍用 storefs 文件做镜像 |
| `openviking` | `adapters/openviking/` | 委托 OpenViking API（自托管或 SaaS） |

`adapters/service.go:39-136` 的 `ListMeta` 暴露每个 provider 的配置 schema：builtin 有 `memory_mode`/`embedding_model_id`/`qdrant_collection`/`context_target_items`/`context_max_total_chars`；mem0 有 `base_url`/`api_key`/`org_id`/`project_id`；openviking 有 `base_url`/`api_key`。

### Registry：懒加载 + 缓存驱逐

`adapters/registry.go` 用 `Factory`（`:13`）模式按 providerType 创建实例，`instances map[string]Provider` 缓存（`:20`）。关键行为：

- `Instantiate`（`:67`）：双重检查锁，已有直接返回，否则调 factory 创建并缓存
- `Remove`（`:99`）：配置更新或删除时驱逐缓存，下次 `Get` 重新创建
- `adapters/service.go` 的 `tryEvictAndReinstantiate`（`:346`）在 `Update` 后强制驱逐重建，保证配置热生效

### builtin 三模式分支

`adapters/builtin/factory.go:28-59` 的 `NewBuiltinRuntimeFromConfig` 按 `memory_mode` 分流：

- `ModeSparse`（`:32`）：建 `sparseRuntime`，依赖 Qdrant + Python sparse 编码服务（`cfg.Sparse.BaseURL`）
- `ModeDense`（`:53`）：建 `denseRuntime`，依赖 Qdrant + embedding 模型（`embedding_model_id`）
- 默认/off（`:57`）：建 `FileRuntime`，只写文件不检索

`factory.go:24-27` 的注释点明设计意图：「off 或未知模式返回 file runtime；sparse/dense 显式请求但初始化失败则报错，让调用方暴露配置问题而非静默降级」。

### mem0 / openviking：外部委托

`adapters/mem0/mem0.go` 的 `Mem0Provider` 把 CRUD/搜索委托给 Mem0 SaaS API，但本地仍持 `storefs.Service` 做文件镜像（带 `source_entry_id`/`source_hash` 同步元数据，`mem0.go:20-24`）。`openviking/openviking.go` 类似但无本地文件层。这两个是「不想自建索引就接 SaaS」的逃生通道。

## 写入记忆链路

入口在对话流 resolver：`internal/conversation/flow/resolver_memory.go:102` 的 `storeMemory`。

```
ChatRequest 完成
  → resolver.storeMemory (resolver_memory.go:102)
  → hooks.EventBeforeMemoryWrite（可 Deny）
  → provider.OnAfterChat (builtin.go:227)
      ├─ 有 LLM：runFormation (formation.go:34)
      │   ├─ llm.Extract          (memllm/client.go:54)  ← memory_extract.md
      │   ├─ gatherCandidates     (formation.go:80)
      │   │   ├─ 对每个 fact 调 runtime.Search
      │   │   └─ 不够再 runtime.GetAll 补
      │   ├─ llm.Decide           (memllm/client.go:109) ← memory_update.md
      │   └─ applyActions         (formation.go:160)
      │       ├─ ADD   → runtime.Add
      │       ├─ UPDATE→ runtime.Update
      │       ├─ DELETE→ runtime.Delete
      │       └─ NOOP  → skip
      └─ 无 LLM：直接 service.Add 存原始消息（legacy 旁路）
  → hooks.EventMemoryExtracted / EventAfterMemoryWrite
```

### formation 管线细节

`builtin/formation.go:34-77` 的 `runFormation`，60s 超时（`:13`）：

1. **Extract**（`:41`）：把 `req.Messages` 拼成 `User: ... Assistant: ...` 转录文本，调 `memllm.Client.Extract`（`memllm/client.go:54-107`）。LLM 用 `memory_extract.md` 作 system prompt，注入 `{{today}}`，返回 `{"facts":[...]}`。最多保留 `maxExtractFacts=10` 条（`memllm/client.go:18`）。
2. **gatherCandidates**（`formation.go:80-157`）：对每个 fact 调 `runtime.Search`（limit = `20/len(facts)`，`:97`），去重；不够 `maxCandidatesPerDecide=30`（`:16`）再用 `GetAll`（limit 50）补。目的是给 Decide 提供已有记忆上下文，避免重复存储。
3. **Decide**（`:58`）：调 `memllm.Client.Decide`（`memllm/client.go:109-138`），用 `memory_update.md` 作 system，user message 按 Mem0 约定拼「当前记忆 + 新事实」（`:174-222`），返回 `{"memory":[{id,text,event,old_memory}]}`，event 归一化为 ADD/UPDATE/DELETE/NOOP（`:258-286`，NONE→NOOP）。最多 `maxDecideActions=20`（`:19`）。
4. **applyActions**（`formation.go:160-233`）：按 event 分发，对 UPDATE/DELETE 做幂等去重（`updated`/`deleted` map，`:161-162`），空 ID/空文本跳过计入 `Skipped`。

### runtime.Add：双写文件 + 索引

以 sparse 为例（`sparse_runtime.go:79-106`）：

1. 构造 `MemoryItem`（ID = `botID:mem_<nanos>`，Hash = sha256，`:90-98`）
2. `store.PersistMemories`（`:99`）→ 写 markdown 日文件 + 同步 `MEMORY.md` overview
3. `upsertSourceItems`（`:102` → `:492`）→ 批量 `encoder.EncodeDocuments` → 逐条 `qdrant.Upsert`（带 `bot_id` payload 过滤）

dense 完全对称（`dense_runtime.go:178-204`、`:575-610`），只是编码换成 `embedDocuments`（调 Twilight SDK `client.EmbedMany`，`dense_runtime.go:138-152`），向量用 `UpsertDense`。

关键点：**先写文件再写索引**。若索引写失败，文件已是真相；下次 `Rebuild` 会补齐。

## 检索记忆链路

入口：`resolver_memory.go:36` 的 `loadMemoryContextMessage`。

```
ChatRequest 进来
  → resolver.loadMemoryContextMessage (resolver_memory.go:36)
  → hooks.EventBeforeMemorySearch（可 Deny）
  → provider.OnBeforeChat (builtin.go:172)
      ├─ overfetchLimit = TargetItems * OverfetchRatio（默认 6*3=18）
      ├─ runtime.Search
      │   sparse: encoder.EncodeQuery → qdrant.Search（sparse vec + bot_id filter）
      │   dense:  embedQuery         → qdrant.SearchDense
      ├─ deduplicateAndSort（按 score 降序）
      └─ packContext (context_packer.go:47)
          ├─ Phase 1: 贪心装入（每条上限 MaxItemChars=360）
          ├─ Phase 2: 不足 TargetItems 则压到 MinItemChars=80 腾位
          ├─ Phase 3: 剩余预算回填给已压缩项
          └─ Phase 4: antiLostInMiddle 重排（best 放头尾）
  → 包成 <memory-context>...</memory-context> user message
  → hooks.EventAfterMemorySearch
```

### OnBeforeChat 实现

`builtin/builtin.go:172-225`：

- 空 query 或空 botID 直接返回 nil（`:176`）
- `fetchLimit = overfetchLimit(p.packer)`（`:180`，默认 18）
- `service.Search` 带 `namespace=bot`/`scopeId=botID`/`bot_id` 过滤，`NoStats=true`（`:181-191`）
- 失败只 Warn 不阻断（`:193`）
- 命中后 `packContext` 打包，每条加 `[显示名, 日期]` 来源标签（`:209-217`，`memorySourceLabel` 在 `:148`）
- 输出 `<memory-context>\nRelevant memory context (use when helpful):\n- ...\n</memory-context>`（`:208-219`）

### packContext：四阶段预算管理

`builtin/context_packer.go:47-160`，默认配置（`:20-26`）：TargetItems=6、MaxTotalChars=1800、MinItemChars=80、MaxItemChars=360、OverfetchRatio=3。

核心思路是「先贪心、再压缩、再回填」：

- **Phase 1**（`:83-99`）：按 score 顺序贪心装入，每条用自然长度（上限 MaxItemChars）。装不下且剩余预算不够 MinItemChars 就跳过；够 MinItemChars 就压缩到 MinItemChars。
- **Phase 2**（`:103-126`）：若没装满 TargetItems，把已选的压到 MinItemChars，腾出预算再装后面的。
- **Phase 3**（`:129-148`）：剩余预算按均分回填给压缩过的项，最多回到 MaxItemChars 或原文长度。
- **Phase 4**（`:152`）：`antiLostInMiddle`（`:165-182`）把最高分放位置 0，次高放末尾，再依次头尾交替——对抗 LLM 对长上下文中段注意力下降的现象。

### Qdrant 检索细节

`internal/memory/qdrant/client.go`：

- sparse 检索（`:180-200`）：`pb.NewQuerySparse(indices, values)` + `Using="sparse"` + `botFilter(botID)`（`:332-338`，`Must: match bot_id`）
- dense 检索（`:203-222`）：`pb.NewQueryDense(vec.Values)` + `botFilter`
- 都带 `WithPayload(true)` 取回 payload（含 memory 文本、source_entry_id 等）

### search_memory MCP 工具

Agent 主动检索走 `builtin/builtin.go:298-392` 的 `CallTool`：

- 工具名 `search_memory`（`adapters/provider.go:9`）
- limit 默认 8，上限 50（`builtin.go:20-22`）
- 多了 chat 参与者鉴权（`:332-353`，`canAccessChat`）：非 bot 本人调用要校验 chat 归属和参与者身份
- 结果按 score 降序，只返回 `id`/`memory`/`score`（`:378-385`）

## 记忆 prompt（extract/update/注入）

三份 prompt 各司其职，注意**实际路径**与学习计划文档略有出入：

### memory_extract.md（抽取事实）

`internal/memory/memllm/prompts/memory_extract.md`（学习计划写的 `internal/agent/prompts/memory_extract.md` 实际不存在，文件在 memllm 下）。

定位：「Personal Information Organizer」，从对话抽取 7 类信息：个人偏好、重要细节、计划意图、活动服务偏好、健康偏好、职业细节、杂项。给 6 个 few-shot 例子（空对话→空 facts、找餐厅→"Looking for a restaurant in San Francisco" 等）。关键规则：

- `Today's date is {{today}}`——`memllm/client.go:86` 用当前时区日期替换
- 只从 user/assistant 消息抽，不碰 system 消息
- 检测用户语言并用同语言记录
- 输出 `{"facts": [...]}`

### memory_update.md（决策更新）

`internal/memory/memllm/prompts/memory_update.md`。

定位：「smart memory manager」，四操作决策：

- **ADD**：新信息，生成新 ID
- **UPDATE**：信息已存在但内容变化（如 "likes cricket" → "loves cricket with friends"），保持原 ID
- **DELETE**：新事实与旧记忆矛盾（如 "dislikes cheese pizza" 否定 "loves cheese pizza"）
- **NONE**：已存在或无关

prompt 给了详尽的对比示例（cricket、pizza 等），强调「保持信息量更大的那个」「更新时 ID 不变」。

`memllm/client.go:174-222` 的 `buildUpdateUserMessage` 严格按 Mem0 约定拼 user message：当前记忆放 triple backtick JSON，新事实放 triple backtick JSON 数组，再附 JSON 输出格式要求。

### _memory.md（Agent 侧写入规则）

`internal/agent/prompts/_memory.md`——这是给 **Agent 本身**看的（注入 system prompt），不是给记忆 LLM 用的。告诉 Agent：

- 每天 `memory/YYYY-MM-DD.md` 是原始日志，`MEMORY.md` 是策展后的长期记忆
- canonical entry 结构：`## Entry mem_20260313_001` + YAML frontmatter（id/created_at/updated_at/metadata）
- 只发新条目，不重写旧内容；不提供 hash（后端生成）
- 涉及已知用户/群组时带 `profile_ref` 链接

### compactSystemPrompt（压缩）

`memllm/client.go:332-346` 内联常量，不是独立 md 文件。压缩 assistant 的目标：按 `target_count` 聚类合并、保留安全关键事实、用 recency 解冲突、丢重复和瞬态细节、`decay_days` 作优先级信号。输出纯 JSON 数组。

## 数据模型（memory_providers 表）

`db/postgres/migrations/0001_init.up.sql:148-157`（`0020_memory_providers.up.sql:5-14` 完全一致，`0025` 是修复重放）：

```sql
CREATE TABLE IF NOT EXISTS memory_providers (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  provider TEXT NOT NULL,              -- builtin / mem0 / openviking
  config JSONB NOT NULL DEFAULT '{}'::jsonb,
  is_default BOOLEAN NOT NULL DEFAULT false,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  CONSTRAINT memory_providers_name_unique UNIQUE (name)
);
ALTER TABLE bots ADD COLUMN IF NOT EXISTS
  memory_provider_id UUID REFERENCES memory_providers(id) ON DELETE SET NULL;
```

关键设计：

- **config 是 JSONB**：不同 provider 放不同字段（builtin 放 `memory_mode`/`embedding_model_id` 等，mem0 放 `base_url`/`api_key` 等）。`adapters/service.go:318-334` 的 `toGetResponse` 反序列化成 `map[string]any` 透传。
- **bots.memory_provider_id ON DELETE SET NULL**：删 provider 不删 bot，bot 退化为无记忆。
- **0020 迁移**（`0020_memory_providers.up.sql:21-46`）：把旧 `bots.memory_model_id`/`embedding_model_id` 迁进默认 builtin provider 的 config，然后 drop 旧列。用动态 SQL（`EXECUTE`）加 `information_schema` 守卫，兼容全新安装（没有旧列）。
- **唯一 name 约束**：`EnsureDefault`（`service.go:300-316`）创建名为 "Built-in Memory" 的默认 provider。

**注意**：记忆条目本身**不存数据库表**。它们以 canonical markdown 存在 bot 工作空间文件系统（`storefs`），向量存 Qdrant。`memory_providers` 表只存 provider 配置，不存记忆数据。这是「文件是真相源」设计的关键体现。

sqlc 生成的查询在 `internal/db/postgres/sqlc/memory_providers.sql.go`（以及 sqlite 双轨 `internal/db/sqlite/sqlc/memory_providers.sql.go`），含 `CreateMemoryProvider`/`GetDefaultMemoryProvider`/`GetMemoryProviderByID`/`ListMemoryProviders`/`UpdateMemoryProvider`/`DeleteMemoryProvider`。

## 设计动机与取舍

1. **文件作 source of truth，Qdrant 作 derived**：人类可读、可手动编辑、可版本控制；索引坏了能从文件全量 `Rebuild`（`sparse_runtime.go:402`、`dense_runtime.go:490`）。代价是写入要双写（文件 + 索引），一致性靠 `Rebuild` 兜底。`syncSourceItems`（`sparse_runtime.go:415`）对比文件与 Qdrant 的 `source_entry_id` payload，补缺失、删陈旧。

2. **三模式而非一模式**：`off` 给最小依赖部署（桌面端单机，无 Qdrant 无编码服务）；`sparse` 给关键词检索场景（无需 embedding API，但要 Python BM25 服务）；`dense` 给语义检索场景（要 embedding API）。`factory.go:24-27` 注释明确：显式请求 sparse/dense 但初始化失败要报错，不静默降级到 off。

3. **LLM 抽取而非全文存储**：把对话压成 facts（`maxExtractFacts=10`），避免记忆爆炸；代价是依赖 LLM 调用，有 `formationTimeout=60s`（`formation.go:13`）。无 LLM 时走 legacy 旁路直接存原始消息（`builtin.go:252-267`）。

4. **借鉴 Mem0 的 Decide 模式**：ADD/UPDATE/DELETE/NOOP 四态决策避免重复记忆堆积。`Mem0Provider` 直接对接 SaaS，`builtin` 是自研等价实现——两套实现同一套语义，用户可按合规/成本选择。

5. **anti-lost-in-the-middle 重排**（`context_packer.go:165-182`）：LLM 对长上下文中段注意力低，把高分记忆放头尾。这是生产级 RAG 的细节功夫。

6. **Overfetch + 贪心打包**：先取 3x 候选（`overfetchLimit`，`context_packer.go:186`），再按 1800 字符预算裁剪到 6 条。平衡召回与 token 成本，且 Phase 2/3 动态压缩回填保证预算用满。

7. **Compact 语义压缩而非删除**（`compactplan/compactplan.go`）：`pinned`/`read_only` 标记的保护项不压缩（`:125-127`）；可压缩项送 LLM 合并，结果写入 active，旧项归档（`archiveItems`，`:264`）而非删除，保留 `superseded_by` 审计链。超过 `MaxCandidateChars=24000`（`:17`）自动分批 + 递归 reduce（`:169-197`）。

8. **dense runtime 的 enable 热检查**（`dense_runtime.go:101-122`）：每次 `Embed` 重新查 `models` 表的 `Enable` 字段，防止缓存 runtime 泄漏已禁用的 embedding 模型。注释（`:98-100`）点明：「cached runtimes outlive the model row in the registry, so the construction-time check alone leaks disabled models until eviction」。

9. **hooks 可干预记忆读写**（`resolver_memory.go:41`、`:116`）：`EventBeforeMemorySearch`/`EventBeforeMemoryWrite` 等 hook 可 `Deny` 记忆操作，给合规审计留口子。

## 验证问题

1. **一次「写入记忆」的完整链路**？从 `resolver.storeMemory` 讲到 `runtime.Add` 的双写（文件 + Qdrant），说清 formation 管线四步（Extract→gatherCandidates→Decide→applyActions）和 LLM 缺席时的 legacy 旁路。

2. **一次「检索记忆」的完整链路**？从 `resolver.loadMemoryContextMessage` 讲到 `<memory-context>` 注入，说清 overfetch、packContext 四阶段、antiLostInMiddle 重排，以及 sparse/dense 两种编码路径。

3. **为什么文件是 source of truth 而 Qdrant 是 derived？如何重建？** 答：人类可读可编辑 + 索引可重建。`Rebuild`（`sparse_runtime.go:402`/`dense_runtime.go:490`）调 `ReadAllMemoryFiles` 读全部 markdown → `syncSourceItems` 对比 Qdrant 既有 payload → 补缺失、删陈旧、更新不一致。

4. **builtin 三模式 off/sparse/dense 各解决什么问题？依赖什么外部服务？** 答：off 零依赖（仅文件）、sparse 依赖 Qdrant + Python BM25 编码服务（`cfg.Sparse.BaseURL`）、dense 依赖 Qdrant + embedding 模型 API（`embedding_model_id`）。切换只改 `memory_providers.config.memory_mode` 并驱逐 registry 缓存。

5. **memory_providers 表的 config JSONB 里能放什么？三模式如何切换？** 答：builtin 放 `memory_mode`/`embedding_model_id`/`qdrant_collection`/`context_target_items`/`context_max_total_chars`（`service.go:44-79`）；mem0 放 `base_url`/`api_key`/`org_id`/`project_id`；openviking 放 `base_url`/`api_key`。切换靠 `Update`（`service.go:251`）→ `tryEvictAndReinstantiate`（`:346`）驱逐 registry 缓存，下次 `Get` 用新 config 重建 runtime。
