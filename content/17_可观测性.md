# 17 — 可观测性

> 一个跑在生产里的 Agent 平台，出了问题怎么定位？Memoh 的回答分三层，但**重量分布很不均匀**：日志层（slog 结构化日志）铺满全代码库（270 个文件），是绝对主力；健康检查层（`/ping` + per-bot runtime checks）做存活与依赖探活；资源指标层（containerd cgroup 采样）只覆盖容器后端。**没有 OpenTelemetry、没有 Prometheus `/metrics`、没有跨服务分布式 trace、没有 request_id 中间件级传播**——这不是疏漏，是它的部署形态决定的：单进程 Go server + 本地桌面端，进程内调用为主，外部依赖（LLM/Qdrant/PG）的「调用链」就是日志里的 `service`/`bot_id`/`session_id` 字段。这一篇拆清楚：日志怎么打、健康检查怎么分层、指标怎么采、以及一个 Agent 系统特有的「业务级 trace」——hook 事件流。

## 核心结论

1. **日志是唯一的全局观测手段**。标准库 `log/slog`，全局 `logger.L`，FX 在启动时 `logger.Init(level, format)`（`cmd/agent/app.go:124`）。约定每个 service 构造时 `log.With(slog.String("service", "xxx"))` 打组件标签，请求级靠 echo `RequestLogger` 中间件记 method/uri/status/latency/remote_ip（`internal/server/server.go:47`）。
2. **健康检查分两级**：进程级 `/ping` + `/health`（HEAD）返回 container backend / version / commit / 能力位（`internal/handlers/ping.go:30`）；Bot 级 `/api/users/bots/:id/checks` 聚合「内置容器检查 + 动态 RuntimeChecker」（`internal/bots/service.go:510`）。
3. **RuntimeChecker 是可插拔的依赖探活框架**。三类 checker：`model`（并发探活 chat/memory/embedding 三个模型，`internal/healthcheck/checkers/model/checker.go:116`）、`channel`（渠道连接状态）、`mcp`（MCP 连接 + 工具列举）。状态机 `ok/warn/error/unknown`，带 i18n `titleKey`，detail 按 `PermissionManage` 脱敏（`internal/handlers/users.go:806`）。
4. **资源指标只存在于 containerd 后端**。`internal/container/containerd/metrics.go:35` 的 `GetContainerMetrics` 两次采样（间隔 200ms）算 CPU 使用率，兼容 cgroup v1/v2，内存取 usage/limit。HTTP 暴露在 `/bots/:id/container/metrics`（`internal/handlers/containerd.go:273`）。Docker/Apple 后端无指标。
5. **Hook 事件流是 Agent 系统特有的「业务 trace」**。`internal/hooks/types.go:18` 定义 28 个事件（PreToolUse/PostToolUse/ToolError/SessionStart/TurnEnd/TurnError/BeforeMemorySearch/AfterMemoryWrite/ChannelDeliveryFailed…），`internal/agent/hooks.go:54` 用 hook 包裹每个工具执行。它既是插件扩展点，也是一条贯穿「会话→工具→记忆→渠道投递」的事件链——只是没有聚合后端，落地为 hook 工具的副作用 + 日志。
6. **SSE 是 Agent 事件的实时出口**。`internal/agentpayload` 定义事件 wire 形状，`session_id` 是路由的 load-bearing 字段（`internal/agentpayload/payload.go:11` 注释明示），`internal/handlers/message_stream.go` 按 `session_id` 把后台任务事件转发给 per-session SSE 订阅者。
7. **诚实标注缺失**：无 OTel trace、无 Prometheus metrics endpoint、无 request_id 全链路传播（`logger.FromContext` 存在但全代码库 0 调用，`request_id` 仅在 `userinput`/`native_source` 局部使用）。错误恢复靠 echo `middleware.Recover()`（`server.go:35`）+ agent 层的 retry/sential（见 [[01_agent_内核]]）。

## 日志层：slog 全家桶

### 初始化与配置

`internal/logger/logger.go:18` 的 `Init(level, format)` 决定全局 logger：

```go
opts := &slog.HandlerOptions{Level: parseLevel(level)}
if strings.ToLower(format) == "json" {
    handler = slog.NewJSONHandler(os.Stdout, opts)
} else {
    handler = slog.NewTextHandler(os.Stdout, opts)
}
L = slog.New(handler)
slog.SetDefault(L)
```

配置来自 `config.LogConfig`（`internal/config/config.go:112`）：

```go
type LogConfig struct {
    Level  string `toml:"level"`   // debug/info/warn/error
    Format string `toml:"format"`  // text/json
}
```

FX 装配在 `cmd/agent/app.go:124` 的 `provideLogger`——先 `logger.Init`，再把 `logger.L` 作为 `*slog.Logger` 注入依赖图。所以**所有 service 拿到的是同一个全局 logger 的派生实例**。

### 组件标签约定

代码库统一用法：service 构造时派生带 `service` 字段的子 logger。几乎每个 service 都这样：

```go
// internal/channelaccess/service.go:69
logger: log.With(slog.String("service", "channelaccess"))
// internal/bots/service.go:53
logger: log.With(slog.String("service", "bots"))
// internal/schedule/service.go:57
logger: log.With(slog.String("service", "schedule"))
```

checker 则用 `checker` 标签（`internal/healthcheck/checkers/model/checker.go:54`：`log.With(slog.String("checker", "healthcheck_model"))`）。这是 Memoh 的「伪 trace」——没有 span，但靠 `service`/`bot_id`/`session_id` 字段能在日志里拼出一条调用链。

### 请求级日志

`internal/server/server.go:47` 的 echo `RequestLogger` 中间件对每个 HTTP 请求打一条：

```go
log.Info("request",
    slog.String("method", v.Method),
    slog.String("uri", safeRequestLogURI(c.Request().URL, v.URI)),
    slog.Int("status", v.Status),
    slog.Duration("latency", v.Latency),
    slog.String("remote_ip", c.RealIP()),
)
```

注意 `safeRequestLogURI`——对 URI 做了脱敏（避免把 token/敏感 query 打出去）。`middleware.Recover()`（`server.go:35`）兜底 panic，防止一个请求炸垮进程。

### 没用起来的上下文传播

`logger.go:35` 提供了 `FromContext`/`WithContext`，可以把派生 logger 塞进 context 流传。但实际全代码库 `logger.FromContext` 调用数 **0**——这个机制基本闲置。`request_id` 也只在 `internal/agent/tools/native_source.go:376`（`slog.String("request_id", req.ID)`）和 `userinput` 局部出现，**没有中间件统一注入**。结论：Memoh 的日志关联靠业务字段（bot_id/session_id），不靠通用 trace 字段。这是一个明确的设计取舍——单进程够用，跨进程场景（容器内 bridge）另算。

## 健康检查层：进程级 + Bot 级

### 进程级：`/ping` 与 `/health`

`internal/handlers/ping.go:30` 暴露两个端点：

```go
e.GET("/ping", h.Ping)
e.HEAD("/health", h.PingHead)
```

`PingResponse` 返回的不仅是「活着」，还有**能力位**：

```go
type PingResponse struct {
    Status                string `json:"status"`
    ContainerBackend      string `json:"container_backend"`
    LocalWorkspaceEnabled bool   `json:"local_workspace_enabled"`
    SnapshotSupported     bool   `json:"snapshot_supported"`
    Version               string `json:"version"`
    CommitHash            string `json:"commit_hash"`
}
```

`/health`（HEAD）只返回 200 空体，给 K8s/负载均衡的存活探针用；`/ping`（GET）带 JSON 体，给人看。`server.go:88` 把 `/ping`、`/health` 列入 JWT 跳过名单——探针不带 token。

### Bot 级：runtime checks 聚合

这是 Memoh 健康检查最有设计感的部分。一个 Bot 是「容器 + 模型 + 渠道 + MCP」的组合，任何一个挂了 Bot 就不可用。`/api/users/bots/:id/checks`（`internal/handlers/users.go:103`）返回这个组合的体检报告。

调用链：

```
GET /bots/:id/checks
  → bots.Service.ListChecks (service.go:510)
  → buildRuntimeChecks (service.go:743)
      ├─ 内置容器检查：container.init / container.record / container.task / container.data_path
      └─ appendDynamicChecks (service.go:949)
            └─ 遍历 s.checkers []RuntimeChecker → 各自 ListChecks
```

`RuntimeChecker` 接口（`internal/bots/types.go:92`）：

```go
type RuntimeChecker interface {
    ListChecks(ctx context.Context, botID string) []BotCheck
}
```

`bots.Service.AddRuntimeChecker`（`service.go:69`）让外部包注册 checker，`healthcheck.NewRuntimeCheckerAdapter`（`internal/healthcheck/adapter.go:18`）把 `healthcheck.Checker` 适配成 `bots.RuntimeChecker`——这是把「healthcheck 子系统」接进「bots 子系统」的桥。

### 三类动态 checker

| Checker | 检查内容 | 关键实现 |
|---|---|---|
| `model` | chat/memory/embedding 三个模型各自探活 | `checkers/model/checker.go:116` 并发 `wg`，每 slot 走 `prober.Test`，区分 `TestStatusOK`/`TestStatusAuthError` |
| `channel` | Bot 绑定的渠道连接状态 | `checkers/channel/checker.go` 读 `ConnectionObserver.ConnectionStatusesByBot` |
| `mcp` | Bot 的 MCP 连接 + 每连接工具列举 | `checkers/mcp/checker.go`，8s 超时 |

每个 `CheckResult` 带 `Status`（ok/warn/error/unknown）、`Summary`、`Detail`、`Metadata`（如 `latency_ms`、`model_id`）。`TitleKey`（如 `bots.checks.titles.modelConnection`）是 i18n key，前端按用户语言渲染标题。

### 脱敏：detail 按权限裁剪

`internal/handlers/users.go:806`：

```go
includeDetails := bots.HasPermission(perms, bots.PermissionManage)
return c.JSON(http.StatusOK, bots.ListChecksResponse{
    Items: scrubBotChecksForResponse(items, includeDetails),
})
```

只有有 `PermissionManage` 的用户能看到 check 的 `Detail`（可能含模型报错原文、连接错误栈）。普通用户只看到 `Summary`。这是「可观测性 vs 信息泄露」的边界处理——健康检查天然会暴露内部错误，必须按权限裁剪。

## 资源指标层：containerd 专属

### 双采样算 CPU

`internal/container/containerd/metrics.go:35` 的 `GetContainerMetrics`：

```go
first, err := sampleTaskMetrics(ctx, task)
// 等 200ms
timer := time.NewTimer(metricsSampleInterval)  // 200ms
second, err := sampleTaskMetrics(ctx, task)
return ContainerMetrics{
    CPU:    buildCPUMetrics(first, second),
    Memory: buildMemoryMetrics(second),
}
```

CPU 使用率 = `(second.cpuUsageNS - first.cpuUsageNS) / elapsedNS * 100`（`buildCPUMetrics`，`metrics.go:130`）。单次采样只有累计值，必须两次采样做差才算出「即时使用率」——这是 cgroup CPU 统计的本质。

### cgroup v1/v2 兼容

`sampleTaskMetrics`（`metrics.go:65`）按 `metric.Data.MessageIs(...)` 分派：

- cgroup v1：`stats.GetCPU().GetUsage().GetTotal()` 等（`sampleFromCgroup1`）
- cgroup v2：`cpu.GetUsageUsec() * 1_000` 等（`sampleFromCgroup2`，v2 用 μs，乘 1000 转 ns 对齐）

内存 `normalizeMemoryLimit`（`metrics.go:160`）把「>1<<60 的 limit」当成「无限制」归零——避免 Linux 内核默认大 limit 把使用率算成 0%。

### 暴露与缺失

HTTP 端点 `/bots/:id/container/metrics`（GET，`internal/handlers/containerd.go:273`；PUT `:274` 用于设置配额）。响应含 CPU/Memory/Storage 三块（`GetContainerMetricsResponse`，`containerd.go:155`）。

**缺口**：这套指标只在 containerd 后端实现（`GetContainerMetrics` 挂在 `DefaultService` 上）。Docker 后端、Apple Virtualization 后端没有等价实现——切后端就丢了资源观测。也没有 Prometheus `/metrics` 端点，指标是按需拉取的 JSON，不进时序库。要长期趋势得自己外挂采集。

## Hook 事件流：Agent 系统的业务级 trace

### 28 个事件贯穿全链路

`internal/hooks/types.go:18` 定义的事件覆盖一条 Agent 对话的几乎每个关键节点：

```
会话级：  SessionStart / TurnEnd / TurnError
Prompt：  BeforePromptBuild / AfterPromptBuild
模型：    BeforeModelCall / AfterModelCall
工具：    PreToolUse / PostToolUse / ToolError
记忆：    BeforeMemorySearch / AfterMemorySearch / BeforeMemoryWrite / AfterMemoryWrite / MemoryExtracted
工作空间： WorkspaceStart / WorkspaceStop / BeforeWorkspaceCommand / AfterWorkspaceCommand
         BeforeFileWrite / AfterFileWrite
审批：    BeforeApprovalCreate / ApprovalRequested / ApprovalResolved / ApprovalTimeout
渠道：    InboundMessageNormalized / BeforeOutboundMessage / AfterOutboundMessage / ChannelDeliveryFailed
```

这条事件链本身就是一条 trace——从「入站消息归一化 → prompt 构建 → 模型调用 → 工具执行（带审批）→ 记忆读写 → 工作空间命令 → 出站消息 → 渠道投递」全程埋点。区别于 OTel 的只是：没有统一的 span 后端，事件落到「用户/插件配置的 hook 工具」+ 日志。

### 工具执行的 hook 包裹

`internal/agent/hooks.go:54` 的 `wrapToolsWithHooks` 把每个 tool 的 `Execute` 包一层：

```go
wrapped[i].Execute = func(execCtx *sdk.ToolExecContext, input any) (any, error) {
    output, execErr := originalExecute(execCtx, input)
    if execErr != nil {
        // 触发 EventToolError
        errReq := a.baseHookRequest(ctx, cfg, hooks.EventToolError)
        errReq.Tool = &hooks.ToolPayload{Name: toolName, ...}
        a.hookService.Run(execContext(ctx, execCtx), errReq, runner)
        return output, execErr
    }
    // 触发 EventPostToolUse
    postReq := a.baseHookRequest(ctx, cfg, hooks.EventPostToolUse)
    ...
}
```

`hookService.Run`（`internal/hooks/service.go`）加载 bot 的 hook 配置（用户配置 + 插件贡献），匹配事件，调用对应 hook 工具。所以「事件」不只是观测，**还能触发副作用**（如审批拦截、日志转发、自定义通知）。这是 Memoh 把「可观测性」和「扩展性」合并到同一套机制的设计——hook 既是 trace 埋点，也是插件入口。

### SSE：事件的实时出口

`internal/agentpayload` 包定义 agent 事件转给前端 SSE 的 wire 形状。`payload.go:11` 的注释点明一个 load-bearing 约定：

> The top-level `session_id` placement is load-bearing: internal/handlers/message_stream.go routes events by reading that key directly.

后台任务（如异步压缩、调度任务）产生的事件，经 `bgManager.SetEventFunc` 转发，`message_stream.go` 读 `session_id` 路由到对应 SSE 订阅者，再 stamp `type`/`bot_id` 发出去。所以前端能实时看到「工具在跑」「记忆在写」「任务完成」——这是用户侧的可观测性。

## 为什么没有 OTel/Prometheus

这是设计取舍，不是技术债：

1. **部署形态**：单进程 Go server（server 模式）或 Electron 内嵌本地 server（desktop 模式）。进程内调用为主，没有微服务网格，分布式 trace 收益低。
2. **外部依赖少且异构**：LLM（Twilight SDK）、Qdrant、PostgreSQL——各自的 SDK 有自己的日志/指标，Memoh 不强求统一。
3. **目标用户**：个人/小团队的多 Bot 平台，不是 SaaS。slog + 健康检查 + 桌面端 UI 展示的 runtime checks，够用。
4. **指标场景窄**：真正需要时序指标的是「容器资源」和「LLM token/延迟」，前者用 cgroup 采样按需拉取，后者记在 compaction log 和 model probe 的 `latency_ms` 里。

代价：跨 Bot 聚合、长期趋势、告警，都需要外挂。如果你要把它改成 SaaS 部署，第一件事就是补 Prometheus exporter + OTel SDK。

## 设计动机小结

| 层 | 解决什么 | 怎么做 | 代价 |
|---|---|---|---|
| 日志 | 「出事了能查」 | slog 全局 + `service` 标签 + echo 请求日志 | 无聚合，靠 grep |
| 健康检查 | 「现在能不能用」 | `/ping` 进程级 + per-bot runtime checks 聚合 | 动态 checker 只覆盖 model/channel/mcp |
| 资源指标 | 「容器吃了多少」 | containerd cgroup 双采样 | 只 containerd 后端，无时序 |
| Hook 事件 | 「Agent 这一轮干了啥」 | 28 事件 + hook 工具 + SSE | 无后端聚合，事件即副作用 |
| 错误恢复 | 「别炸」 | echo Recover + agent retry/sential | 依赖各层自己处理 |
