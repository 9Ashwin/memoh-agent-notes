# 07 — 容器 / 工作空间

> **1 分钟叙事**：Memoh 给每个 Bot 配一个隔离的「工作空间」——多数情况是一个容器（Docker / containerd / Apple Virtualization），桌面端可以是本地目录。容器里跑一个叫 `bridge` 的 Go 二进制，监听 Unix domain socket（`/run/memoh/bridge.sock`），对外暴露一组 gRPC 接口（文件读写、exec、tunnel、reverse HTTP）。服务端的 `workspace.Manager` 通过一个 gRPC 连接池（`bridge.Pool`）复用与各 Bot 容器的连接，**走 UDS 而非 TCP**——免认证、免端口暴露，靠文件系统 socket 权限做信任边界。多后端抽象（`factory` + `docker` / `containerd` / `apple` 适配器 + `RuntimeRouter`）解决的问题是「同一套 Manager 生命周期逻辑要跑在 Linux 服务器（containerd）、Mac（Apple Virtualization）和桌面端（纯本地 bufconn）三种运行时上」。display 开启时 bridge 还会监督 Xvnc + headed Chrome（CDP 9222），让 Agent 能驱动真实浏览器。

## 核心结论

1. **工作空间 = 容器（或本地目录）+ bridge 二进制 + UDS gRPC**。bridge 由 host 通过 `/opt/memoh` 只读 bind mount 注入，bot 数据住容器可写层（snapshot），三者解耦（`manager.go:328-342`）。
2. **`Manager` 是唯一调和入口**：DB 记录 ↔ 运行时状态对账，容器缺了重建、task 停了重启、启动时全量 `ReconcileContainers`（`manager_lifecycle.go:512-604`）。
3. **`bridge.Pool` 按 botID 缓存 gRPC 连接**，自动剔除 stale 连接（`Shutdown` / `TransientFailure` / `Connecting` 超 30s），`WaitForWorkspaceReady` 用 `Stat("/")` 探活（`client.go:684-719`、`manager.go:258-282`）。
4. **多后端靠接口分组 + 前缀路由**：`ContainerService` / `WorkloadService` / `NetworkService` / `SnapshotService` 四组能力（`service.go:64-103`），`RuntimeRouter` 按 `local-` 前缀或 `StorageRef.Driver=="local"` 路由到 `LocalService`（`runtime_router.go:42-54`）。
5. **资源配额按 backend 能力分级**：硬限制（CPU/内存）变更需要重建容器，因为 Docker/containerd 的 cgroup 限制在 create 时固化；local 与 Apple 不支持硬限制（`resource_limits.go:198-220`、`247-258`）。

## workspace manager：调和与 gRPC 连接池

### Manager 结构与连接池

`Manager`（`manager.go:86-101`）持有四样核心东西：

- `service runtimeService` —— 容器运行时抽象（见下节），是所有 create/start/stop/list 的实际执行者；
- `grpcPool *bridge.Pool` —— gRPC 连接池，`NewManager` 时用 `m.dialTarget` 函数构造（`manager.go:129`）；
- `containerLocks map[string]*sync.Mutex` —— 按 containerID 加锁，防止并发操作同一 bot 的容器（`manager.go:162-173`）；
- `legacyIPs map[string]string` —— 旧式 `mcp-` 容器的 TCP IP 缓存，兼容 bridge 架构前的部署。

### dialTarget：UDS 优先，TCP 仅给 legacy 与 Kata VM

`dialTarget`（`manager.go:194-207`）是连接池的灵魂——它决定「怎么连到 bot 的 bridge」：

1. 先问 `service` 是否实现 `BridgeTarget(string) string`（Docker 适配器实现了，返回 host:port）；`manager.go:195-199`
2. 再查 `legacyIPs`，命中则 `passthrough:///<ip>:9090`；`manager.go:200-205`
3. 否则走 `unix://` + `socketPath(botID)`。`manager.go:206`

默认就是 UDS。只有两种情况退化到 TCP：legacy 容器（`mcp-` 前缀，bridge 架构前用 TCP gRPC）和 Kata 运行时（UDS 不能跨 VM 边界，`manager.go:232-234`）。`usesTCPBridge` 就是这两个的并集。

### MCPClient 与探活

`MCPClient`（`manager.go:245-256`）先尝试 `service.(bridge.Provider).MCPClient`——这条路给 `LocalService` 用（直接返回 in-process client，不走网络）；不行再回退到 `grpcPool.Get`。

`WaitForWorkspaceReady`（`manager.go:258-282`）是创建容器后的就绪探针：45s 总超时，每轮 3s RPC 超时调 `client.Stat("/")`，失败就 `grpcPool.Remove(botID)` 踢掉连接重试，间隔 500ms。这个设计承认 bridge 启动有延迟（容器 task running ≠ gRPC server ready），所以用 Stat 探活而非仅靠 task 状态。

### ReconcileContainers：启动对账

`ReconcileContainers`（`manager_lifecycle.go:512-604`）在服务启动时跑：

1. `ListAutoStartContainers` 从 DB 拿所有 `auto_start=true` 的容器记录；
2. 对每条记录 `service.GetContainer` 查运行时——not found 就 `SetupBotContainer` 重建；
3. 存在但 task 没 running 就 `EnsureRunning` 重启；
4. legacy 容器（`mcp-`）单独走 TCP 路径，`SetLegacyIP` 让 `dialTarget` 能找到它；
5. running 的补一次 `ensureContainerNetwork`（容器可能在但网络掉了）。

这是「DB 是期望状态、运行时是实际状态」的调和循环——和 K8s controller 的思路一致，只是粒度到容器级。

### ContainerID 三级解析

`ContainerID`（`manager_lifecycle.go:28-68`）解析 bot 对应的容器 ID，三级回退：

1. DB 查 `GetContainerByBotID`（最快，命中直接返回）；
2. `ListContainersByLabel(BotLabelKey, botID)` 按 label 查（DB 没记录但容器在）；
3. `ListContainers` 全量扫，逐个 `BotIDFromContainerInfo` 匹配（兜底）；
4. 都没有 → `ErrContainerNotFound`。

三级回退保证了「容器被外部工具创建、DB 没同步」也能被发现，代价是兜底路径 O(n) 扫描。

## bridge 架构：gRPC over UDS

### proto 契约

`bridgepb/bridge.proto:7-20` 定义 `ContainerService`，11 个 RPC：

| RPC | 用途 | 风格 |
|---|---|---|
| `ReadFile` / `WriteFile` / `DeleteFile` | 文件读写 | 一元 |
| `ListDir` / `Stat` / `Mkdir` / `Rename` | 文件系统操作 | 一元 |
| `Exec` | 跑命令，支持 PTY / stdin / resize | 双向流 |
| `Tunnel` | 反向 TCP 隧道（host → 容器内任意 TCP） | 双向流 |
| `ReverseHTTP` | 容器内 HTTP 反向代理到 host | 双向流 |
| `ReadRaw` / `WriteRaw` | 大文件流式读写 | 服务端 / 客户端流 |

这套接口让 host 端的 Agent 工具（fsops / container / browser）完全不需要 SSH 或 docker exec——所有交互走 gRPC。

### Client 与连接池

`bridge.Client`（`client.go:28-33`）包装 `*grpc.ClientConn` + 生成的 stub。`DialTLS`（`client.go:53-88`）的关键：

- 默认 `insecure.NewCredentials()`——UDS 不需要 TLS，靠 socket 文件权限（`sockDir` 0o750，`manager.go:368`）做信任边界；
- TCP target 且配置了 mTLS 时才切 TLS credentials，并把 authority 钉到证书 SAN 的合成 DNS 名（`client.go:74-76`）；
- keepalive 30s ping、10s timeout、`PermitWithoutStream: true`——idle 连接也保活；
- 收发消息上限 16MiB（`client.go:62-65`），匹配 bridge 端 `MaxRecvMsgSize`。

`Pool.Get`（`client.go:684-719`）的 stale 检测是亮点：除了 `Shutdown` / `TransientFailure`，还检测「`Connecting` 状态超过 30s」（`connectingTimeout`，`client.go:25`）——卡在连接中的 sock 视为死连接踢掉重连。命中 stale 后先 `Remove` 再重 dial，并用双检锁避免并发重复 dial。

### Tunnel：让 host 复用容器内网络

`Client.DialContext`（`client.go:421-451`）实现了 `net.Conn` 接口，但底层是 gRPC `Tunnel` 双向流——host 端调 `DialContext("tcp", "127.0.0.1:5999")` 时，实际是在容器内打开一个 TCP 连接，数据通过 `TunnelFrame`（Open / Data / Close）双向流转（`client.go:466-589`）。

这个机制让桌面端的 VNC viewer、CDP 客户端无需直接访问容器网络——host 上的工具通过 bridge 隧道就能连容器内 `127.0.0.1:5999`（Xvnc）或 `127.0.0.1:9222`（Chrome CDP）。`Manager.DisplayDialContext`（`manager.go:469-475`）就是这条路。

### LocalService：无容器的 bridge

桌面端「本地工作空间」不跑真实容器，但复用同一套 bridge 协议。`LocalService.newBridgeClient`（`local_service.go:342-380`）用 `bufconn.Listener`（内存 gRPC transport）+ in-process `grpc.Server` 注册 `bridgesvc.Server`，工作目录直接指向 host 文件系统路径。容器操作（create/start/stop）退化为 JSON 元数据文件读写（`local_service.go:418-449`），`GetTaskInfo` 返回 `os.Getpid()` 假装 running（`local_service.go:230-241`）。

这是「适配器模式」的极致——上层 Manager 完全不感知「容器」是真容器还是本地目录，统一走 `runtimeService` 接口。

## 运行时抽象层（factory + docker/containerd/apple 适配器）

### 接口分组

`container/service.go:64-103` 把运行时能力分成 5 个接口：

- `ContainerService` —— create / get / list / delete / listByLabel（元数据与生命周期）
- `WorkloadService` —— start / stop / deleteTask / getTaskInfo / metrics / listTasks（task 调度）
- `NetworkService` —— setupNetwork / removeNetwork / checkNetwork（CNI 附件）
- `SnapshotService` —— commit / list / prepare / restoreContainer（快照与恢复）
- `ImageService` —— pull / get / list / delete / resolveRemoteDigest（镜像管理）

`runtimeService`（`runtime.go:7-12`）只组合前四个——workspace Manager 不直接用 `ImageService`，镜像操作走另外路径。`Service`（`service.go:98-103`）是全部五个的并集，给 `provider.ProvideService` 用。

### 工厂选择

`provider/provider.go:16-45` 的 `ProvideService` 按 `backend` 字符串 switch：

- `BackendContainerd`（`factory.go:7`）→ `containerdadapter.NewService`，连 `containerd.sock`；
- `BackendDocker`（`factory.go:9`）→ `dockeradapter.NewService`，用 Docker SDK；
- `BackendApple`（`factory.go:8`）→ `appleadapter.NewService`，基于 `acgo/socktainer`（Apple Virtualization framework）。

### RuntimeRouter：local 与 container 的分发

`RuntimeRouter`（`runtime_router.go:11-18`）包装 `container runtimeService` + `*LocalService`，所有方法按 ID 前缀路由：`routeByID`（`runtime_router.go:42-47`）看 `local-` 前缀走 local，否则走 container；`routeCreate`（`runtime_router.go:49-54`）多看一眼 `StorageRef.Driver=="local"`。`ListContainers` / `ListTasks` 这种聚合查询会合并两边结果（`runtime_router.go:64-77`）。

### Docker 适配器深读

`docker/service.go` 是把 `container.Service` 接口映射到 Docker SDK：

**CreateContainer**（`docker/service.go:126-169`）：
- `HostConfig.Mounts` 由 `toDockerMounts`（505-519）转换 `MountSpec`，识别 `ro` 选项；
- `HostConfig.Memory` / `NanoCPUs` 直接对应 `ResourceLimits`（`CPUMillicores * 1_000_000`，`docker/service.go:146-148`）；
- `PortBindings` 把 `9090/tcp` 绑到 `127.0.0.1` 随机 host port（`docker/service.go:139-141`）——Docker 后端走 TCP 9090 而非 UDS；
- `cfg.Env` 注入 `BRIDGE_TCP_ADDR=:9090`（`docker/service.go:155`）。

**BridgeTarget**（`docker/service.go:171-189`）：inspect 容器，优先返回 host port 绑定（`firstHostPort`），否则返回容器 IP + 9090。这是 `Manager.dialTarget` 第一优先级走的路径。

**CommitSnapshot**（`docker/service.go:369-398`）：用 `ContainerCommit` 把容器可写层打成 `memoh-workspace-snapshot:<name>` 镜像，parent 关系记到 label。Docker 没有 containerd 的 snapshotter 概念，这里用「commit 成镜像」模拟快照。

**mapDockerErr**（`docker/service.go:740-754`）：把 Docker 的 `errdefs` 错误映射成 `container.ErrNotFound` / `ErrAlreadyExists` / `ErrRuntime`，让上层用 `IsNotFound` / `IsAlreadyExists` 统一判断。

### containerd / apple 适配器（列表标注）

- **containerd**（`internal/container/containerd/`）：基于 containerd v2 client，有 `client.go`（连接）、`network.go`（CNI）、`resolv.go`（DNS 配置注入）、`timezone.go`（时区）、`metrics.go`（CPU/内存 metrics 采集）、`aliases.go`。是 Linux 服务器主力后端，支持完整 snapshotter（overlayfs 等）与 CDI 设备（GPU）。`service.go:34-41` 的 `DefaultService` 持有 `runtimeType` 字段，`Manager.usesKataRuntime`（`manager.go:224-230`）据此识别 Kata Containers（VM 隔离，需 TCP bridge）。
- **apple**（`internal/container/apple/`）：基于 `github.com/memohai/acgo/socktainer`，走 Apple Virtualization framework（macOS 上跑轻量 VM）。`service.go:24-27` 的 `ServiceConfig` 接 `SocketPath` + `BinaryPath`。资源硬限制不支持（`resource_limits.go:208-209`）。

## cmd/bridge：容器内二进制 + display（Xvnc / headed Chrome CDP）

### 入口与 gRPC server

`cmd/bridge/main.go:143-242` 的 `main`：

1. `initDataDir`（36-53）：确保 `/data` 存在，从 `/opt/memoh/templates` 种子模板（AGENTS.md、技能骨架），首次启动还迁移旧 `IDENTITY.md` → `AGENTS.md`（`migrateLegacyIdentityFile`，121-141）；
2. 把 `/opt/memoh/toolkit/bin` 追加到 `PATH`（151），让 exec 子进程能找到 `npx`/`uvx`；
3. **PID 1 zombie reaper**（161-168）：bridge 在容器里是 PID 1，exec 出去的子进程若不被 wait 会变僵尸，这里用 `syscall.Wait4(-1, ...)` 持续回收；
4. 监听 UDS（`BRIDGE_SOCKET_PATH`，默认 `/run/memoh/bridge.sock`），TCP 仅当 `BRIDGE_TCP_ADDR` 设了且是 loopback 才启用（`isBridgeTCPListenAddrAllowed`，244-250）——非 loopback TCP 直接拒绝启动，避免无认证暴露；
5. TCP 模式强制 mTLS（`main.go:211-221`），材料缺失即拒绝启动，不回退明文；
6. 注册 `ContainerServiceServer` + reflection，keepalive 参数（`MaxConnectionAge 30m` 等）防长连接僵死。

### display 监督

`cmd/bridge/display.go` 仅在 `MEMOH_DISPLAY_ENABLED=true` 时激活（`manager.go:420-426` 注入该环境变量）。监督链路：

1. `superviseXvnc`（68-147）：循环拉起 `Xvnc :99 -geometry 1280x960 -depth 24 -SecurityTypes None -localhost -rfbport 5999`，崩了按指数退避重启；
2. `startDisplaySession`（325-344）：等 X socket 就绪后，按优先级拉桌面——`startxfce4` > `xfce4-session` > `xfwm4` > `twm`（兜底），再起 `xsetroot` 设背景、`xterm`/`xfce4-terminal`、浏览器；
3. `superviseDesktopSession`（425-449）：每 5s 检查桌面健康，挂了重启；
4. `startDisplayBrowser`（532-563）：起 `google-chrome` / `chromium`，关键参数 `--remote-debugging-port=9222 --remote-debugging-address=127.0.0.1 --remote-allow-origins=* --user-data-dir=/tmp/memoh-display-browser`——**headed Chrome（带 GUI），CDP 监听 9222**，不是 headless；
5. `browserProcessRunning`（654-697）：扫 `/proc` 判断浏览器是否在跑，区分主进程与 renderer 子进程（看 `--type=`）。

host 端通过 `Manager.DisplayDialContext` → `bridge.Client.DialContext`（Tunnel RPC）连容器内 `127.0.0.1:5999`（VNC）或 `127.0.0.1:9222`（CDP），无需暴露端口。这是「headed browser use」的基础——Agent 能看到并操作真实浏览器 UI。

### ACP tools proxy

`cmd/bridge/tools_proxy.go:17-52` 起一个 loopback HTTP server（`MEMOH_ACP_TOOLS_PROXY_ADDR`，默认 `bridge.ACPToolsProxyAddr`），handler 是 `ReverseHTTPBroker`。容器内的工具通过 `ReverseHTTP` RPC 把 HTTP 请求反向发到 host，host 端 Agent 工具再走这个 proxy 访问——用于容器内需要回调 host 服务的场景（如 ACP 协议）。

## resource_limits：每 Bot 配额

### 能力探测

`ResourceLimitCapabilitiesFor`（`resource_limits.go:198-220`）按 backend + runtime 返回能力矩阵：

| backend / runtime | CPU 硬限制 | 内存硬限制 | 存储硬限制 | 存储软限制 |
|---|---|---|---|---|
| local | ✗ | ✗ | ✗ | ✓ |
| apple | ✗ | ✗ | ✗ | ✓ |
| vm / kata | ✓ | ✓ | ✓ | ✓ |
| docker / containerd（默认） | ✓ | ✓ | ✗ | ✓ |

local / apple 没有 cgroup 语义，只能做软限制（存储用 du 统计）；vm 全支持；docker/containerd 支持 CPU + 内存硬限制但存储只能软限制（`resource_limits.go:215-219`）。

### 变更需重建

`hardLimitDiffers`（`resource_limits.go:247-258`）比较 desired 与 applied（从容器 label 读回，`resourceLimitsFromLabels`，229-237）：若硬限制变了，`RequiresRecreate=true`，状态置 `pending_recreate`（`resource_limits.go:176-177`）。原因：Docker 的 `NanoCPUs` / `Memory` 在 `container create` 时固化到 cgroup，运行中改不了——只能重建。软限制（存储）不需要重建，运行时统计即可（`resource_limits.go:182-194`）。

### 持久化

配额存 `bot_workspace_resource_limit` 表（`SetResourceLimits` / `UpsertBotWorkspaceResourceLimits`，`resource_limits.go:97-120`），同时写到容器 label（`memoh.workspace.resource.cpu_millicores` / `memory_bytes`，`resource_limits.go:222-227`）——label 是「applied」状态的真值，DB 是「desired」状态的真值，两者差就是 `pending_recreate`。

## 端到端时序（用户消息 → bridge 执行工具 → 结果回传）

```

  │  消息
  ▼
Server HTTP → 对话编排(flow resolver + pipeline)
  │
  ▼
Agent.Stream → 组装 prompt + 工具列表 → LLM
  │  LLM 决定调 workspace 工具(如 fsops.ReadFile / container.Exec)
  ▼
工具层 → workspace.Manager.MCPClient(ctx, botID)        manager.go:245
  │  优先 LocalService.MCPClient(local bufconn)
  │  否则 bridge.Pool.Get(ctx, botID)                    client.go:684
  ▼
Pool.Get 命中缓存? ──否──> DialTLS(target)               client.go:53
  │                         target = dialTarget(botID)    manager.go:194
  │                         默认 "unix:///var/lib/memoh/run/<botID>/bridge.sock"
  │                         (Docker 后端走 TCP host:port)
  ▼
bridge.Client.Exec(ctx, cmd, workDir, timeout)           client.go:184
  │  gRPC bidirectional stream: ContainerService.Exec
  ▼
==== UDS 内核传输（host socket ↔ 容器 /run/memoh/bridge.sock）====
  ▼
容器内 bridge gRPC server (cmd/bridge/main.go:222)
  │  bridgesvc.Server.Exec                            server.go
  ▼
os/exec 跑命令 → stdout/stderr/exit 通过 ExecOutput stream 回传
  │
  ▼
Client 拼成 ExecResult{Stdout, Stderr, ExitCode}        client.go:252-257
  │
  ▼
工具结果回 Agent → 拼 prompt → LLM 继续或结束
  │
  ▼
Agent SSE 输出 → 用户
```

关键点：整条链路**只有一处网络跳**（UDS 内核），host 端的连接池让多次工具调用复用同一 gRPC 连接，Tunnel/ReverseHTTP 让容器内回调 host 也走同一条 UDS——没有任何端口对外暴露。

## 设计动机与取舍

1. **UDS 而非 TCP**：免认证（socket 文件权限 0o750 即信任边界）、免端口暴露（`iptables`/防火墙零配置）、零拷贝内核传输。代价是 host 必须能访问 socket 文件路径——所以 `/run/memoh` 用 bind mount 共享（`manager.go:385-390`）。Kata VM 跨边界时 UDS 失效，才退化到 TCP + mTLS（`manager.go:232-234`、`main.go:211-221`）。
2. **bridge 二进制而非 sidecar 容器**：runtime（bridge + toolkit）通过 `/opt/memoh` 只读 bind mount 注入（`manager.go:379-384`），bot 数据住容器可写层。好处是「换 runtime 只换镜像，不动数据卷」——`CleanupBotContainer(preserveData=true)` 能导出 `/data` 后重建（`manager_lifecycle.go:486-503`）。
3. **多后端靠接口分组而非抽象类**：Go 没有 inherited class，用「小接口组合 + 适配器 struct」实现多态。`RuntimeRouter` 用前缀路由而非策略模式，是因为 local 与 container 的能力差异太大（local 无 snapshot、无 metrics），硬塞一个接口会有大量 `ErrNotSupported`。
4. **PID 1 zombie reaper**：bridge 是容器 PID 1，`os/exec` 出去的子进程若不被 wait 会变僵尸。Go 1.20+ 在 Linux 5.3+ 用 `pidfd_open` 避免竞争（`main.go:159-168` 注释），老内核靠 `Wait4` 兜底。
5. **硬限制变更需重建**：cgroup v1/v2 的 CPU/memory 限制在 cgroup 创建时固化，运行中改 `docker update` 有诸多限制且不通用。Memoh 选择「label 记 applied、DB 记 desired、diff 触发重建」——简单可靠，代价是改配额要重启容器（数据通过 snapshot preserve 不丢）。
6. **headed Chrome 而非 headless**：display 开启时起 Xvnc + 有头 Chrome（CDP 9222），让 Agent 能操作真实浏览器 UI（看到渲染、处理弹窗）。headless Playwright 适合爬取，但有头适合「需要视觉反馈的浏览器 use」。代价是容器要装 X11 + 浏览器，镜像更大。
7. **legacy 兼容路径**：`mcp-` 前缀容器是 bridge 架构前的部署，走 TCP 9090。`ReconcileContainers` 单独处理它们（`manager_lifecycle.go:549-572`），`dialTarget` 走 `legacyIPs` 缓存。注释明说「Remove when all deployments have migrated」（`manager_lifecycle.go:549-550`）——是技术债，不是设计。
