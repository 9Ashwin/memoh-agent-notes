# 04 — 对话流编排

> **1 分钟叙事**：Memoh 把"一次对话怎么跑起来"拆成了两条互补的路径。**flow resolver** 是请求驱动的同步路径——用户发一条消息，`resolve()` 立刻组装上下文（历史 + 记忆 + 附件 + 系统提示）、调用 Agent、落库，是 chat 模式的默认通道。**pipeline（DiscussDriver）** 是事件驱动的异步路径——渠道消息先被 `AdaptInbound` 归一成 CanonicalEvent，经 `Reduce` 投影成 IntermediateContext（IC）、`Render` 成 RenderedContext（RC），再由一个 per-session 长生命周期 goroutine 消费 RC、决定是否触发回复，是 discuss 群聊模式的通道。两条路径并非独立王国：resolver 通过 `SetPipeline` 持有 pipeline，`resolve()` 内部会判断"这个 session 有没有 pipeline 数据"——有就走 `buildMessagesFromPipeline`（从 RC + TR 组装），没有就退回 legacy 的 `loadMessagesForTurnRun`。而 discuss driver 又通过 `RunConfigResolver` 接口（由 Resolver 实现）复用 resolver 的模型选择 / 凭证解析 / 工具装配 / 系统提示生成。这样"消息组装 / 记忆注入 / 历史裁剪 / Agent 调用"四件事在两条路径间解耦又共享：flow 负责一次性组装，pipeline 负责"渲染好的上下文 + 消费游标"，二者在 `buildBaseRunConfig` 与 `ResolveRunConfig` 这两个共享入口汇合。

## 核心结论

1. **conversation.Service 是个"瘦壳"**：只管会话容器的 CRUD 与读权限（`service.go:29-43`），参与者模型已退化到"只有 owner"——`AddParticipant` 直接返回 `ErrPermissionDenied`（`service.go:189-191`），`RemoveParticipant` 是 no-op（`service.go:245-247`）。真正"让对话跑起来"的逻辑全在 flow。
2. **flow.Resolver 是个"上帝对象"，但有清晰的编排主轴**：`resolve()`（`resolver.go:287`）是唯一的上下文组装入口，顺序固定：buildBaseRunConfig → loadMemoryContextMessage → 判 pipeline/legacy → 裁剪 → stripToolMessages → repairToolCallClosures。
3. **pipeline 是 DCP（Deterministic Context Pipeline）**：四层 adapt → reduce(IC) → render(RC) → compose(RC+TR)，外加 DiscussDriver 的事件循环与消费游标。它不是 flow 的替代品，而是 flow 的"上下文预渲染层"。
4. **两条路径的开关是 `usePipeline`**（`resolver.go:338-343`）：`r.pipeline != nil && sessionID != "" && turnRunAllowsPipelineContext(run)` 且 pipeline 中已加载 IC。命中则用 RC，否则用 legacy 历史。
5. **历史裁剪有三道闸**：token 预算截断（`resolver_history.go:136`）、压缩摘要替换（`resolver_history.go:200`）、tool 消息剥离（`resolver_history.go:360`）。70% 预算触发同步压缩（`resolver.go:372-401`）。
6. **记忆注入是"双向"的**：读时 `loadMemoryContextMessage` 把检索结果作为 user 消息塞进尾部（`resolver_memory.go:36`）；写时 `storeMemory` 在轮次结束后回调 `OnAfterChat` 抽取记忆（`resolver_memory.go:102`）。两侧都包了 before/after hooks。
7. **流式有自适应空闲超时与中断兜底**：`withIdleTimeout` 每个事件 reset（`resolver_stream.go:158`）；中断时 `persistPartialResult` 给孤立 tool_call 注入合成 error tool_result，保证下一轮可续（`resolver_stream.go:497`）。

## conversation service：会话 CRUD 与路由

`Service` 结构体极简（`service.go:29-32`）：一个 `dbstore.Queries` + 一个带 `service=conversation` 标签的 logger。它的职责边界很清楚——**只管会话容器本身，不管消息流转**。

**会话模型**：`Conversation`（`types.go:31-41`）有 `Kind` 三种——`direct` / `group` / `thread`（`types.go:11-15`），`thread` 携带 `ParentChatID` 形成树。`Create`（`service.go:46-93`）校验 kind、解析 bot/channel identity UUID、序列化 metadata，落到 `CreateChat` 查询。

**参与者退化**：值得注意的演进痕迹——`AddParticipant` 直接 `return ErrPermissionDenied`（`service.go:189-191`），注释明说"after removing bot member sharing"；`ListParticipants` 只返回 owner（`service.go:228-242`）。这意味着 Memoh 曾经支持多参与者共享 bot 成员，现在收敛为"一个会话一个 owner"。

**读权限两条路**：
- `GetReadAccess`（`service.go:112-136`）返回 `ConversationReadAccess{AccessMode, ParticipantRole, LastObservedAt}`，`AccessMode` 有两个值：`participant`（直接参与者）与 `channel_identity_observed`（渠道身份观察到，即群聊可见）（`types.go:25-28`）。
- `IsParticipant`（`service.go:219-225`）只对 owner 返回 true。

**Settings 与模型解析**：`Settings.ModelID`（`types.go:75-78`）允许 per-chat 覆盖模型。`UpdateSettings`（`service.go:266-292`）调 `resolveModelUUID`（`service.go:397-423`），后者支持 UUID 与 `model_id` slug 双路径——先试 UUID 解析 + `GetModelByID`，失败则 `ListModelsByModelID`，多结果报 `ErrModelIDAmbiguous`。

**接口分离**（`interfaces.go`）：`Reader`（Get）、`ParticipantChecker`（IsParticipant）、`Accessor`（聚合 + GetReadAccess）。这种小接口拆分是为了让上层按需依赖，避免把整个 Service 塞进调用方。

## flow resolver：消息组装 / 记忆注入 / 历史裁剪 / Agent 调用

### Resolver 的依赖与构造

`Resolver` 结构体（`resolver.go:82-117`）持有了约 20 个依赖：agent、modelsService、queries、memoryRegistry、conversationSvc、messageService、settingsService、accountService、sessionService、compactionService、eventPublisher、skillLoader、assetLoader、channelStore、botPermissions、pipeline、streamHTTPClient、bgManager、toolApproval、userInput、hookService。这是个明显的"上帝对象"，但它的复杂度来自 Memoh 的集成面太广——一次对话要协调模型、凭证、技能、记忆、工具审批、用户输入、钩子、容器资产。

构造（`NewResolver`，`resolver.go:120-172`）时为 LLM 流式专门建了一个 `http.Client`：`Timeout: 10min`、`ResponseHeaderTimeout: 30s`、`DialTimeout: 30s`，注释说"catch servers that accept TCP but never respond"。各依赖通过 `Set*` 方法延迟注入（`SetMemoryRegistry` / `SetPipeline` / `SetCompactionService` 等），这是为了打破 Uber FX 的 DI 循环。

### resolve()：编排主轴

`resolve()`（`resolver.go:287-489`）是整个 flow 的心脏，步骤固定：

1. **参数校验**（`resolver.go:288-296`）：query 或 attachments 必须有一个，bot/chat id 必须非空。
2. **buildBaseRunConfig**（`resolver.go:298-326`，实现在 `resolver.go:571-673`）：构建 `agentpkg.RunConfig`——选模型、解析凭证、加载技能、组装身份与时区、构造工具审批 handler。这是 chat 与 discuss 共享的入口（`Resolver.ResolveRunConfig` 在 `resolver.go:1064-1107` 也调它）。
3. **记忆注入（读）**（`resolver.go:327`）：`loadMemoryContextMessage` 返回一条 user 角色的上下文消息。
4. **判路径**（`resolver.go:338-343`）：
   ```
   usePipeline = r.pipeline != nil && sessionID != "" && turnRunAllowsPipelineContext(run)
   ```
   且 `pipeline.GetIC(sessionID)` 必须已加载。`turnRunAllowsPipelineContext`（`resolver_turns.go:326-333`）只允许 `session_head` 与 `bot_history` 两种 scope 走 pipeline，`turn_head`（变体重写）与 `empty` 不走。
5. **上下文组装**：
   - pipeline 路径（`resolver.go:352-353`）：`buildMessagesFromPipeline`（`resolver_history.go:263-302`）从 `pipeline.GetRC(sessionID)` 取渲染好的 segments，调 `ComposeContext(rc, trs, "")` 与 TR 交错合并，再做 token 裁剪。
   - legacy 路径（`resolver.go:354-402`）：`loadMessagesForTurnRun`（按 scope 加载，24h 窗口，`resolver_history.go:32-89`）→ `pruneHistoryForGateway` → `dedupePersistedCurrentUserMessage`（去重已持久化的当前用户消息，`resolver_history.go:91-125`）→ `replaceCompactedMessages`（压缩摘要替换，`resolver_history.go:200-257`）→ `trimMessagesByTokens`（token 裁剪，`resolver_history.go:136-198`）。
6. **同步压缩触发**（`resolver.go:372-401`）：当 `estimatedTokens >= contextTokenBudget * 70%` 时，`runCompactionSync` 阻塞压缩，然后重新加载消息并 `stripToolMessages`（因为已有摘要，tool 噪音不必留）。
7. **消息尾部拼接**（`resolver.go:404-416`）：memoryMsg 追加到末尾；非 pipeline 路径再追加 `req.Messages`；`sanitizeMessages`；消息数 >10 时 `stripToolMessages`；`repairToolCallClosures` 修复孤立 tool 消息。
8. **用户头格式化**（`resolver.go:419-437`）：`FormatUserHeader` 把发送者、渠道、时间、附件路径等打成 YAML 风格 header，作为 `runCfg.Query`。
9. **注入通道**（`resolver.go:447-478`）：若 `req.InjectCh != nil`，起 goroutine 把 `conversation.InjectMessage` 转成 `agentpkg.InjectMessage`，并注册 `InjectedRecorder` 记录插入位置——这是"工具轮次间向运行中的 agent 流注入用户消息"的机制（`types.go:265-293`）。

### 历史裁剪三道闸

**第一道：token 预算截断**（`resolver_history.go:136-198`）。从新到旧累加 `estimateMessageTokens`（char/4 估算，`resolver_history.go:127-134`），超预算则 `cutoff`。关键细节：截断后会跳过开头的 tool 消息（`resolver_history.go:166-168`），因为"orphaned tool message must follow a preceding assistant tool call"，否则 provider 会 400。截断时在头部插一条 system notice（`resolver_history.go:185-192`），告诉 LLM 历史被裁了、可用工具检索。

**第二道：压缩摘要替换**（`resolver_history.go:200-257`）。按 `CompactID` 分组，加载 `compaction_log`，状态为 `ok` 的整组消息替换成一条 `<summary>...</summary>` 的 user 消息。失败则保留原组。

**第三道：tool 消息剥离**（`resolver_history.go:360-483`）。`stripToolMessages` 删除 tool 角色消息，但保留 `ask_user` 的结果（对话语义可见）；对 assistant 消息，`stripNonAskUserToolCalls` 只保留 ask_user 的 tool_call 与可见文本。注释说"tool outputs are large and waste tokens; the LLM doesn't need raw tool results when summaries and memory tools are available for lookup"。

### 记忆注入

`resolveMemoryProvider`（`resolver_memory.go:13-34`）从 `botSettings.MemoryProviderID` 解析 provider，空则返回 nil。

**读**（`loadMemoryContextMessage`，`resolver_memory.go:36-100`）：先跑 `EventBeforeMemorySearch` hook（可 Deny）；调 `p.OnBeforeChat` 拿 `ContextText`；再跑 `EventAfterMemorySearch` hook（可 AppendContext）；最终包成一条 user 消息。无结果时也走 after hook，允许 hook 补充上下文。

**写**（`storeMemory`，`resolver_memory.go:102-154`）：轮次结束后调。`EventBeforeMemoryWrite` hook（可 Deny）→ `p.OnAfterChat`（带时区、显示名）→ `EventMemoryExtracted` → `EventAfterMemoryWrite`。`toProviderMessages`（`resolver_memory.go:156-170`）只保留非空文本，role 缺省为 assistant。

### 模型选择

`selectChatModel`（`resolver_model_selection.go:18-56`）优先级：request > chat settings > bot settings。`fetchChatModel`（`resolver_model_selection.go:58-97`）支持 UUID 与 slug 双路径，校验 `ModelTypeChat` 与 `Enable`，再校验 provider `Enable`。

### reasoning 决策

`resolveReasoningConfig`（`resolver.go:700-733`）处理三种 thinking mode：`none`（返回 nil）/ `adaptive`（on/off）/ `toggle`（on/off + per-message override）。`anthropicEffortEra`（`resolver.go:741-750`）检测 Anthropic 4.6+ 的 effort tier（none/minimal/xhigh/max），把云厂商变体（bedrock/vertex/azure/openrouter）也提升为 adaptive，避免走 legacy `budget_tokens` 路径被 4.7+ 拒绝。`effectiveReasoningEfforts`（`resolver.go:779-794`）过滤掉 OpenAI wire 不支持的 `max`。

### Turn 变体与并发控制

`TurnRunMode`（`resolver_turns.go:22-27`）四种：legacy / normal / rewrite / continuation。`TurnContextScope`（`resolver_turns.go:31-36`）四种：empty / turn_head / session_head / bot_history。`VariantTransition`（`resolver_turns.go:51-55`）三种动作：none / replace_base_head（普通延续，替换 head）/ create_sibling（重写，创建兄弟分支）。

**并发控制**：`enterSessionTurn`（`resolver_turns.go:88-103`）用 per-session 互斥锁 + 引用计数（`sessionTurnRefs`，`resolver_turns.go:112`）保证同一 session 的 turn 串行。`tryEnterIdleSessionTurn`（`resolver_turns.go:105-123`）是非阻塞版本，只在 refcount=0 时进入。

**变体转移**（`applyVariantTransition`，`resolver_turns.go:594-678`）：在事务内更新 `bot_session_turn_head`。rewrite 模式创建 sibling head（`CreateSessionTurnHead`），normal 模式替换 base head（`ReplaceSessionTurnHead`），最后 `UpdateSessionDefaultHeadTurnIfValid` 更新会话默认 head。

### 流式与中断兜底

`StreamChat`（`resolver_stream.go:81-252`）是 HTTP chunk 流，`StreamChatWS`（`resolver_stream.go:256-432`）是 WS 事件流。两者结构高度相似：

- 先 `isACPAgentSession` 判断是否走 ACP 外部运行时（Claude Code/Codex）。
- `enterSessionTurn` 串行化。
- `prepareTurnRun` → `applyUserMessageHook` → `resolve` → `prepareRunConfig`。
- `withIdleTimeout`：自适应空闲超时，每个事件 `Reset()`（`resolver_stream.go:158`），tool call 时 `RecordToolCall()`。
- **终态持久化**（`persistTerminalSnapshot`，`resolver_stream.go:437-477`）：跳过无可见输出的 abort（`snap.aborted && !snap.visibleOutput`，`resolver_stream.go:439-446`）；跳过无 assistant 输出的快照（`hasPersistableAssistantOutput`，`resolver_stream.go:479-486`）。
- **中断兜底**（`persistPartialResult`，`resolver_stream.go:497-550`）：无终态快照时，若有 partial messages，`AllowPendingToolCalls=false` 触发 `repairToolCallClosures` 注入合成 error tool_result（`resolver_stream.go:510-518` 注释）。这保证了"被中断的对话可被要求继续"。
- **注入消息交错**（`interleaveInjectedMessages`，`resolver_stream.go:558-581`）：按 `InsertAfter` 索引把注入的 user 消息插回 round。

### 调度与心跳触发

`TriggerSchedule`（`resolver_trigger.go:20-74`）与 `TriggerHeartbeat`（`resolver_trigger.go:77-148`）都走 `legacyTurnRun` + `resolve` + `agent.Generate` + `storeRound`。区别在 prompt：schedule 用 `GenerateSchedulePrompt`，heartbeat 先读容器内 `/data/HEARTBEAT.md` checklist 再用 `GenerateHeartbeatPrompt`。heartbeat 还会判断输出是否 `HEARTBEAT_OK` 决定 status。

## pipeline：discuss / chat 另一条路径（adapt / projection / rendering / driver）

### DCP 四层

**Adapt 层**（`adapt.go`）：`AdaptInbound`（`adapt.go:14-24`）把 `channel.InboundMessage` 按 `metadata.event_type` 分流成 `MessageEvent` / `EditEvent` / `ServiceEvent`。它把渠道特定的 `MessagePart`（codeblock/link/mention/emoji/text + styles）归一成平台无关的 `ContentNode` 树（`adapt.go:97-194`），附件归一成 `Attachment`（含 `ContentHash` 与 `FilePath`，`adapt.go:209-237`）。

**Projection 层**（`context.go`）：`IntermediateContext`（`context.go:76-81`）= SessionID + Nodes + Users + ChatTitle。`Reduce`（`context.go:135-148`）是纯函数，不修改输入 IC（`cloneIC`，`context.go:150-163`）。`reduceMessage`（`context.go:165-255`）做三件事：去重（按 MessageID）+ 用户改名检测（生成 `user_renamed` 系统事件，`context.go:182-196`）+ 回复解析（找目标消息，填充 `ReplyToSender` 与 `ReplyToPreview`，`context.go:213-232`）。`reduceEdit` 改写内容、`reduceDelete` 标记 deleted、`reduceService` 处理群生命周期事件（`context.go:257-324`）。

**Rendering 层**（`rendering.go`）：`Render`（`rendering.go:45-59`）把 IC 每个 node 渲染成一个 `RenderedSegment`。消息渲染成 XML 风格（`rendering.go:77-173`）：`<message id="..." sender="..." t="..." channel="..." conversation="..." type="..." target="...">` 包裹正文、`<in-reply-to>`、附件 `<attachment>` / `<image>`。`ImageRefs`（`rendering.go:154-162`）把图片附件以 `content_hash` 形式携带，供后续 inline 为 vision 输入。系统事件渲染成 `<event type="members_joined" .../>` 等（`rendering.go:183-234`）。

**Compose 层**（`projection.go`）：`MergeContext`（`projection.go:64-129`）把 RC segments 与 TR entries 按时间戳交错合并——RC 用 `ReceivedAtMs`，TR 用 `RequestedAtMs`，相等时 RC 优先；连续 RC 之间合并成一条 user 消息。`ComposeContext`（`projection.go:132-147`）在 Merge 基础上可选前置 compaction summary，并估算 token（`charsPerToken=2`，`projection.go:10`）。

### Pipeline 容器

`Pipeline`（`pipeline.go:8-13`）是 per-session IC/RC 的 goroutine-safe 容器。`PushEvent`（`pipeline.go:26-41`）= reduce IC + render RC。`ReplaySession`（`pipeline.go:45-58`）从持久化事件冷启动重建。`GetRC` / `GetIC` / `DropSession` / `UpdateRenderParams` 提供生命周期管理。

### DiscussDriver：事件驱动循环

`DiscussDriver`（`driver.go:84-89`）管理 per-session 长生命周期 goroutine。`NotifyRC`（`driver.go:129-159`）在 RC 到达时推送，若 goroutine 未运行则启动一个。`runSession`（`driver.go:200-251`）是事件循环：`select` 在 `stopCh` / `idle.C`（10 分钟空闲超时，`driver.go:198`）/ `rcCh` 之间。收到 RC 后 drain 信道，然后 `handleReply`。

`handleReplyWithAgent`（`driver.go:257-380`）是 discuss 的核心：

1. `loadTurnResponses`（`driver.go:648-670`）**全量**加载 session 的 assistant/tool 消息，无时间截断。注释（`driver.go:643-647`）解释：截断 TR 而完整重放 RC 会造成"用户消息可见、bot 早期回复缺失"的非对称上下文，迷惑 LLM 与 loop-detection。裁剪应交给 compaction。
2. **冷启动 anchor**（`driver.go:269-271`）：`lastProcessedMs = max(anchorFromTRs(trs), loadDiscussCursor)`。这避免 idle-timeout 重启后把整段历史当新消息重答。
3. **触发判定**（`driver.go:277-279`）：`LatestExternalEventMs(rc, lastProcessedMs) == 0` 则无新外部事件，跳过。
4. **ComposeContext** 组装。
5. **ResolveRunConfig 复用 resolver**（`driver.go:294-300`）：discuss 不自己选模型/装配工具，而是调 `Resolver.ResolveRunConfig`（`resolver.go:1064-1107`），后者走 `buildBaseRunConfig` + `prepareRunConfig`。
6. **ACP 路径**（`driver.go:301-326`）：若 `RuntimeType == RuntimeACPAgent`，走 `streamDiscussACPRuntime`。参与门槛：DM 总是回复（`channel.IsPrivateConversationType`），群聊需 `wasRecentlyMentioned`。未达门槛时只推进游标不启动运行时（`driver.go:318-321`），避免为群聊被动闲聊预热 Codex/Claude Code。
7. **模型路径**（`driver.go:327-360`）：`agent.Stream` + 晚绑定 prompt（`buildLateBindingPrompt`，`driver.go:716-730`），核心是"必须用 `send` 工具说话，文本输出对用户不可见"。
8. **游标推进**（`driver.go:374-379`）：`advanceDiscussCursor` 推进到本次消费的最新 RC `ReceivedAtMs`（非 wall-clock），避免误标记生成中到达的消息。

### 持久化与 TR 解码

`EventStore`（`persistence.go`）：`PersistEvent`（`persistence.go:39-89`）写 `bot_session_events` 表，`ON CONFLICT DO NOTHING` 去重（`persistence.go:79-81`）。`LoadEvents`（`persistence.go:92-117`）按 `received_at_ms` 排序。`GetDiscussConsumedCursor` / `UpsertDiscussConsumedCursor`（`persistence.go:133-181`）维护 discuss 消费游标，scope 默认 `default`，也可按 route 或 source 维度。

`DecodeTurnResponseEntry`（`turn_response.go:17-46`）把持久化消息解码成 TR entry。关键设计（`turn_response.go:14-16` 注释）：tool calls/results 必须保留为原生结构化内容，**不能**渲染成 assistant 可见的伪协议文本——否则模型会模仿文本而不发真实 tool call。`nativeAssistantContent`（`turn_response.go:62-130`）处理三种持久化格式：plain string / parts array / 顶层 ToolCalls。reasoning parts 故意省略（`turn_response.go:96-98`），因为"reasoning is model-internal and must not leak back into subsequent prompts verbatim"。

## flow vs pipeline 的关系与取舍

**关系是"共享底座 + 两条上行路径"**：

```
                   ┌─ flow.Chat/StreamChat (同步, 请求驱动) ──→ resolve() ──→ agent.Generate/Stream
用户消息 ─→ 渠道适配 ┤                                                        ↑
                   └─ pipeline.PushEvent (异步, 事件驱动) ──→ DiscussDriver ──→ ResolveRunConfig + agent.Stream
```

两条路径共享的底座是 `buildBaseRunConfig`（`resolver.go:571-673`）与 `prepareRunConfig`（`resolver.go:1110-1205`）——模型选择、凭证解析、技能加载、工具审批、系统提示生成都在这里。flow 直接调，discuss 通过 `RunConfigResolver` 接口调。

**上下文组装的分歧**：
- flow legacy 路径：`loadMessagesForTurnRun` 从 `bot_history_messages` 表加载原始消息 → 裁剪 → 拼接 memoryMsg + reqMessages。
- pipeline 路径：`buildMessagesFromPipeline` 从 `pipeline.GetRC` 取预渲染的 segments + `loadTurnResponses` 取 TR → `ComposeContext` 交错合并。RC 是"已经渲染好的 XML 风格上下文"，TR 是"bot 自己的历史回复"。

**何时走哪条**：`usePipeline`（`resolver.go:338-343`）只看"pipeline 配置了 + session 有 IC"。这意味着即便走 flow.Chat，只要 session 在 pipeline 里有数据，也会从 RC 组装。真正的"纯 pipeline 模式"是 DiscussDriver——它不经过 `resolve()`，而是 `ResolveRunConfig` + 自己组装 messages。

**取舍**：
| 维度 | flow（chat 模式） | pipeline（discuss 模式） |
|---|---|---|
| 触发 | 请求驱动，一问一答 | 事件驱动，RC 变化即可能触发 |
| 上下文 | 每次现加载历史 + 裁剪 | 预渲染 RC + 全量 TR，游标推进 |
| 批量 | 单轮 | 可聚合 drain 多条 RC 再回复 |
| 晚绑定 | 无（直接用 query） | 有（`buildLateBindingPrompt` 强制 send 工具） |
| 并发 | per-session 互斥锁 | per-session goroutine + 10min 空闲超时 |
| 适用 | 交互式 1:1 | 群聊被动响应、需去重/批量 |
| ACP | StreamChat 内部分流 | 独立参与门槛（DM/mention） |
| 代价 | 每次重算上下文 | 维护 IC/RC 双层状态 + 消费游标 |

**为什么 discuss 要单独一条路径**：群聊场景下，bot 不该对每条消息都回。DiscussDriver 用 RC 的 `MentionsMe` / `RepliesToMe` 标记 + 消费游标实现"只回被 @ 的、且不重复回已处理的"。flow 没有这个机制——它假定每条入站消息都该回。

**为什么 chat 也要能走 pipeline 上下文**：一旦 session 在 discuss 模式下跑过（有 IC/RC），切换到 chat 模式时，从 RC 组装上下文比从 `bot_history_messages` 重加载更一致——避免两种模式对同一 session 产生不同的上下文视图。

## 设计动机与取舍

1. **上帝对象 Resolver 的合理性**：20 个依赖看似臃肿，但"一次对话"本身就要协调这么多子系统。Memoh 的选择是把编排集中在一个对象里，通过 `Set*` 延迟注入打破循环，而不是拆成多个小服务再互相调用——后者在集成面这么广时反而更难追踪。代价是测试要 mock 很多东西（看 `resolver_test.go` 的体积）。

2. **消息格式边界唯一性**：`resolver_messages.go` 顶部注释明确"this is the only place where this conversion should happen"。SDK `Message` 与持久化 `ModelMessage` 的转换被收口在 resolver 边界，避免到处都是格式转换逻辑。`ModelMessage.Content` 用 `json.RawMessage`（`types.go:97`）允许字符串或数组两种内容形态共存，`TextContent()` 自动适配。

3. **裁剪三道闸的顺序**：token 截断 → 压缩替换 → tool 剥离。token 截断先做粗筛；压缩替换在剩余消息里找 compact 组替换；tool 剥离最后做精筛。70% 阈值触发同步压缩是"宁可阻塞也要保上下文健康"的保守选择——注释（`resolver.go:370-373`）明说"keep the context healthy and avoid edge-case timeouts"。

4. **tool 消息剥离但保留 ask_user**：tool 输出体积大、对 LLM 后续推理价值低（有 memory 与 search 工具可查），但 `ask_user` 的问答是对话语义的一部分，必须保留（`resolver_history.go:357-359` 注释）。这是个很实际的取舍。

5. **pipeline 的 TR 全量加载**：`loadTurnResponses` 不截断（`driver.go:643-647` 注释），因为截断 TR 而完整重放 RC 会造成非对称上下文。这把裁剪责任完全推给 compaction，是"单一职责"的体现。

6. **discuss 游标用 RC 时间戳而非 wall-clock**：`advanceDiscussCursor`（`driver.go:500-517`）推进到 `latestRCReceivedAtMs`，注释（`driver.go:374-378`）说"messages that arrive DURING LLM generation will land in a newer RC with ReceivedAtMs > this cursor and correctly trigger another round; wall-clock would wrongly mark them processed"。这是个容易踩坑的细节——用 wall-clock 会把生成中到达的消息误标为已处理。

7. **TR 保留原生结构化内容**：`turn_response.go:14-16` 注释强调不能把 tool call 渲染成伪协议文本，否则模型会模仿文本。这是个反直觉但重要的设计——把结构化数据原样透传给 LLM，比"渲染好看"更安全。

8. **流式中断兜底**：`persistPartialResult` 给孤立 tool_call 注入合成 error tool_result（`resolver_stream.go:510-518`），保证下一轮可续。这解决了"流被中断后对话状态不一致"的痛点，是个生产级细节。

## 验证问题

1. **说清 flow 与 pipeline 两条路径的关系与取舍**。（提示：共享 `buildBaseRunConfig` 底座；flow 是请求驱动同步、pipeline 是事件驱动异步；`usePipeline` 是切换开关；discuss 用游标 + 晚绑定 + 参与门槛处理群聊被动响应，flow 假定每条消息都该回。）

2. `resolve()` 的编排顺序是什么？为什么记忆消息放在尾部而不是头部？（提示：buildBaseRunConfig → loadMemory → 判路径 → 裁剪 → stripTool → repair；记忆放尾部是因为它是"当前轮的检索补充"而非历史，且 LLM 对尾部内容更敏感。）

3. 历史裁剪三道闸分别是什么？为什么 tool 消息要剥离但 ask_user 要保留？（提示：token 截断、压缩替换、tool 剥离；ask_user 是对话语义可见的，tool 输出是噪音且有 memory 工具可查。）

4. DiscussDriver 的冷启动 anchor 为什么是 `max(anchorFromTRs, loadDiscussCursor)`？如果只用其中一个会怎样？（提示：只用 TR anchor 会忽略跨重启的游标持久化，导致已处理消息被重答；只用游标会在游标丢失时重答整段历史。）

5. 流式中断时，`persistPartialResult` 如何保证对话状态可续？为什么 `AllowPendingToolCalls=false`？（提示：注入合成 error tool_result 修复孤立 tool_call；false 触发 `repairToolCallClosures`，让 provider 序列化器（尤其 Anthropic）不会因 tool_call 无配对 result 而 400。）
