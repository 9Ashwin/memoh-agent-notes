# 13 — 桌面端

> Memoh 的桌面端不是又一个 Web 壳。它是一道"本地优先"的边界：Electron 主进程代替用户管住一整个本地后端的生命周期——先渲染 SQLite/workspace 配置，拉起内嵌 Qdrant，再 spawn 出 `memoh-server serve` 绑在 `127.0.0.1:18731`，最后把 server 地址和自动登录令牌通过 IPC 喂给渲染层。打包时连 server 二进制、CLI 二进制、bridge 运行时、Qdrant 二进制、provider 模板一起塞进 `Contents/Resources/`，让一个 dmg/AppImage/exe 装完就能离线跑全栈。它解决三件事：①用户不用自建部署栈就能用；②向量检索不依赖云 Qdrant；③终端用户能在命令行里 `memoh start/stop/chat` 复用同一套 userData，因为桌面和 CLI 共享同一份 pid 文件契约。

## 核心结论

1. **桌面端 = Electron 主进程 + 本地 server + 内嵌 Qdrant + 打包 CLI**，四者通过 `app.getPath('userData')` 下的文件协作，不通过 IPC 跨进程传递运行时状态（`apps/desktop/AGENTS.md:9-13`）。
2. **`local-server.ts` 是唯一启动门**：dev 下 `go build ./cmd/agent`，packaged 下解析 `Resources/server/memoh-server`，渲染 `config.toml`，跑 `migrate up`，spawn `serve`，写 pid/log，等 `/ping` 就绪（`apps/desktop/src/main/local-server.ts:671-703`）。
3. **内嵌 Qdrant 是 per-user 进程**：二进制、config、storage、pid、ports 全部落在 `userData/qdrant/`，动态抢两个端口（http+grpc）并持久化到 `ports.json`，冲突时自动换端口（`apps/desktop/src/main/qdrant.ts:35-57, 326-363`）。
4. **桌面与 CLI 共享 pid 契约**：`daemon.ts`（TS）和 `internal/tui/local/service.go`（Go）的 `ManagedPid` JSON 结构完全一致，桌面 spawn 的 server 能被 `memoh stop` 杀掉，反之亦然（`internal/tui/local/service.go:17-24`，`apps/desktop/src/main/daemon.ts:3-7`）。
5. **productName 锁定是硬契约**：本地桌面 app 叫 `Memoh Local`，在线桌面 app 叫 `Memoh`，两者 userData 目录不同、数据隔离；Go CLI 硬编码 `productName = "Memoh Local"`（`internal/tui/local/paths.go:18`，`apps/desktop/electron-builder.yml:2`）。
6. **dev slot 机制**仅限开发：用 `mise run desktop:dev -- <keyword>` 起多个并行实例，每个 slot 独占 SQLite/端口/pid/log，但 Qdrant 是共享的（`apps/desktop/AGENTS.md:322-376`）。

## local-server.ts：本地 server 启动门（18731）

### 端口与 slot

默认端口固定 18731（`local-server.ts:18`）。`activeSlot()` 读 `MEMOH_SLOT` 环境变量，空或 `default` 走单实例布局；命名 slot 必须带 `MEMOH_SLOT_SERVER_PORT`，否则直接抛错——绝不静默回落 18731，避免和默认实例撞车（`local-server.ts:23-46`）。slot 后缀 `.<name>` 拼到 pid/log/config 文件名上，数据目录隔离到 `data/instances/<slot>/`（`local-server.ts:50-66`）。

### 二进制解析：dev vs packaged

`serverCommand()` 按 `app.isPackaged` 分两路（`local-server.ts:139-169`）：

- **dev**：调 `buildDevServerBinary` 跑 `go build -o ./cmd/agent`，产物落在 `desktopServerWorkDir()/bin/`（命名 slot 隔离到 `data/instances/<slot>/bin/`，避免两个 worktree 编译到同一路径互踩，`local-server.ts:117-137`）。配置模板来自仓库内 `conf/app.local.toml`，provider 模板来自 `conf/providers`。
- **packaged**：直接取 `desktopResourcePath('server', 'memoh-server')`，配置模板来自 `Resources/config/app.local.toml`。`desktopResourcePath` 在 packaged 时映射到 `process.resourcesPath`，dev 时映射到 `apps/desktop/resources/`（`paths.ts:19-24`）。

### 配置渲染：把模板变成可运行 config

`prepareConfig` 读模板，`applyLocalConfigDefaults` 做五件事（`local-server.ts:266-301`）：

1. `__HOME__` 占位符替换成真实 home。
2. 写 `[container].data_root` / `[container].runtime_dir` / `[local].metadata_root` 到 slot 数据目录。
3. 写 `[sqlite].path` 到 `<dataDir>/memoh.db`——这是桌面端 SQLite 数据库落点。
4. 写 `[registry].providers_dir`。
5. 命名 slot 时额外写 `[server].addr = ":<slotPort>"`；有 Qdrant 时写 `[qdrant].base_url`。

`setTomlString` 是手写的行级 TOML 改写器：定位 `[section]` 块，正则匹配 `key = "..."` 替换值，找不到就抛错（`local-server.ts:356-380`）。没用 TOML 库——因为只改已知 key，行级改写足够且零依赖。`detectDockerHost` 探测 `~/.docker/run/docker.sock` 或 `/var/run/docker.sock`，写进 `[docker].host`（`local-server.ts:310-354`）。

### runtime 准备：bridge 交叉编译

`prepareRuntime` 准备 bridge 运行时（容器里跑的 Linux 二进制，`local-server.ts:405-437`）：

- **dev**：`go build -o bridge ./cmd/bridge`，强制 `GOOS=linux`、`GOARCH=arm64/amd64`（`dockerBridgeArch`，`local-server.ts:472-481`）。bridge 在容器内执行，所以宿主是 mac/win 也要编 Linux 版。再同步 bridge 模板和 workspace toolkit。
- **packaged**：直接把 `Resources/runtime/` 整目录拷到 slot runtime 目录。

### 启动序列：ensureLocalServer

`ensureLocalServer` 是主入口（`local-server.ts:671-703`），顺序如下：

1. 先 `bundledServerIdentity` 跑 `memoh-server version` 拿当前打包二进制的版本+commit。
2. `probeServer` GET `/ping`（1s 超时，`local-server.ts:497-515`）：
   - 已有 server 且是本桌面管的（pid 活）→ 重启它以挂上内嵌 Qdrant。
   - 已有 server 但不是本桌面管的 → 抛错，提示用户先停掉（`local-server.ts:682-684`）。这避免静默连到外部 server 导致 memory 用错 Qdrant。
3. `ensureEmbeddedQdrant()` 拿 Qdrant gRPC URL。
4. `spawnServer`：`prepareRuntime` → `runMigrations` → `spawn` detached → `writeManagedServerPid`（`local-server.ts:526-559`）。
5. `waitForServer` 30s 轮询 `/ping`（500ms 间隔，`local-server.ts:517-524`）。
6. `ensureDesktopAuthToken` 自动登录。

### 迁移与脏版本修复

`runMigrations` 跑 `migrate up`（`local-server.ts:561-580`）。失败时检查输出是否含 `Dirty database version 2`——是的话先 `migrate force 2` 修复再重试。这是针对一个已知脏状态的定向修复，不是通用重试。

### 自动登录

`ensureDesktopAuthToken` 从 `config.toml` 的 `[admin]` 段读 username/password（`readAdminCredentials`，`local-server.ts:760-789`），POST `/auth/login` 拿 `access_token`，缓存在内存里供渲染层通过 `desktop:auth-token` IPC 取用。用户无感登录。

### OAuth 回调代理

`ensureProviderOAuthCallbackProxy` 在 1455 端口起一个 node http server，把回调转发到 `LOCAL_SERVER_BASE_URL`（`local-server.ts:241-264`）。同时监听 `127.0.0.1` 和 `::1`。provider OAuth 重定向到 `http://localhost:1455/...`，桌面把这个流量代理到本地 server。这是固定端口，所以 dev slot 里只有一个 slot 能拿到它（`AGENTS.md:369-370`）。

## qdrant.ts：内嵌 Qdrant 进程管理

### per-user 目录布局

所有 Qdrant 运行时文件都在 `userData/qdrant/` 下（`qdrant.ts:35-57`）：

| 文件 | 作用 |
|------|------|
| `storage/` | 向量数据持久化目录，写进 config.yaml 的 `storage_path` |
| `config.yaml` | 动态生成的 Qdrant 配置（host/port/storage_path/cluster） |
| `qdrant.log` | Qdrant stdout/stderr |
| `qdrant.pid.json` | 进程 pid 描述（和 server 共用 `ManagedPid` 形状） |
| `ports.json` | 持久化的 http+grpc 端口对 |

二进制路径按目标三元组定位：`Resources/qdrant/<platform>-<arch>/qdrant`（`qdrant.ts:63-73`）。启动前 `chmod 0o755`（非 win32，`qdrant.ts:226-239`）。

### 端口管理：持久化 + 动态抢占

`validatePorts` 校验端口对：两个整数、在 1-65535、不相等（`qdrant.ts:75-87`）。`readQdrantPorts` 从 `ports.json` 读，`writeQdrantPorts` 以 `0o600` 写（`qdrant.ts:89-100`）。

抢端口用临时 `net.Server`：`reservePort(n)` 监听 → 拿到实际端口 → `unref` 并返回 close 函数（`qdrant.ts:137-159`）。传 0 让 OS 分配。`reserveQdrantPorts` 同时抢 http+grpc 两个，任一失败都释放已抢的（`qdrant.ts:173-192`）。

### 启动策略：persisted → new

`ensureEmbeddedQdrant` 的复用逻辑（`qdrant.ts:365-410`）：

1. 已 ready 直接返回。
2. 读持久化端口 + pid，若进程活且 `/healthz` 通 → 直接认领，不重启。
3. pid 在但健康检查失败 → `stopManagedPid` 杀掉。
4. `launchWithPersistedOrNewPorts`：先试持久化端口（`qdrant.ts:342-363`），`EADDRINUSE` 则 `launchWithNewPorts`（`qdrant.ts:326-340`，3 次重试）。

这个策略的目标是端口稳定性——尽量复用上次的端口，避免每次启动都换，但撞了就换。

### spawn 细节

`spawnQdrant`（`qdrant.ts:241-292`）：

- 先 `writeQdrantConfig` 生成 config.yaml：`host: 127.0.0.1`、`cluster.enabled: false`、`storage_path` 指向 `userData/qdrant/storage/`（`qdrant.ts:113-128`）。
- 开 logFd 追加模式，**先释放端口预约再 spawn**——因为 spawn 后 Qdrant 会自己 bind 同一端口，预约不释放就会撞自己（`qdrant.ts:248-249`）。
- `detached: true` + `unref()`，子进程脱离父进程组，桌面退出后 Qdrant 不会跟着死（除非主动 stop）。
- `env.QDRANT__TELEMETRY_DISABLED = 'true'` + `--disable-telemetry` 双保险关遥测。
- stdio 全部进 logFd（`['ignore', logFd, logFd]`）。

### 就绪探测

`probeQdrant` GET `/healthz`（1s 超时，`qdrant.ts:194-205`）。`waitForQdrant` 30s 轮询，300ms 间隔，且每次检查子进程是否已 exit（`child.exitCode !== null`），exit 了立即失败不傻等（`qdrant.ts:207-215`）。

失败后 `qdrantLogMentionsAddrInUse` 扫日志尾部 16KB 找 `EADDRINUSE`/`Address already in use`/`os error 48`/`os error 98` 等模式（`qdrant.ts:217-224`），把错误升级成 `EADDRINUSE` 触发换端口重试。

## electron-builder.yml：打包资源

`extraResources` 把七个目录映射进 `Contents/Resources/`（`electron-builder.yml:20-38`）：

| from | to | 说明 |
|------|----|------|
| `resources/server` | `server` | memoh-server 二进制 |
| `resources/cli` | `cli` | memoh CLI 二进制 |
| `resources/runtime` | `runtime` | bridge 二进制 + 模板 |
| `resources/config` | `config` | `app.local.toml` 模板 |
| `resources/providers` | `providers` | provider registry YAML |
| `resources/qdrant` | `qdrant` | 按 `MEMOH_DESKTOP_QDRANT_TARGET` 过滤目标平台二进制 |
| `resources/gstreamer` | `gstreamer` | 按 `MEMOH_DESKTOP_GSTREAMER_TARGET` 过滤 |

Qdrant 和 GStreamer 用 `filter` 字段按构建时的环境变量选择目标平台子目录——一次构建只打一个目标平台的二进制，避免包体膨胀（`electron-builder.yml:31-38`）。

平台目标：mac → dmg（arm64+x64，最低 macOS 12，`electron-builder.yml:39-49`）；linux → AppImage+deb+rpm（x64）；win → NSIS（x64）。`notarize: true` 走 Apple 公证。`npmRebuild: false` 避免重编原生依赖。

构建管线 `scripts/build.mjs` 在 electron-builder 之前跑 `prepare-local-server.mjs`（三次 `go build`：agent→server、memoh→cli、bridge→runtime，`AGENTS.md:481-489`）、`prepare-qdrant.mjs`、`prepare-gstreamer.mjs`，把目标平台资源先摆好。

## cmd/memoh：桌面配套 CLI

Cobra 结构（`cmd/memoh/root.go`）：

- `newRootCommand` 默认走 TUI（Bubble Tea，`root.go:27-29, 67-78`）。
- `--server` flag 覆盖默认 `http://127.0.0.1:18731`，传了就禁用自动登录（`root.go:43`，`bots.go:110-117`）。
- 子命令：`chat`、`bots`、`tui`、`version`、以及 service 命令组（`start/stop/restart/status/logs`，`service.go:17-25`）。

**这是桌面 edition**——没有 docker-compose / login 子命令，一切操作针对桌面已管的本地 server（`AGENTS.md:460-465`）。

### service 命令：纯进程级

`newStartCommand`（`service.go:27-57`）调 `local.StartServer`，后者解析 `BundledServerBinary()`——通过 `os.Executable()` 找到自己所在 `Resources/cli/`，再上一级 `Resources/` 拼 `server/memoh-server`（`paths.go:177-200`）。`--wait` 默认 true，等 `/ping` 就绪。`configPath` 默认 `ResolveConfigPath()`：先找 packaged 的 `userData/config.toml`，再找 dev 的 `userData/local-server/config.toml`，都没有就报错让用户先开一次桌面（`paths.go:92-108`）。

`newStopCommand`（`service.go:59-86`）读 pid 文件 → SIGTERM → 等 timeout（默认 5s）→ SIGKILL。`newStatusCommand`（`service.go:124-163`）同时查 pid 活性和 HTTP `/ping`。

### chat / bots

`chat`（`chat.go`）：`--bot` 和 `--message` 必填，`--session` 可选。无 session 时先 `CreateSession`，再 `StreamChat` 走 WebSocket，事件 `start/message/error/end` 分别打印。

`bots create`（`bots.go:28-66`）：`--name` 必填，REST 创建。`bots delete <id>` 要 `--yes` 确认。

### 共享契约：ManagedPid

`internal/tui/local/service.go:17-24` 的 `ManagedPid` 结构（`pid`/`command`/`startedAt`）与 `apps/desktop/src/main/daemon.ts:3-7` 完全镜像。`WritePidFile` 用 `tmp + rename` 原子写（`service.go:50-63`）。这是桌面和 CLI 互操作的基础——`memoh stop` 能杀桌面 spawn 的 server，桌面 `stopManagedServer` 也能杀 CLI spawn 的。

## 桌面端 vs server 模式数据隔离

### 两套 productName，两套 userData

`electron-builder.yml:2` 把本地桌面 app 命名为 `Memoh Local`，而 `electron-builder.online.yml` 命名为 `Memoh`（`AGENTS.md:516-518`）。Electron 的 `app.getPath('userData')` 由 app 名决定，所以：

- **Memoh Local**（本地优先）：userData = `~/Library/Application Support/Memoh Local`（macOS）。所有本地数据在这里：SQLite、Qdrant storage、config.toml、pid/log。它内嵌 server + Qdrant，数据全本地。
- **Memoh**（在线）：userData = `~/Library/Application Support/Memoh`。它连远程部署的 server，本地不存业务数据。

Go CLI 硬编码 `productName = "Memoh Local"`（`paths.go:18`），所以 CLI 只认本地优先那套 userData。**如果改了桌面 productName 却没改 paths.go，CLI 会读写另一个 userData 目录，和桌面静默分叉**（`AGENTS.md:521-525`）。

### 桌面端内部的数据落点

默认 slot 下（`local-server.ts:57-66`）：

| 数据 | 路径（相对 userData） |
|------|----------------------|
| SQLite | `local-server/data/local/memoh.db` |
| 容器数据根 | `local-server/data/local` |
| bridge runtime | `local-server/data/runtime` |
| Qdrant 向量 | `qdrant/storage/` |
| Qdrant 端口 | `qdrant/ports.json` |
| server pid | `local-server.pid.json` |
| server 日志 | `local-server.log` |
| config | `config.toml` |
| CLI token | `cli-token.json` |

命名 slot 下 SQLite 隔离到 `local-server/data/instances/<slot>/memoh.db`，pid/log/config 带 `.<slot>` 后缀（`local-server.ts:50-66, 178-184`）。但 **Qdrant 不分 slot**——所有 slot 共用一个 Qdrant 进程和 storage，只有 bot id 重复且都写 memory 时才会交叉污染（`AGENTS.md:365-368`）。

### 与 server 模式（部署）的隔离

部署模式下，server 跑在远程，SQLite 可能换成 Postgres（见 [[12_数据库双后端]]），Qdrant 是远程集群。桌面端把这些全拉回本地：SQLite 文件、单机 Qdrant、本地 bridge。两者不共享数据——桌面端是独立的本地实例，不是远程 server 的客户端（除了在线版 Memoh 桌面 app）。

## 设计动机与取舍

**为什么 Electron 主进程管 server 生命周期，而不是让用户自己起？** 用户不需要懂 Go 编译、Qdrant 配置、端口分配。桌面把这一整套封进 `ensureLocalServer`，启动即用。代价是主进程要承担进程管理责任——pid 文件、健康检查、端口冲突、脏迁移修复都得它处理。

**为什么内嵌 Qdrant 而不是用 SQLite 做向量检索？** Qdrant 是专用向量数据库，检索质量和性能远超 SQLite 扩展。内嵌方式让用户零配置拥有向量检索，代价是包体变大（每平台一份二进制）、多管一个进程。per-user 目录隔离保证多用户/多实例不互踩，端口动态抢占避免和已装 Qdrant 撞。

**为什么 CLI 和桌面共享 pid 文件而不是各自管各自的？** 用户可能从桌面起 server（开机自启），也可能从终端 `memoh start` 起。共享 pid 文件让两边都能 stop/restart 对方起的进程，不会出现"桌面起的 server 终端杀不掉"的僵局。代价是 JSON 结构必须跨语言严格一致——TS 和 Go 各写一份，靠注释互相提醒（`service.go:17-20`）。

**为什么 dev slot 不隔离 Qdrant？** 隔离 Qdrant 要 per-slot 二进制实例 + 端口 + storage，开发场景下收益不大（slot 主要解决 SQLite 迁移冲突）。AGENTS.md 明确列为 v1 限制（`AGENTS.md:365-368`）。

**为什么手写 TOML 改写器而不是用库？** `setTomlString` 只改已知 key，行级正则替换足够。引入 TOML 库要加依赖、要处理序列化保留格式（注释、空行），收益不抵成本。代价是 key 不存在时抛错，要求模板和改写器同步演进。

## 验证问题

1. **桌面端与 server 模式的数据隔离：Memoh vs Memoh Local 的 userData 分别在哪？谁写什么？**
   答：本地优先桌面 app 叫 `Memoh Local`，userData 在 `~/Library/Application Support/Memoh Local`（macOS），内嵌 server + Qdrant，存 SQLite/Qdrant/config/pid 全套本地数据。在线桌面 app 叫 `Memoh`，userData 在 `~/Library/Application Support/Memoh`，连远程 server，不存业务数据。Go CLI 硬编码 `productName = "Memoh Local"`（`internal/tui/local/paths.go:18`），只认本地优先那套。两者 productName 必须两侧同步改，否则 CLI 和桌面读写不同 userData 静默分叉（`AGENTS.md:516-525`）。

2. **内嵌 Qdrant 的 per-user 进程管理：端口、pid、storage 分别落在哪？冲突时怎么办？**
   答：全部落在 `userData/qdrant/`：`ports.json`（http+grpc 端口对）、`qdrant.pid.json`、`config.yaml`、`storage/`、`qdrant.log`（`qdrant.ts:35-57`）。启动策略是"持久化端口优先，冲突则换新"：先读 `ports.json` 抢端口，`EADDRINUSE` 就 `reserveQdrantPorts()` 传 0 让 OS 分配新端口，写回 `ports.json`，最多重试 3 次（`qdrant.ts:326-363`）。spawn 时先释放端口预约再起进程，避免 Qdrant bind 时撞到自己的预约（`qdrant.ts:248-249`）。健康检查走 `/healthz`，30s 超时。复用逻辑：pid 活且 `/healthz` 通就直接认领不重启（`qdrant.ts:373-381`）。

3. **`ensureLocalServer` 发现 18731 上已有 server 但不是本桌面管的，会怎么做？为什么不能直接复用？**
   答：抛错提示用户先停掉（`local-server.ts:682-684`）。因为那个 server 没挂本桌面的内嵌 Qdrant，复用会导致 memory 写到错的地方。只有 pid 文件证明是本桌面管的才会重启它以挂上 Qdrant（`local-server.ts:678-680`）。

4. **`runMigrations` 失败时什么情况下会自动修复？怎么修？**
   答：当输出含 `Dirty database version 2` 时，先跑 `migrate force 2` 强制把版本标记为 2，再重试 `migrate up`（`local-server.ts:567-577`）。这是针对已知脏状态的定向修复，不是通用重试——其他迁移失败直接抛错。

5. **`memoh stop` 怎么杀掉桌面起的 server？跨语言契约靠什么？**
   答：靠 `ManagedPid` JSON 结构一致。桌面 `daemon.ts` 和 CLI `service.go` 都写 `{pid, command, startedAt}`（`daemon.ts:3-7`，`service.go:17-24`）。`memoh stop` 读 `userData/local-server.pid.json` 拿 pid，`IsAlive` 检查存活，SIGTERM 等 5s，没退出就 SIGKILL（`service.go:216-243`）。桌面 `stopManagedServer` 走同样路径（`local-server.ts:652-664`）。两边的 pid 文件路径都解析到同一个 `userData/local-server.pid.json`。

## Connections

- [[07_容器工作空间]] — 桌面端 `[container]` 配置驱动容器工作空间，bridge runtime 是容器内运行的 Linux 二进制，dev 下交叉编译 GOOS=linux。容器工作空间用 Docker（本地 desktop config 默认），本地工作空间是可信主机文件夹（`AGENTS.md:308-313`）。
- [[12_数据库双后端]] — 桌面端固定用 SQLite（`config.toml` 的 `[sqlite].path`），部署模式可换 Postgres。`runMigrations` 跑的是同一套迁移，双后端兼容在此体现。
- [[08_多渠道适配]] — 桌面 CLI（`memoh chat`/`memoh tui`）是 Memoh 多渠道接入的一种，和 Web、API 等渠道共享同一套 server HTTP/WebSocket 协议。provider OAuth 回调代理（1455 端口）是渠道适配的配套设施。
