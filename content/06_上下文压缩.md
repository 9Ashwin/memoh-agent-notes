# 06 — 上下文压缩

> LLM 的上下文窗口是有限的「工作记忆」。一轮轮对话堆下去，历史消息会把窗口撑爆——要么请求被截断、要么费用失控、要么模型注意力被稀释。Memoh 的上下文压缩（compaction）做的事很直接：当历史累积到阈值，调用一个独立的 LLM 把「旧消息」摘要成一段短文本，下次组装上下文时用这段摘要替代原始消息。本质是「用一次 LLM 调用换长期的 token 节省」，并且**原消息不删、只标记**，摘要失败还能降级回原始消息。这一篇拆清楚：何时触发、怎么切分要压缩的消息、摘要怎么存、原消息怎么处理。

## 核心结论

1. **压缩 = LLM 摘要旧消息 + 标记原消息**。原消息物理保留，只更新 `compact_id` 外键；加载时同组消息被替换成一条 `<summary>` 摘要消息（`internal/conversation/flow/resolver_history.go:200` 的 `replaceCompactedMessages`）。
2. **三条触发路径**：异步阈值触发（`maybeCompact`）、同步阈值触发（`runCompactionSync`）、手动触发（HTTP API + `/compact` 命令）。
3. **两套切分策略**：异步用 `splitByRatio`（默认压前 80%），同步用 `splitByTarget`（保留最新 ~2000 token，其余全压）。
4. **摘要存独立表** `bot_history_message_compacts`，带状态机 `pending/ok/error`、usage、model_id，是一次可审计的「压缩作业」记录。
5. **设计取向是「宁可保留也不丢」**：原消息保留可回溯、compact 记录删除时消息外键 `ON DELETE SET NULL` 自动「解压」、摘要为空或失败时降级用原消息。

## compaction service 与 prompt

### Service 结构

`internal/compaction/service.go:20` 定义 `Service`，只持有 `queries`（sqlc 生成的 DB 接口）、`hookService`（前置/后置钩子）、`logger`。无状态，所有上下文通过 `TriggerConfig` 传参。

`TriggerConfig`（`internal/compaction/types.go:30`）是压缩作业的完整入参：BotID/SessionID/ModelID/凭证/`Ratio`/`TotalInputTokens`/`MaxCompactTokens`/`TargetTokens`/`PromptCacheTTL`。其中 `TargetTokens > 0` 表示同步模式（压到目标 token 数），否则走异步比例模式。

### 入口方法

- `TriggerCompaction`（`service.go:44`）：**异步**，`go func()` 起 goroutine，用 `context.WithoutCancel(ctx)` 脱离请求生命周期。失败只记日志。
- `RunCompactionSync`（`service.go:54`）：**同步**，阻塞返回 error。
- `runCompaction`（`service.go:58`）：实际执行。流程是：`PreCompact` 钩子 → 建 log 行 → `doCompaction` → `PostCompact` 钩子（defer）。钩子 `DecisionDeny` 会直接 abort。

### 摘要生成的 prompt

`internal/compaction/prompt.go:8` 的 `systemPrompt` 是一段硬编码英文指令，要求摘要保留：

- 关键事实、决策、共识
- 用户偏好与请求
- 继续对话所需的重要上下文
- 名字、日期、数字、具体细节
- 工具调用结果（只留关键产出，忽略中间步骤/错误）

一个关键约束：当存在 `<prior_context>`（之前几次压缩的摘要）时，**只作为参考理解对话流，不得在输出中重复或改写其内容**——避免摘要被反复「折叠」后信息失真。这通过 `buildUserPrompt`（`prompt.go:26`）把 priorSummaries 用 `---` 拼接包进 `<prior_context>` 标签实现。

摘要调用走 Twilight SDK 的 `sdk.GenerateTextResult`（`service.go:212`），并经 `models.ApplyPromptCache`（`service.go:207`）套上 prompt 缓存以降低成本。

## 触发时机与调用链

### 路径一：异步阈值触发（`maybeCompact`）

`internal/conversation/flow/resolver_compaction.go:15` 定义 `maybeCompact`，调用点有四处：

| 调用点 | 时机 | 入参 |
|---|---|---|
| `resolver.go:535` | `Chat()` 同步对话成功后 | `result.Usage.InputTokens` |
| `resolver_stream.go:473` | stream 终态快照持久化后 | `extractInputTokensFromUsage(snap.usage)` |
| `resolver_stream.go:530` | stream 中断但有部分输出，持久化后 | `rc.estimatedTokens` |
| `resolver_stream.go:548` | stream 中断且无部分输出 | `rc.estimatedTokens` |

后两个调用点（`persistPartialResult`，`resolver_stream.go:497`）很关键：**失败路径也触发压缩**。注释（`resolver_stream.go:526-528`）写明动机——防止超大上下文死锁（LLM 永远失败 → 永远到不了成功后的压缩触发 → 上下文只增不减）。

`maybeCompact` 的触发条件（`resolver_compaction.go:25-38`）：
1. `botSettings.CompactionEnabled == true`
2. `botSettings.CompactionThreshold > 0`（migration 默认 100000，`db/postgres/migrations/0040_compaction.up.sql:27`）
3. `ShouldCompact(inputTokens, threshold)` 即 `inputTokens >= threshold`（`service.go:39`）

满足后调 `buildCompactionConfig`（`resolver_compaction.go:109`）解析压缩模型与凭证，若 `cfg.ModelID == ""`（未配模型或模型被禁用）则静默跳过，最后 `r.compactionService.TriggerCompaction(ctx, cfg)` 异步触发。

### 路径二：同步阈值触发（`runCompactionSync`）

`resolver.go:372-401` 在 `resolve()` 加载消息后判断：

```
compactionThreshold = contextTokenBudget * 70 / 100
if estimatedTokens >= compactionThreshold {
    r.runCompactionSync(ctx, req, estimatedTokens)
    // 重新加载消息、replaceCompactedMessages、stripToolMessages
}
```

注意这里是 **contextTokenBudget 的 70%**，不是 `botSettings.CompactionThreshold`。`contextTokenBudget` 是用户配置的「单次请求上下文上限」，而 `CompactionThreshold`（默认 10 万）是异步触发线。两条线独立：同步线防「这次请求就要爆」，异步线防「历史累积过大」。

同步模式特有配置（`resolver_compaction.go:161-164`）：
- `cfg.MaxCompactTokens = compactModel.Config.ContextWindow * 90 / 100`：压缩输入上限为压缩模型窗口的 90%。
- `cfg.TargetTokens = 2000`：**激进压缩**，只保留最新约 2000 token 的消息，其余全压。注释（`resolver_compaction.go:162-163`）说明：细节靠 `memory_read`/`search` 工具按需取回，摘要只提供参考上下文。

同步压缩后还会 `stripToolMessages`（`resolver.go:400`）——进一步剥掉体积大的 tool 消息，因为摘要已涵盖工具产出。

### 路径三：手动触发

- **HTTP**：`POST /bots/:bot_id/sessions/:session_id/compact`（`internal/handlers/compaction.go:58` 注册，`:137` 处理）。同步执行，返回最新 log 的 status/summary/message_count。
- **命令**：`/compact run`（`internal/command/compact.go:29`），同样同步。

手动触发的 `Ratio: 100`、`TotalInputTokens: 1`（`command/compact.go:108-109`、`handlers/compaction.go:212-213`）——`TotalInputTokens` 设 1 是为了让 `ShouldCompact` 必过，但手动路径其实不检查 `ShouldCompact`，直接走 `RunCompactionSync`。Ratio=100 意味着 `splitByRatio` 返回全部消息（`service.go:334`）。

## 压缩摘要存储（bot_history_message_compacts 表）

### Schema

PostgreSQL（`db/postgres/migrations/0040_compaction.up.sql:5`）：

| 列 | 类型 | 说明 |
|---|---|---|
| id | UUID PK | `gen_random_uuid()` |
| bot_id | UUID NOT NULL | 外键 bots，`ON DELETE CASCADE` |
| session_id | UUID | 外键 bot_sessions，`ON DELETE SET NULL` |
| status | TEXT | `pending/ok/error`，CHECK 约束，默认 pending |
| summary | TEXT | 摘要正文，默认 `''`（非 NULL） |
| message_count | INTEGER | 本次压缩的消息数 |
| error_message | TEXT | 失败原因 |
| usage | JSONB | 压缩 LLM 调用的 token usage |
| model_id | UUID | 外键 models，`ON DELETE SET NULL` |
| started_at / completed_at | TIMESTAMPTZ | 起止时间 |

索引 `idx_compacts_bot_session(bot_id, session_id, started_at DESC)` 支持按会话倒序查 log。

SQLite 版本在 `db/sqlite/migrations/0001_init.up.sql:967`，结构同构（TEXT 主键、`CURRENT_TIMESTAMP` 默认值），双后端字段对齐。

### 与消息表的关联

`0040_compaction.up.sql:22` 给 `bot_history_messages` 加列：

```
ALTER TABLE bot_history_messages ADD COLUMN compact_id UUID
  REFERENCES bot_history_message_compacts(id) ON DELETE SET NULL;
```

外键 `ON DELETE SET NULL` 是关键：**删除 compact 记录时，被压缩的消息自动「解压」回 `compact_id = NULL`**，下次加载又会作为原始消息出现。这就是「原消息不删」设计的数据层保障。

### sqlc 查询

`db/postgres/queries/compaction_logs.sql` 定义了 7 个查询：`CreateCompactionLog`（插 pending 行）、`CompleteCompactionLog`（更新 status/summary/usage/completed_at）、`GetCompactionLogByID`、`ListCompactionLogsByBot`（分页）、`CountCompactionLogsByBot`、`ListCompactionLogsBySession`（按会话正序，用于取 priorSummaries）、`DeleteCompactionLogsByBot`。

消息侧两个关键查询（`db/postgres/queries/messages.sql:1107`）：

- `MarkMessagesCompacted :exec`：`UPDATE bot_history_messages SET compact_id = $1 WHERE id = ANY($2::uuid[])`——批量打标。
- `ListUncompactedMessagesBySession :many`：用递归 CTE 从 `bot_sessions.default_head_turn_id` 往回遍历 `bot_history_turns` 父链，只选 `compact_id IS NULL` 的消息。注释（`messages.sql:1113-1114`）强调：「Compaction uses the session's server-canonical default head, not a client's transient selected variant」——压缩跟着服务端 canonical head 走，不被客户端临时选的 variant 干扰。

## 压缩后原消息处理

这是整个子系统最值得记的设计。分三步：

### 第 1 步：压缩时打标，不删原消息

`service.go:225` 调 `MarkMessagesCompacted`，把 `toCompact` 里所有消息的 `compact_id` 设为本次 log 的 ID。原消息内容、role、usage 全部原样保留在 `bot_history_messages` 表里。

### 第 2 步：加载时替换为摘要

`replaceCompactedMessages`（`internal/conversation/flow/resolver_history.go:200`）在 `resolve()`（`resolver.go:365`）、`resolve_user_input`（`resolver_user_input.go:316`）、`resolve_tool_approval`（`resolver_tool_approval.go:301`）三处加载消息后调用。逻辑：

1. 扫描已加载消息，按 `compact_id` 分组（`resolver_history.go:205-210`）。
2. 对每个非空 `compact_id`，调 `GetCompactionLogByID` 取 log；只接受 `status == "ok" && summary != ""` 的（`:226`）。
3. 遍历消息：无 `compact_id` 的原样保留；有 `compact_id` 的，**每组只输出一条** role=user 的 `<summary>\n{summary}\n</summary>` 消息（`:249-254`），其余同组消息跳过。
4. **降级**：如果某个 `compact_id` 的 log 不存在、status 非 ok、或 summary 为空，则该组的原始消息全部保留不替换（`:243-247`）。

注意摘要被包成 `user` 角色注入——没有引入新的 system 消息，保持消息序列的 role 简单。

### 第 3 步：同步压缩后还会剥 tool 消息

`resolver.go:400` 在同步压缩重载消息后调 `stripToolMessages`，把体积大的 tool 消息从最近上下文里剥掉（摘要已涵盖工具产出）。

### 切分策略细节

`doCompaction`（`service.go:132`）选消息的流程：

1. `ListUncompactedMessagesBySession` 取所有未压缩消息（按 turn 深度倒序，即最旧的在前）。
2. 若 `cfg.TargetTokens > 0`（同步模式）→ `splitByTarget`：从最新往老累加 token，超过 `TargetTokens` 的更老消息全部压。
3. 否则（异步/手动）→ `splitByRatio`：算出要保留的 token 数 `keepTokens = totalInputTokens * (100 - ratio) / 100`，从最新往老累加，超过 keepTokens 的更老消息全部压。ratio=100 时全压。
4. `trimCompactMessages`（`service.go:401`）：如果待压消息总 token 超过 `MaxCompactTokens`（默认 30000，或压缩模型窗口的 90%），从最旧开始丢弃，避免压缩输入本身撑爆压缩模型的窗口。

token 估算（`estimateRowTokens`，`service.go:389`）：优先用消息自带的 `usage.output_tokens`，没有则按 `len(content) / 4` 粗估。

## 设计动机与取舍

**为什么原消息保留？** 可审计、可回溯、可「解压」。删除是不可逆的——一旦摘要写错了，历史就真没了。保留原消息 + 外键 `ON DELETE SET NULL` 让「删除 compact 记录」成为无损的回滚操作。代价是存储空间，但对话文本相对便宜。

**为什么摘要作为 user 消息注入？** 不引入新 role、不污染 system prompt、不需要改下游 provider 序列化逻辑。代价是失去 role 信息（原 user/assistant/tool 混在一起被压成一段），但摘要本身已用「Role: Content」格式保留了角色（`prompt.go:38`）。

**为什么同步线用 70% 而不是 100%？** 注释（`resolver.go:370-371`）说：在边缘 case 下 100% 阈值会超时。70% 留出余量，让压缩在「还没爆」时就把上下文压回去。

**为什么同步模式 TargetTokens=2000 这么激进？** 注释（`resolver_compaction.go:162-163`）说：细节靠 `memory_read`/`search` 工具按需取回，摘要只做参考。这是「摘要 + 工具召回」的组合拳，而不是「摘要要尽可能完整」。

**为什么失败路径也触发压缩？** 防死锁。超大上下文 → LLM 调用失败 → 没有成功路径 → 永远不触发异步压缩 → 上下文只增不减。`persistPartialResult` 里显式触发 `maybeCompact`（`resolver_stream.go:530, 548`）打破这个循环。

**取舍：摘要质量依赖压缩模型。** 没配 `CompactionModelID` 时，`buildCompactionConfig` 返回空 cfg 静默跳过（`resolver_compaction.go:53-58`）；模型被 `Enable=false` 也跳过（`:124-130`）。压缩模型和对话模型可以不同——可以用便宜模型压缩、用强模型对话。

## 验证问题

1. **何时触发压缩？** 三条路径：(a) 异步——`Chat`/stream 成功或失败持久化后，`inputTokens >= botSettings.CompactionThreshold`（默认 10 万）且 `CompactionEnabled`；(b) 同步——`resolve()` 加载消息后 `estimatedTokens >= contextTokenBudget * 70%`，阻塞压缩后重载；(c) 手动——HTTP `POST .../compact` 或 `/compact run` 命令。
2. **压缩后原消息如何处理？** 原消息**不删除**，`MarkMessagesCompacted` 只把 `bot_history_messages.compact_id` 设为本次 log 的 ID。加载时 `replaceCompactedMessages` 把同组消息替换成一条 user 角色的 `<summary>` 消息；若 log 非 ok 或 summary 为空则降级保留原消息。删除 compact 记录时外键 `ON DELETE SET NULL` 让消息自动「解压」。
3. **`splitByRatio` 和 `splitByTarget` 的区别？** 前者按比例（异步/手动，默认压前 80%），后者按目标 token 数（同步，保留最新约 2000 token）。两者都从最新消息往老累加 token 决定切分点。
4. **`MaxCompactTokens` 是什么？** 压缩输入的上限，防止待压消息本身撑爆压缩模型窗口。取压缩模型 `ContextWindow * 90%`；未配置则保守默认 30000。超出时 `trimCompactMessages` 从最旧开始丢弃。
5. **`<prior_context>` 在 prompt 里起什么作用？** 注入之前所有压缩摘要作为参考，帮助 LLM 理解对话流，但 prompt 明确禁止在输出中重复或改写其内容——避免摘要被反复折叠后信息失真。
