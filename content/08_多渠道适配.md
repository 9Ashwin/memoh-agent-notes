# 08 — 多渠道适配

> **1 分钟叙事**：Memoh 要把同一个 Agent 接到 Telegram、飞书、Discord、QQ、企微、Matrix 等十几个 IM 平台上。这些平台的接入方式天差地别——Telegram 走长轮询，飞书走 WebSocket 或 Webhook 回调，企微走回调加密，Matrix 走 `/sync` 长连接；它们的配置schema 不同（Bot Token vs App ID/Secret）、消息格式不同（Markdown vs 富文本 Post）、能力不同（Telegram 能编辑/撤回消息，飞书能加表情回应但不能撤回）、身份体系也不同（Telegram 的 user_id vs 飞书的 open_id）。如果在 Agent 内核里写满 `if channel == "telegram"` 的分支，代码会迅速腐烂。
>
> Memoh 的解法是「**统一抽象层 + 能力声明 + 三层身份模型**」：所有平台差异被压进一个 `Adapter` 接口及其一组可选能力接口（`Sender`/`Receiver`/`StreamSender`/`MessageEditor`/`Reactor`…），`Registry` 做中央调度，`Manager` 管连接生命周期和消息收发，`Lifecycle` 协调配置持久化与运行时连接的事务性。身份则分三层：`identities` 把平台原生 subject 归一为全局 `channel_identity_id`，`channelaccess`（bind 层）把这个身份绑定到 Web 账号 user_id，`acl` 基于来源会话类型做 `chat.trigger` 放行/拒绝。Agent 内核只面对统一的 `InboundMessage`/`OutboundMessage`，永远不需要知道对面是哪个平台。

## 核心结论

1. **适配器是「基础接口 + 可选能力」的组合**：`Adapter` 只强制 `Type()` + `Descriptor()`（adapter.go:79-82），其余能力（`Sender`、`StreamSender`、`Receiver`、`WebhookReceiver`、`MessageEditor`、`Reactor`、`SelfDiscoverer`…）全部是可选接口，由各适配器按平台实际支持情况实现。`Registry` 通过类型断言按需取出（registry.go:230-307）。这让「飞书不支持撤回但支持表情回应」这类差异不污染核心。

2. **`Descriptor` 是静态能力声明表**：每个适配器在 `Descriptor()` 里一次性声明 `Capabilities`（文本/Markdown/富文本/附件/流式/编辑/撤回…）、`OutboundPolicy`（分块上限、chunker 模式）、`ConfigSchema`（前端渲染表单用）、`TargetSpec`（投递目标格式提示）（adapter.go:87-97）。出站流水线在发消息前用这些声明做能力校验和分块。

3. **连接生命周期是声明式 reconcile**：`Manager` 每 5 分钟全量 reconcile 一次（manager.go:213-230），把 DB 里的 `ChannelConfig` 与内存中的 `connectionEntry` 对账——新增的启动、变更的重启、删除的停止。`EnsureConnection` 做单点精准启停（connection.go:213-221）。`Lifecycle` 在配置写入和连接启停之间做事务性协调，失败回滚（lifecycle.go:45-81）。

4. **入站是异步队列 + worker 池**：适配器收到消息后调 `handler`，`Manager.handleInbound` 把任务塞进 256 容量的 `inboundQueue`，4 个 worker goroutine 异步消费（inbound.go:24-83）。队列满时返回 `ErrInboundQueueFull` 而不是阻塞适配器的长连接。

5. **身份是三层解耦**：`identities` 做「平台原生 subject → 全局 channel_identity_id」的无条件归一（identities/service.go:101-129）；`channelaccess` 做「channel_identity_id → user_id」的账号绑定（通过 `/link <code>` 命令消费一次性码，channelaccess/service.go:182-209）；`acl` 基于_bot + 身份 + 来源会话_做 `chat.trigger` 放行（acl/service.go:44-77）。三层各管一件事，互不耦合。

6. **ACL 是「默认效果 + 规则覆盖」模型**：每个 bot 有一个默认效果（allow/deny），规则只在对应模式下生效——黑名单模式下只有 deny 规则起作用，白名单模式下只有 allow 规则起作用（acl/service.go:41-43）。规则可以按 `channel_identity_id`、`subject_channel_type`、`source_scope`（会话类型/会话ID/线程ID）精确定位。

## 统一抽象层（adapter/service/manager/registry）

### Adapter 接口与能力声明

所有渠道适配器必须实现的基础接口极简（adapter.go:79-82）：

```go
type Adapter interface {
    Type() ChannelType
    Descriptor() Descriptor
}
```

`Descriptor` 是只读元数据容器（adapter.go:87-97），承载五类信息：

| 字段 | 用途 |
|---|---|
| `Capabilities` | 平台能力矩阵，出站前用来校验消息内容是否被支持 |
| `OutboundPolicy` | 分块策略：文本上限、富文本上限、chunker 模式、媒体顺序 |
| `ConfigSchema` | Bot 维度配置表单（如 Telegram 的 botToken、飞书的 appId/appSecret） |
| `UserConfigSchema` | 用户绑定配置表单（如 Telegram 的 chat_id、飞书的 open_id） |
| `TargetSpec` | 投递目标格式提示，供前端 UI 渲染 |

真正的请求处理通过**可选能力接口**表达（adapter.go:99-195）。核心的几个：

- `Sender` / `StreamSender`：一次性发送 / 流式发送
- `Receiver`：长连接接收（返回 `Connection`）
- `WebhookReceiver`：HTTP 回调接收
- `MessageEditor`：编辑/撤回已发消息
- `Reactor`：表情回应
- `SelfDiscoverer`：从平台反查 Bot 自身身份
- `ConfigNormalizer` / `TargetResolver` / `BindingMatcher`：配置归一化、目标解析、绑定匹配

这种设计的好处是：核心代码通过 `Registry.GetSender(channelType)` 拿到的是 `(Sender, bool)`，不支持就降级或报错，**永远不会调到不存在的方法**。

### Registry：中央调度器

`Registry` 是一个 `map[ChannelType]Adapter` 加 RWMutex（registry.go:15-18）。它对外提供两类方法：

1. **适配器存取**：`Register` / `Unregister` / `Get` / `List` / `Types`（registry.go:28-106）
2. **能力分发**：`GetSender` / `GetStreamSender` / `GetReceiver` / `GetWebhookReceiver` / `GetReactor` / `GetMessageEditor` 等——每个都是「先 `Get` 拿适配器，再类型断言取可选接口」（registry.go:230-307）

`Registry` 还封装了配置相关的分发：`NormalizeConfig` / `NormalizeUserConfig` / `ResolveTargetFromUserConfig` / `MatchUserBinding` / `BuildUserBindingConfig`（registry.go:352-431）。这些方法让核心代码不需要知道具体渠道，只通过 Registry 这个统一入口调用。

### Manager：连接生命周期 + 消息收发协调器

`Manager` 持有 registry、service（配置存储）、processor（入站处理器）、连接池、入站队列（manager.go:61-79）。它的职责分三块：

**连接生命周期**（connection.go + lifecycle.go）：
- `Start` 启动周期性 reconcile 循环（默认 5 分钟）+ 入站 worker 池（manager.go:208-230）
- `refresh` 全量对账：遍历所有渠道类型，从 DB 拉配置，调 `reconcile`（connection.go:16-36）
- `reconcile` 做 diff：active 集合里的 `ensureConnection`，不在 active 里的 `Stop`（connection.go:38-92）
- `ensureConnection` 处理配置变更：若 `UpdatedAt` 变了，先 Stop 旧连接再 Connect 新连接，带双重检查防并发竞态（connection.go:94-209）

**入站分发**（inbound.go）：
- `HandleInbound` 是适配器回调的入口，塞队列后立即返回（inbound.go:25-43）
- `handleInbound` 在 worker 里执行：构造 `StreamReplySender`，调 `processor.HandleInbound`（inbound.go:45-57）

**出站分发**（outbound.go + manager.go:233-289）：
- `Send` 解析 target（从 `ChannelIdentityID` 反查用户绑定配置）、做能力校验、分块、调 `sender.Send`
- `newReplySender` 构造一个绑定了当前 cfg 的 `StreamReplySender`，供入站处理器回包用（outbound.go:679-689）

### Service / Lifecycle：配置持久化与事务协调

`Store`（service.go:28-31）封装 DB CRUD，关键方法：
- `UpsertConfig`：写配置前先 `NormalizeConfig`，若适配器声明 `RequireDiscoveryOnEnable` 还会调 `DiscoverSelf` 反查 Bot 身份（service.go:42-134）
- `ResolveEffectiveConfig`：对 configless 渠道（如 local）返回合成配置（service.go:355-384）
- `ResolveChannelIdentityBinding`：遍历某渠道的所有用户绑定，用 `MatchUserBinding` 找到匹配的 channel_identity_id（service.go:492-506）

`Lifecycle`（lifecycle.go:29-32）在 `Store` 和 `ConnectionController` 之间做事务协调：

```
UpsertBotChannelConfig:
  1. 读 previous 配置
  2. store.UpsertConfig 写入 DB
  3. 若 disabled → controller.RemoveConnection
  4. 若 enabled → controller.EnsureConnection
  5. 若 EnsureConnection 失败 → rollback（恢复 previous 或删除）
```

这个回滚机制是关键——**配置写入成功但连接启动失败时，DB 不会被留在「配置说已启用但实际没连接」的不一致状态**（lifecycle.go:74-81、136-160）。

## inbound/outbound 消息流转

### 入站流向

```
平台消息
  ↓
适配器 Connect 内部（长轮询/WebSocket/Webhook）
  ↓ 构造 InboundMessage
Manager.handleInbound (inbound.go:45)
  ↓ 构造 StreamReplySender
  ↓ processor.HandleInbound
入站队列 (inboundQueue, 容量 256)
  ↓
worker 池 (4 个 goroutine)
  ↓ handleInbound
ChannelInboundProcessor (inbound/channel.go)
  ↓ IdentityResolver.Resolve (两阶段身份解析)
  ↓ ACL gate (acl.Evaluate)
  ↓ SessionEnsurer / Agent 流式回复
```

`InboundMessage` 是统一信封（types.go:69-80），携带 `Channel`、`Message`、`BotID`、`ReplyTarget`、`Sender Identity`、`Conversation`。`RoutingKey()` 生成稳定路由键：`platform:bot_id:conversation_id[:sender_id]`，群聊才追加 sender_id（types.go:84-106）。

入站队列满时返回 `ErrInboundQueueFull`（inbound.go:37-42），**不阻塞适配器的接收循环**——这对长连接平台很重要，避免因下游处理慢导致 WebSocket 断开。

### 出站流向

```
Agent 产生回复
  ↓
StreamReplySender.Send / OpenStream (outbound.go:699-760)
  ↓ resolveOutboundTarget (别名解析)
  ↓ GetOutboundCapabilities (能力校验)
  ↓ buildOutboundMessagesWithCaps (分块)
  ↓ sender.Send / streamSender.OpenStream
适配器翻译为平台 API 调用
```

出站的关键是**能力感知的分块**：`OutboundPolicy.TextChunkLimit` / `RichTextChunkLimit` 控制单条消息长度（Telegram 4096/32768，飞书用 `feishuStreamMaxRunes`），`ChunkerMode` 决定按文本还是 Markdown 规则切分（outbound.go:36-81）。流式发送时 `managerOutboundStream` 还会做流分片、附件降级等处理（outbound.go:767-778）。

## 适配器对比（telegram vs feishu）

两者都实现 `Sender` + `StreamSender` + `Receiver`，但接入方式和能力差异显著。

### 配置与接入方式

| 维度 | Telegram | 飞书 |
|---|---|---|
| 凭据 | 单一 `botToken` | `appId` + `appSecret` + `encryptKey` + `verificationToken` |
| 网络代理 | `apiBaseURL`（反向代理）+ `httpProxyUrl` | `region`（feishu.cn / larksuite.com） |
| 入站模式 | 仅长轮询（`tele.LongPoller`，30s 超时） | `inboundMode`：`websocket`（默认）或 `webhook` |
| SDK | `gopkg.in/telebot.v4` | `github.com/larksuite/oapi-sdk-go/v3` |

telegram.go:30-38 定义了关键常量：长轮询超时 30s、媒体组收集窗口 700ms、update 去重 TTL 10 分钟。

飞书的 `Connect` 在 `inboundMode == webhook` 时直接返回空连接——Webhook 模式下接收走 HTTP handler，不需要长连接（feishu.go:374-379）。WebSocket 模式则启动 `wsclient.Run`，带自动重连（3s 退避）和**终端错误识别**：`larkws.ClientError`（鉴权失败）不重连，直接 `conn.MarkStopped()` 让 reconcile 感知到通道已死（feishu.go:412-460）。

### 能力差异

| 能力 | Telegram | 飞书 |
|---|---|---|
| 编辑/撤回 | ✅ `MessageEditor` | ❌ |
| 表情回应 | ❌ | ✅ `Reactor` + `ProcessingStatusNotifier`（用 Typing 表情表示处理中） |
| 按钮 | ✅ `Buttons` + inline keyboard | ❌（仅 URLButtons） |
| 富文本 | ✅ 但偏 Markdown | ✅ 重度支持（Post 富文本结构） |
| 流式 | ✅ `Streaming` + `BlockStreaming` | ✅ 同 |
| 文本上限 | 4096（text）/ 32768（rich） | `feishuStreamMaxRunes` |

飞书用「Typing 表情回应」代替 Telegram 的「编辑消息」来表示处理中状态——这是典型的**能力差异驱动交互设计**：Telegram 不断编辑一条「思考中…」的消息，飞书则给原消息加一个打字表情，处理完移除（feishu.go:192-200）。

### 入站消息构造

Telegram 的 `Connect` 里有**媒体组（相册）合并逻辑**：Telegram 把相册拆成多条独立 update 发送，适配器用 `telegramMediaGroupBuffer` + 700ms 窗口聚合后再投递（telegram.go:330-405）。还有 **update 去重**：`seenUpdates` map + 10 分钟 TTL，防止长轮询重复投递（telegram.go:416-424）。

飞书则通过 lark SDK 的 `eventDispatcher.OnP2MessageReceiveV1` 接收事件，提取 `extractFeishuInbound` 后还会做**发送者画像补全**（`enrichSenderProfile`）和**引用消息解析**（`enrichQuotedMessage`），因为飞书的 webhook 回调里不包含完整用户信息（webhook_handler.go:66-75）。

### 目标解析

Telegram 的 `TargetSpec` 是 `chat_id | @username`，`UserConfigSchema` 有 `username`/`user_id`/`chat_id`（telegram.go:234-248）。

飞书是 `open_id:xxx | user_id:xxx | chat_id:xxx`，`UserConfigSchema` 有 `open_id`/`user_id`（feishu.go:173-187）。`TargetResolver.ResolveTarget` 从用户绑定配置里提取投递目标。

## identities / bind / acl

三者构成「**归一身份 → 绑定账号 → 鉴权放行**」的流水线，职责严格分离。

### identities：渠道身份归一

`channel/identities/service.go` 提供 `(channel, channel_subject_id) → channel_identity_id` 的全局映射。

- `ResolveByChannelIdentity`：upsert by (channel, channel_subject_id)，返回 `ChannelIdentity`（identities/service.go:101-129）。这是**无条件**的——只要消息进来，就归一出一个全局 ID。
- `ListUserIDsByChannelIdentity`：反查这个身份绑定了哪些 user_id（identities/service.go:185-205）。注释明确说明：需要 runtime owner 时应要求恰好一个结果，未绑定或歧义绑定都不能安全持有 workspace runtime。

`ChannelIdentity` 结构（identities/types.go:6-15）持有 `Channel`、`ChannelSubjectID`、`DisplayName`、`AvatarURL`。这是**跨渠道**的身份实体——同一个真人可能在 Telegram 和飞书各有一个 ChannelIdentity，但绑定到同一个 user_id。

### bind（channelaccess）：身份 → 账号绑定

`channelaccess` 包实现「IM 身份绑定到 Web 账号」的流程：

1. 用户在 Web 端调 `IssueLinkCode` 生成一次性 token（channelaccess/service.go:143-178）
2. 用户在 IM 里向 Bot 发 `/link <token>`
3. Bot 调 `ConsumeLinkCode(token, channelIdentityID)` 消费 token，写入 `user_channel_identity_bindings` 表（channelaccess/service.go:182-209）

`Manager` 类型（channelaccess/types.go:29-46）表达**有效 Manage 状态**：本地 override（`HasOverride`）优先于继承授权（`Inherited`），`Bound` 标记是否绑定了本 workspace 成员。这是「Channel Access」权限模型的载体——即使某 ChannelIdentity 没绑账号，也可以通过本地 override 强制授予或撤销 Manage 权限。

### acl：来源感知的 chat.trigger 鉴权

`acl/service.go` 的 `Evaluate` 是入站鉴权的核心（acl/service.go:44-77）。它在 `ChannelInboundProcessor` 里被调用（inbound/channel.go:602-621），**在事件进入 pipeline 之前**判断是否放行：

```go
allowed, _ := p.acl.Evaluate(ctx, acl.EvaluateRequest{
    BotID:             identity.BotID,
    ChannelIdentityID: identity.ChannelIdentityID,
    ChannelType:       msg.Channel.String(),
    SourceScope: acl.SourceScope{
        ConversationType: channel.NormalizeConversationType(msg.Conversation.Type),
        ConversationID:   msg.Conversation.ID,
        ThreadID:         threadID,
    },
})
```

关键设计点：

1. **默认效果 + 规则覆盖**（acl/service.go:41-43 的注释）：bot 有 `default_effect`（allow/deny），规则只在对应模式生效。黑名单模式（default=allow）下只有 deny 规则有用；白名单模式（default=deny）下只有 allow 规则有用。这避免了「规则爆炸」——大部分情况用默认值，只对例外建规则。

2. **来源感知**：`SourceScope`（acl/types.go:64-68）可以按 `ConversationType`（private/group/thread）、`ConversationID`、`ThreadID` 精确匹配。`preset` 系统提供常用模板：`allow_all`、`private_only`、`group_only`、`group_and_thread_only`、`deny_all`（acl/presets.go:16-99）。

3. **被拒绝时不是静默**：`isDirectedAtBot(msg)` 为真（DM 或 @提及）时回复一条 access/bind 提示，群聊里的无指向消息则静默丢弃，避免刷屏（inbound/channel.go:623-653）。

4. **Manage 权限继承**：`channelaccess.inheritedManage` 遍历该 ChannelIdentity 绑定的所有 user_id，只要有一个在本 bot 上有 `PermissionManage`，就继承 Manage 权限（channelaccess/service.go:102-128）。本地 override 可以强制覆盖这个继承值。

### 三者协作的完整链路

```
消息进入
  ↓
IdentityResolver.Resolve (inbound/identity.go:122-181)
  Phase 1: ResolveByChannelIdentity → channel_identity_id（无条件归一）
  Phase 2: ListUserIDsByChannelIdentity → user_id（若唯一绑定）
  ↓ owner bypass: 若 user_id == bot owner，直接放行
  ↓ 否则交给 ACL
ACL gate (inbound/channel.go:607-621)
  Evaluate(bot, channel_identity_id, channel_type, source_scope)
  ↓ allow → 进入 pipeline
  ↓ deny  → persistPassiveMessage + 可选提示回复
```

## 设计动机与取舍

**为什么用可选接口而不是大而全的 Adapter 接口？** 如果 `Adapter` 接口包含 `Send`/`Edit`/`React`/`Stream` 所有方法，不支持的平台只能返回 `ErrNotSupported`，调用方必须处理这个无处不在的错误。可选接口让「不支持」在类型系统层面就被排除——`GetReactor` 返回 `(Reactor, false)`，调用方在调之前就知道不支持，代码路径更干净。

**为什么 config 存 DB 而不是配置文件？** 多租户需求：每个 bot 有独立的渠道配置，运行时通过 API 增删改。`Store` + `Lifecycle` 的事务协调保证「配置写入」和「连接启停」原子性，失败回滚避免不一致。

**为什么入站用队列而不是直接处理？** 适配器的接收循环（长轮询/WebSocket）必须快速 return，否则会导致平台重连或消息积压。256 容量的队列 + 4 worker 让下游处理慢时只是丢队列满的错误，不影响接收。代价是**背压不优雅**——队列满直接返回错误，不会反向限流平台。

**为什么身份分三层？** 平台身份模型差异巨大（Telegram user_id 是数字、飞书 open_id 是 `ou_xxx`、Matrix 是 `@user:server`），直接在 ACL 里写 `if telegram` 会爆炸。`identities` 把所有平台 subject 归一成一个 UUID，`channelaccess` 把 UUID 绑到 Web 账号，`acl` 只面对 UUID 和会话上下文。每层可以独立演进——比如未来加一个新平台，只需要写适配器 + 在 identities 里建映射，bind/acl 完全不用改。

**为什么 ACL 是默认效果 + 规则覆盖而不是全量规则？** 大部分 bot 的策略是「允许所有人」或「只允许私聊」，用 default effect 一行搞定。只有需要精细控制（如「允许 A 但禁止 A 在群 X 发言」）才建规则。这是「最小惊讶」的权限设计——简单场景简单配。

## 验证问题

1. **挑 Telegram 和飞书两个适配器，说清统一抽象层如何屏蔽它们的接入差异。**
   - 提示：从 `Connect` 入站方式（长轮询 vs WebSocket/Webhook）、`Descriptor.Capabilities`（Edit/Unsend vs Reactions）、`ConfigSchema`（botToken vs appId/appSecret）、`OutboundPolicy`（分块上限）四个维度对比。重点是 `Manager` 和 `Registry` 只面对统一接口，不感知这些差异。

2. **说清 identities / bind（channelaccess）/ acl 三者的职责边界和协作链路。**
   - 提示：identities 做「平台 subject → 全局 channel_identity_id」的无条件归一；channelaccess 做「channel_identity_id → user_id」的账号绑定（通过 `/link` 码）；acl 做「bot + 身份 + 来源会话」的 chat.trigger 放行。三者顺序：归一 → 绑定 → 鉴权。为什么不能合并？因为归一是无条件的全局映射，绑定是用户主动行为，鉴权是 bot 级策略——三者生命周期和触发条件不同。

3. **`Manager.ensureConnection` 在配置变更时如何避免新旧连接竞态？**
   - 提示：看 connection.go:94-209 的三重检查——入口检查 `UpdatedAt`、Stop 旧连接后再次检查、Connect 成功后第三次检查。为什么需要三重？因为 Stop 是异步的，期间另一个 goroutine 可能已经启了新连接。

4. **`Lifecycle.UpsertBotChannelConfig` 在连接启动失败时如何回滚？为什么要回滚？**
   - 提示：看 lifecycle.go:74-81、136-160。不回滚会导致 DB 里配置显示「已启用」但运行时没有连接，reconcile 会反复尝试启动并失败，造成「幽灵配置」。回滚恢复到 previous 状态或删除新配置，保持 DB 与运行时一致。

5. **ACL 的「默认效果 + 规则覆盖」模型在黑名单和白名单模式下分别如何生效？**
   - 提示：看 acl/service.go:41-43 注释和 presets.go。黑名单（default=allow）下 deny 规则才生效，allow 规则是噪音；白名单（default=deny）下 allow 规则才生效。`preset_private_only` = default deny + 一条 allow(private) 规则。
