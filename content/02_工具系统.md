# 02 — 工具系统

> 1 分钟叙事

Agent 要真正做事，光会说话不够，得会「调用工具」。但工具系统真正难的不是「怎么执行一个工具」，而是「怎么让模型在每一次请求里，恰好知道现在有哪些工具可用、每个工具该怎么用」。

Memoh 的工具集是高度动态的：同一个 bot，在 chat 模式下有 `send`/`react`，在 schedule 模式下文本输出被禁言；主代理能用 `spawn_agent`/`search_memory`，子代理却连 `memory` 都看不到；开了显示器的工作空间才有 `browser_action`/`computer_action`，没开 TTS 的 bot 绝不能让模型看到 `speak`。如果把这些工具的「用法说明」写死在 system prompt 里，就会出现一种典型 bug：工具没注册，prompt 却还在教模型怎么调它，模型于是幻觉调用一个根本不存在的工具。

Memoh 的解法一句话讲完：**工具用法住在工具里，不进静态 prompt**。每个 `ToolProvider` 在返回工具的同时，可以选择实现 `ToolUsage` 接口，按当前 session 的真实情况生成一段用法指南；`assembleTools` 只从「实际返回了工具」的 provider 那里收集用法，注入到 system prompt 的 `## Tool usage` 段。工具没注册，用法就凭空消失——模型永远看不到它不该看到的工具。这套机制由 `prompt_test.go` 的一组守卫测试锁死，谁敢把工具用法重新写回静态 prompt，CI 当场红给你看。

---

## 核心结论

1. **两个接口，一个角色**。`ToolProvider`（`tools/types.go:159`）负责「返回工具」，`ToolUsage`（`tools/types.go:213`）是可选的「用法指南」能力。一个 provider 可以只实现前者，但几乎所有内置 provider 都同时实现后者。

2. **用法与注册同源**。`assembleTools`（`agent.go:826`）只在 provider「本回合真的返回了工具」时才收集它的 `Usage()`（`agent.go:870-879`）。这保证用法指南和工具注册共享同一个 gating 决策，永远不会漂移。

3. **工具名不硬编码**。用法文本里要提别的工具，必须走 `available.Ref(ToolSend())` 这类方法（`toolset.go:31`），它会在工具确实可用时返回 `` `send` `` 这样的反引号引用，不可用就返回空。这从语法层面杜绝了「提到一个不存在的工具」。

4. **注入位置固定**。用法段被拼成 `## Tool usage\n\n...`，由 `appendToolUsageToSystem`（`agent.go:897`）插到 `## Workspace instruction files` 之前（`agent.go:906-908`），且仅当 `cfg.SupportsToolCall` 为真时才走这一步（`agent.go:153`、`agent.go:632`）——纯文本模型连工具都没有，更不需要用法指南。

5. **静态 prompt 片段已全部删除**。提交 `c78c3bee` "unify tool guidance with tool availability (#643)" 删掉了 `prompts/_tools.md`、`_memory.md`、`_contacts.md`、`_schedule_task.md`、`_subagent.md` 等五个静态片段，从此工具用法只有一个来源：`ToolUsage`。

## ToolProvider 接口与 assembleTools

### ToolProvider：按 session 返回工具

```go
// tools/types.go:159-161
type ToolProvider interface {
    Tools(ctx context.Context, session SessionContext) ([]sdk.Tool, error)
}
```

注释写得很清楚：「`Tools()` 每次请求都会被调用；实现方可以根据 session 上下文返回不同的工具集（例如子代理限制、bot 设置）」。这是整个动态性的根——工具列表不是启动时固定的，而是每轮生成前重新求值。

`SessionContext`（`tools/types.go:76-95`）是请求级身份包，携带 `SessionType`、`CurrentPlatform`、`ReplyTarget`、`IsSubagent`、`CanRequestUserInput`、`SupportsImageInput`、`Skills`、`Emitter` 等。它身上还挂了几个判断方法，是各 provider 做 gating 的依据：

- `CanAskUser()`（`types.go:99`）= `CanRequestUserInput && sessionmode.IsInteractive(SessionType)`
- `CanOmitMessagingTarget()`（`types.go:122`）：当前会话能否省略 `platform`/`target`，heartbeat/schedule 必须显式指定
- `CanUseLocalMessagingShortcut()`（`types.go:136`）：能否用 live stream 直推附件，而非走 channel sender

### assembleTools：收集工具 + 收集用法

`agent.go:826-895` 是核心装配函数。关键片段：

```go
// agent.go:864-880
for _, provider := range a.toolProviders {
    providerTools, err := provider.Tools(ctx, session)
    if err != nil { ... continue }
    if len(providerTools) == 0 {
        continue
    }
    allTools = append(allTools, providerTools...)
    // 只有「实际返回了工具」的 provider，才会被收集 Usage
    if usageProvider, ok := provider.(tools.ToolUsage); ok {
        usageRegistrations = append(usageRegistrations, usageRegistration{provider: usageProvider})
    }
}
```

注意 `if len(providerTools) == 0 { continue }` 这一行——它就是「用法与注册同源」的物理保证。一个 provider 即使实现了 `ToolUsage`，只要本回合没返回任何工具（比如 `MemoryProvider` 在 subagent session 下返回 `nil`），它的 `Usage()` 就永远不会被调用。

收集完工具后，再统一求值用法（`agent.go:884-889`）：先构造 `availableTools`（完整注册集），再把每个 provider 的 `Usage(ctx, session, availableTools)` 拼起来。这里有个顺序细节：**先收集全部工具，再统一调用 `Usage`**，这样 `Usage` 里拿到的 `available` 是当前回合的完整工具集，能安全地跨 provider 引用别的工具名。

最后包装成 `## Tool usage\n\n...`（`agent.go:891-893`）返回，由调用方注入 system prompt。

### 注入点与 SupportsToolCall 守卫

```go
// agent.go:153-168 (Stream)
if cfg.SupportsToolCall {
    var toolUsage string
    sdkTools, toolUsage, err = a.assembleTools(streamCtx, cfg, streamEmitter, cfg.LiveToolStream)
    ...
    if toolUsage != "" {
        // Must run before buildGenerateOptions so prompt caching and
        // background task summaries see the usage-augmented text.
        cfg.System = appendToolUsageToSystem(cfg.System, toolUsage)
    }
}
```

`Generate` 路径完全对称（`agent.go:632-642`）。两处都有 `cfg.SupportsToolCall` 守卫——不支持工具调用的模型连 `assembleTools` 都不进，用法段自然也不会出现。注释还点出一个时序约束：注入必须在 `buildGenerateOptions` 之前完成，否则 prompt caching 和 background 任务摘要看到的就不是含用法的文本，缓存 key 会对不上。

`appendToolUsageToSystem`（`agent.go:897-911`）的插入策略：找到 `## Workspace instruction files` 锚点，把用法插在它前面；找不到锚点就追加到末尾。这意味着用法段始终在 workspace 指令文件（如 `AGENTS.md`）之前，但在核心身份/模式契约之后。

## ToolUsage：工具用法住在工具里

### 接口定义

```go
// tools/types.go:204-215
// ToolUsage is an optional capability a ToolProvider may also implement to
// contribute group-level usage guidance to the system prompt ...
// The agent injects the returned text only when the same provider actually
// returns tools for the session, so the guidance shares that provider's gating
// and stays in lockstep with the tools that provider registers.
// available contains the complete registered tool set for this session;
// use available.Ref/Refs before naming cross-provider tools.
type ToolUsage interface {
    Usage(ctx context.Context, session SessionContext, available AvailableTools) string
}
```

这段注释是整个设计的灵魂，三句话讲清了三条规则：

1. **group-level**：用法是「这组工具怎么协同」的指南，不是单个工具的 description（那个在 `sdk.Tool.Description`）。
2. **lockstep**：用法与注册共享 gating，provider 没注册工具就不贡献用法。
3. **available.Ref/Refs**：要提别的工具名，必须先问 `available`，不能硬编码。

### AvailableTools：工具名引用的安全门

```go
// tools/internal/toolset/toolset.go:8-10
type Available struct {
    names map[toolname.Name]struct{}
}
// toolset.go:31-36
func (a Available) Ref(name toolname.Name) (string, bool) {
    if !a.Has(name) { return "", false }
    return "`" + name.String() + "`", true
}
```

`Available` 的 backing map 在 `toolset` 包外不可构造（`toolset.go:8` 注释明说：「so Usage code cannot forge availability by constructing or mutating the map directly」）。`Ref` 返回的是反引号包裹的工具名（`` `send` ``），不可用就返回空字符串。用法代码的典型模式：

```go
if sendRef, ok := available.Ref(ToolSend()); ok {
    parts = append(parts, "Use "+sendRef+" to ...")
}
```

这样即使 provider A 想引用 provider B 的工具，也只在 B 真的注册了那个工具时才会把名字写进 prompt。

### ToolName：不可外部构造的类型

`ToolName`（`tools/internal/toolname/toolname.go:11-13`）是一个值类型，但它的构造函数 `newName` 是包内私有，外部只能通过 `ToolRead()`/`ToolSend()` 等访问器拿到实例。这保证「工具名」不会被人随手 `toolname.Name{value: "fake"}` 伪造，所有合法工具名都在 `all` 数组里登记（`toolname.go:78-86`），共 41 个内置工具。

### sdk.Tool：工具自身的载体

```go
// twilight-ai/sdk/tool.go:58-67
type Tool struct {
    Name            string
    Description     string          // 工具自身的说明，进 tools 数组
    Parameters      any             // JSON schema
    Execute         ToolExecuteFunc
    RequireApproval bool
    CacheControl    *CacheControl   // Anthropic prompt caching
}
```

注意区分：`Description` 是「这个工具干什么」的单行说明，随工具数组一起发给模型；`ToolUsage` 是「这组工具怎么协同」的多行指南，拼进 system prompt。两者分工不同，前者是工具自描述，后者是组级编排。

## 代表工具剖析

Memoh 有 41 个内置工具，分属十几个 provider。这里挑 6 个有代表性的深读，其余在末尾列表标注。

### 1. ContainerProvider — 最基础的文件系统工具组

`tools/container.go`，提供 `read`/`write`/`list`/`edit`/`apply_patch`/`exec`。

**Usage（`container.go:67-92`）** 产出 `## Basic Tools` 段，逐个列出已注册工具的一句话说明。注意 `read` 的描述会根据 `SupportsImageInput` 动态变化：

```go
// container.go:69-75
if ref, ok := available.Ref(ToolRead()); ok {
    text := ref + ": read file content"
    if session.SupportsImageInput {
        text += " (also supports images: PNG, JPEG, GIF, WebP)"
    }
    parts = append(parts, text)
}
```

**Tools（`container.go:94+`）** 里 `read` 的 `Description` 同样按 `SupportsImageInput` 分支（`container.go:99-102`）——多模态模型会看到「Also supports reading image files ... binary images are loaded into model context automatically」，纯文本模型则看不到这句。这是「工具自描述也按 session 变化」的范例。

ContainerProvider 是少数不检查 `IsSubagent` 的 provider：子代理也需要读写文件。它的 gating 依据是 workspace 是否可解析（`resolveToolWorkspace`）。

### 2. MessageProvider — 按 SessionType 分支最丰富的工具

`tools/message.go`，提供 `send`/`react`。

**Usage（`message.go:34-62`）** 的核心是按 `session.SessionType` 给完全不同的用法：

- `Discuss` 模式：「Use `send` to speak in the observed conversation; if you do not call it, you stay silent.」（`message.go:39`）——discuss 模式下普通文本输出被禁，必须用 `send` 才发声。
- `Schedule`/`Heartbeat` 模式：「Use `send` only when the background task needs to notify a person or channel; specify `platform` and `target`.」（`message.go:41`）——后台任务必须显式指定目标。
- 默认（Chat）：根据 `CanOmitMessagingTarget()` 决定是否要求显式 `platform`/`target`。

**Tools（`message.go:64-121`）** 同样分支：subagent 直接返回 `nil`（`message.go:65-67`）；`send` 的 `description`、`platform`/`target` 是否 required，都由 `sendToolPromptMetadata`（`message.go:273-296`）按 session 算出。这意味着同一个 `send` 工具，在 chat 和 schedule 模式下发给模型的参数 schema 是不一样的——required 字段不同。

MessageProvider 还演示了「side-effect 流」：当 `send` 命中当前会话且 `Emitter` 可用时，附件/反应不走 channel sender，而是通过 `session.Emitter(ToolStreamEvent{...})`（`message.go:333`、`message.go:402`）直推 agent stream，落到当前 tool call block 上。

### 3. MemoryProvider — MCP 动态发现 + 工具名引用

`tools/memory.go`，提供 `search_memory`。

这个 provider 最特别的地方：**它的工具不是静态定义的，而是通过 MCP 动态发现**（`memory.go:58-81`）。`resolveProvider` 按 bot 设置找到 memory 后端，`ListTools` 拿到 MCP 工具描述符，再包装成 `sdk.Tool`。`search_memory` 这个名字甚至来自 `memprovider.ToolSearchMemory` 常量（`toolname.go:47`），不是硬编码字符串。

**Usage（`memory.go:38-47`）** 极简，但点出一个重要原则——「记忆与最新消息冲突时，以最新消息为准」：

```go
return usageSection("Long-term memory", []string{
    "Use " + ref + " to recall durable user preferences, prior conversations, ...",
    "When retrieved memory conflicts with the latest user message or visible context, treat the latest user message and current context as authoritative.",
})
```

这正是历史 bug 之一：旧版静态 `_memory.md` 里硬写了 `` `search_memory` ``，subagent 没有 memory 工具时 prompt 仍然提到它。现在 `available.Ref(ToolSearchMemory())` 拿不到引用就直接返回空字符串（`memory.go:39-42`），用法段整段消失。

### 4. SpawnProvider — 跨工具引用与子代理限制

`tools/subagent.go`，提供 `spawn_agent`/`send_message`/`list_agents`，以及背景任务控制工具。

**Usage（`subagent.go:220-251`）** 是跨工具引用的教科书：

```go
// subagent.go:235-240
if backgroundTools := available.Refs(ToolWaitUntil(), ToolGetBackgroundStatus(),
    ToolListBackground(), ToolKillBackground()); len(backgroundTools) > 0 {
    if canStartBackground {
        parts = append(parts, "For long work, set `run_in_background: true`. The call returns a task ID immediately. Use `wait_until(task_id)`, then `get_background_status(task_id)` to read `result`.")
    }
    parts = append(parts, "Manage running agent tasks with "+joinRefs(backgroundTools, "and")+".")
}
```

`available.Refs` 一次查多个工具，只返回存在的；`joinRefs`（`types.go:191-202`）把它们用「and/or」连成自然语言。这样无论背景工具是全注册、部分注册、还是都没注册，用法段都能自适应。

**Tools（`subagent.go:253-256`）** 开头一句 `if session.IsSubagent || p.agent == nil { return nil, nil }`——子代理不能再生子代理，防止递归爆炸。子代理的受限工具集在 Usage 里明确告知：「Each subagent has a restricted worker tool set: file tools, exec/background tools, web search, and web fetch.」（`subagent.go:227`）。

### 5. BrowserProvider — headed browser 与 Computer Use

`tools/browser.go`，提供 `browser_action`/`browser_observe`/`computer_observe`/`computer_action`/`browser_remote_session`。详见后文「Browser Use / Computer Use」专节。

### 6. AskUserProvider — RequireApproval 与 ACP 发现阶段

`tools/ask_user.go`，提供 `ask_user`。

**Usage（`ask_user.go:21-38`）** 产出 `## User Input` 段，核心规则是「用户要你出选择题/测验，必须调 `ask_user`，不能用普通文本模拟」。这段用法只有在 `canExposeAskUserTool(session)` 为真时才产出（`ask_user.go:22`）。

**Tools（`ask_user.go:40-117`）** 有两个特别点：

1. `RequireApproval: true`（`ask_user.go:105`）——这个工具默认需要审批。`markApprovalTools`（`agent.go:913-921`）会给 write/edit/apply_patch/exec 都打上 `RequireApproval`，而 `ask_user` 是自己主动要求的。
2. `canExposeAskUserTool`（`ask_user.go:119-128`）有一个 ACP 发现阶段豁口：ACP agent（如 Hermes）会在 prompt 激活前先发现 MCP 工具并缓存，所以 `CanListUserInput && SessionType==ACPAgent` 时也让它看到 `ask_user`，但执行仍要求 `CanRequestUserInput`，避免后台调用挂死。这个细节和 [[11_acp插件用户输入]] 紧密相关。

`ask_user` 的 Execute 本身（`ask_user.go:106-115`）只做参数校验，真正解析走 `NativeToolSource.callAskUser`（`native_source.go:219-283`）的 userinput flow，因为它需要挂起等待用户在 UI 上作答。

## Browser Use / Computer Use

### headed workspace display，不是 headless Playwright

这是 Memoh 工具系统里最容易误解的一块。先看常量（`browser.go:28-30`）：

```go
browserCDPAddress = "127.0.0.1:9222"
browserCDPBaseURL = "http://" + browserCDPAddress
```

这个 CDP 端点不是跑在宿主机上，而是跑在**工作空间容器内部**的虚拟桌面里。`BrowserProvider` 的依赖是 `bridge.Provider`（`browser.go:49`），所有操作都通过 `containers.MCPClient(ctx, botID)` 拿到 bridge client，再在容器内执行。`computer_a11y.go:50` 的命令把这个事实写明白了：

```go
cmd := fmt.Sprintf("DISPLAY=:99 %s %s", shellQuote(a11yCLIPath), shellQuoteArgs(args))
```

`DISPLAY=:99` 是 Xvfb 虚拟显示器，`a11y-cli` 在 `/opt/memoh/toolkit/display/bin/` 下。也就是说，工作空间容器里跑着一个完整的 headed Chrome（有 GUI 的 Chrome），模型拿到的是真实渲染的页面，能用 GUI 工具点按钮、填表单，甚至能操作 Chrome 之外的原生对话框、其他桌面应用。

这与 **headless Playwright** 是两条完全不同的路：

| 维度 | Memoh headed browser | headless Playwright |
|---|---|---|
| 浏览器形态 | 虚拟桌面上的 headed Chrome | 无 GUI 的 headless Chrome |
| 操作入口 | `browser_action`/`browser_observe`（GUI 工具）或 `browser_remote_session`（CDP escape hatch） | Playwright API（代码驱动） |
| 能否操作非浏览器 | 能，`computer_action` 可点桌面任意位置 | 不能，只能操作浏览器内 |
| 桌面 fallback | 有，原生对话框/非浏览器应用走 Computer 工具 | 无 |
| 适用场景 | 需要 GUI 交互、视觉判断、跨应用操作 | 纯代码自动化，更快但更脆 |

`browser_remote_session`（`browser.go:211-222`）是给「Playwright 或其他 CDP 客户端」的逃生口——它的 description 明说「Exposes the workspace Chrome CDP endpoint for `chromium.connectOverCDP` or other CDP clients」，返回 `cdp_url`/`connect_over_cdp`（`browser.go:307-321`）。Usage 里对它的定位是「Only when running Playwright or other CDP automation inside the workspace is clearly better than the GUI tools above」（`browser.go:102`）——先 GUI，实在不行才上代码驱动。

### Computer Use：a11y 优先，坐标兜底

`computer_observe`/`computer_action` 是「整个桌面」的回退。Usage（`browser.go:92-98`）讲清了优先级：

> **Computer**: Whole-desktop fallback for native dialogs, non-browser apps, or when the browser path fails. Start with a snapshot to get an accessibility tree with element refs, then drive actions with those refs; raw coordinates are a last-resort fallback.

`computer_a11y.go` 的 `computerA11ySnapshot`（`computer_a11y.go:72`）调 `a11y-cli snapshot`，返回带 `ref`/`role`/`name`/`center` 的无障碍树。`computerA11yClick`/`computerA11yEdit`（`computer_a11y.go:90-100`）用 `--ref` 定位元素，而不是裸坐标。只有原生对话框、拖拽、指针悬停这类无 a11y 目标的场景，才退化到 `(x, y)` 坐标（`browser.go:195-196` 的 `x`/`y` 参数）。

这套设计与 Anthropic 的 Computer Use 范式一致：优先用语义化的 accessibility tree，坐标是最后手段，因为坐标会随分辨率/窗口位置漂移，而 ref 不会。

### 截图不自动进对话

一个反直觉的细节（`browser.go:107-116`）：observe 工具的截图存到工作空间路径，**不会**自动作为附件发到对话。模型要看得用 `read` 工具读那个路径（`browser.go:111-113`），且只有 `SupportsImageInput` 为真时才提示这条路径。这避免了每次 observe 都往上下文塞一张大图，把 token 撑爆。

## prompt_test.go 守卫测试

这组测试是「工具用法住在工具里」这条铁律的执行者。它们不测功能正确性，只测一件事：**system prompt 不得静态提及任何条件工具**。

### 三个守卫

**1. `TestGenerateSystemPromptOmitsToolSpecificMemorySearchGuidance`（`prompt_test.go:207-227`）**

最具体的守卫，直接针对 `search_memory` 这个历史 bug。遍历所有 session 类型，断言 prompt 不包含 `` `search_memory` `` 反引号引用，也不包含 `{{memorySearchSection}}` 占位符——后者是旧模板渲染失败时会泄漏的原文。

**2. `TestGenerateSystemPromptDoesNotReintroduceStaticToolSections`（`prompt_test.go:232-259`）**

针对「静态段落回归」的守卫。断言 prompt 不得包含这些旧段落标题：`## Basic Tools`、`## Contacts & Messaging`、`## Sessions & History`、`## Schedule Tasks`、`## Subagents`、`## User Input`。这些正是 `c78c3bee` 删掉的五个静态片段 + `_tools.md` 里的 `## Basic Tools`/`## User Input`。注意：这些标题现在会出现在 `ToolUsage` 注入的 `## Tool usage` 子段里（如 `usageSection("Basic Tools", ...)`），但那是动态生成的，守卫只检查「顶层静态 prompt」不含它们。

**3. `TestGenerateSystemPromptDoesNotEnumerateConditionalTools`（`prompt_test.go:261-287`）**

最严格的守卫。遍历全部 41 个 `BuiltInToolNames()`，对每个工具名调用 `promptEnumeratesTool` 检测。违规报错信息一针见血：

```
system prompt for %s must not statically enumerate conditional tool %q;
put its usage in ToolUsage gated by registration
```

### 检测函数 promptEnumeratesTool

`prompt_test.go:289-311` 定义了「枚举工具」的判定规则，按严格度递增：

1. 反引号包裹：`` `send` `` ——最直接的提及
2. 函数调用语法：`send(` ——模型可能模仿调用
3. 下划线名作为裸词：`search_memory` 出现在句中
4. 语义提及：`read`/`write` 还额外检测「read and write files」「reading/writing files」这类自然语言表述（`prompt_test.go:300-309`），因为旧 prompt 常用这种措辞暗示文件工具存在
5. 动词搭配：「call/use/invoke tool X」「X tool/capability」

`TestPromptEnumeratesToolDetectsCallSyntax`（`prompt_test.go:313-336`）是这组规则的回归测试，确保检测器本身能识别 `spawn({ tasks: [] })`、`read("/tmp/image.png")`、`you can read and write files there freely` 等各种变体。

### 历史 bug 复盘

结合 `c78c3bee` 删除的静态片段，可以还原三个典型 bug：

- **speak bug**：旧 `_tools.md` 等片段静态提及 `speak`，但 bot 没配 TTS（`TtsModelID` 为空）时 `TTSProvider.Tools` 返回 nil（`tts.go:95-97`），模型却看到用法指南，于是反复尝试调用不存在的 `speak`。
- **search_memory bug**：旧 `_memory.md` 静态写「Use `search_memory` to recall ...」，但 subagent session 下 `MemoryProvider.Tools` 返回 nil（`memory.go:50-51`），子代理模型仍被告知有这个工具。
- **schedule bug**：旧 `_schedule_task.md` 静态写「Use `schedule` to create a new task」，subagent 同样没有 schedule 工具（`schedule.go:70`）。

守卫测试的存在，让任何人想「图省事」把工具说明写回静态 prompt 时，CI 立刻拦截。这是用测试固化架构决策的典范。

## 设计动机与取舍

### 为什么不把用法塞进 sdk.Tool.Description？

`Description` 是单工具自描述，随工具数组发给模型。理论上可以把「这组工具怎么协同」也塞进每个工具的 Description，但有几个问题：

1. **跨工具引用无法保证一致性**。`send` 的 Description 想提 `react`，但 `react` 可能没注册；硬编码就会重现静态 prompt 的 bug。
2. **Description 会被 prompt caching 缓存**。`sdk.Tool.CacheControl`（`tool.go:66`）按工具粒度缓存，如果 Description 里塞了依赖 session 的动态内容，缓存命中率会暴跌。
3. **组级指南天然属于 system prompt**，不属于某个工具。把它放进 `## Tool usage` 段，语义清晰，时序也清晰（在 workspace 指令之前）。

### 为什么 Usage 要等所有工具收集完再统一调用？

`assembleTools` 的两阶段：先收集全部工具（`agent.go:864-880`），再统一调 `Usage`（`agent.go:885-889`）。如果边收集边调，`MemoryProvider.Usage` 拿到的 `available` 就看不到排在它后面的 `SpawnProvider` 的工具，跨引用会失真。两阶段保证了 `available` 是完整快照。

### 为什么 Available 的 backing map 不可外部构造？

`toolset.go:8` 注释：「so Usage code cannot forge availability」。如果 provider 能自己 `toolset.New([]Name{ToolSend()})` 造一个假的 available，`available.Ref(ToolSend())` 就会返回真，用法段就会提到一个可能根本没注册的工具。把构造函数藏起来，强制走 `NewAvailableTools(allTools)`（`types.go:166`），available 永远是真实注册集的镜像。

### 取舍：用法段长度

用法段是注入 system prompt 的，每个 token 都进上下文。Memoh 的做法是每个 provider 的 `Usage` 都用 `available.Ref` 做条件拼接，只提已注册的工具，且每条尽量一行。`usageSection`（`types.go:177-189`）会过滤空条目。这是「够用即止」——既不像静态 prompt 那样冗余，也不至于让模型猜不到工具怎么配合。

## 验证问题

1. **复述 prompt_test.go 守卫测试解决了哪三个历史 bug，以及守卫的具体判定规则。**
   答：speak bug（bot 没配 TTS 但 prompt 提及 speak）、search_memory bug（subagent 没 memory 工具但 prompt 提及）、schedule bug（subagent 没 schedule 工具但 prompt 提及）。守卫 `TestGenerateSystemPromptDoesNotEnumerateConditionalTools` 遍历全部 41 个内置工具名，用 `promptEnumeratesTool`（`prompt_test.go:289`）检测反引号引用、函数调用语法、下划线裸词、以及「call/use/invoke tool X」等语义提及；`read`/`write` 还额外检测「read and write files」自然语言表述。违规报错明确指向「put its usage in ToolUsage gated by registration」。

2. **说清 headed browser CDP 与 headless Playwright 的区别，以及 Memoh 为什么选前者。**
   答：headless Playwright 是无 GUI 的浏览器自动化，只能操作浏览器内、靠 API 驱动、遇到原生对话框就束手无策。Memoh 的 browser 跑在工作空间容器的虚拟桌面（`DISPLAY=:99`）上，是 headed Chrome，模型能用 GUI 工具（`browser_action`/`computer_action`）点页面、填表单，也能操作 Chrome 之外的原生对话框和桌面应用（`computer_a11y.go` 的 a11y-cli）。`browser_remote_session` 是给 Playwright 等 CDP 客户端的逃生口（`browser.go:211`），但定位是「GUI 工具搞不定时才用」。选前者是因为 Memoh 定位是通用 agent，要能处理任意桌面任务，不只是网页抓取。

3. **`assembleTools` 为什么「先收集全部工具，再统一调 Usage」两阶段？如果合并成一阶段会出什么问题？**
   答：为了保证 `Usage` 拿到的 `available` 是完整注册集。合并成一阶段，排在前面的 provider 调 `Usage` 时看不到后面 provider 的工具，跨工具引用（如 `ContactsProvider.Usage` 引用 `send`/`speak`）会失真，提到一个实际已注册但当时还没收集到的工具时返回空，用法段就漏了协同指南。两阶段保证 `available` 是真实快照。

4. **`ask_user` 工具为什么 `RequireApproval: true`，且 Execute 本身只做参数校验？真正的解析在哪？**
   答：`ask_user` 需要挂起等待用户在 UI 上作答，这是异步交互，不能在普通 `Execute` 里阻塞。`RequireApproval: true`（`ask_user.go:105`）让它走审批通道；Execute 只校验参数（`ask_user.go:106-115`），真正解析在 `NativeToolSource.callAskUser`（`native_source.go:219-283`）走 `userinput.RunFlow`，通过 `emitUserInputRequest` 把问题推到 live prompt stream 的 tool call block 上，然后 `WaitForRegisteredResponse` 等用户作答。这个设计与 [[11_acp插件用户输入]] 的 ACP/MCP 流程一致。

5. **子代理（subagent）能用到哪些工具？这个限制在代码里如何统一实现？**
   答：子代理有受限工具集：文件工具（read/write/list/edit/apply_patch）、exec/background、web_search、web_fetch（见 `subagent.go:227`）。限制的统一实现是几乎所有 provider 的 `Tools()` 开头都有 `if session.IsSubagent { return nil, nil }`：`message.go:65`、`memory.go:50`、`contacts.go:56`、`schedule.go:70`、`browser.go:126`、`ask_user.go:41`、`subagent.go:254`、`tts.go:84`。`ContainerProvider` 是少数例外——子代理也要读写文件。这种「按 provider 各自 gating」而非「全局黑名单」的方式，让每个 provider 独立决定自己的可见性，新增 provider 时只需在 `Tools()` 里加一行判断。
