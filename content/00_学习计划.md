# Memoh 学习计划

> 全中文。目标：把 Memoh 作为一个**完整的 Agent 平台**吃透——不只是「能跑」，而是搞懂「一个多成员、长记忆、可执行工具的 Agent 系统是怎么从零搭起来的」。
>
> 学习材料按**由内到外、由核心到外围**组织：先 Agent 内核，再记忆/工具/编排，最后容器/渠道/桌面端。每篇笔记指回真实文件 + 行号，学完能自己复述设计动机。

## 这是什么项目

Memoh 是一个**多成员、结构化长期记忆的 AI Agent 平台**，每个 Bot 可挂独立容器工作空间，能编辑文件、执行命令、跑工具、甚至「构建自己」。支持 Telegram / Discord / Lark / DingTalk / WeChat / Matrix / Email 等多渠道接入。

部署形态两条线：
- **Server 模式**：Go 后端（Echo:8080）+ Vue 前端（Vite:8082）+ 外部 PostgreSQL + 外部 Qdrant
- **Desktop 模式**：Electron 客户端在 `127.0.0.1:18731` 拉起本地 server，自带 SQLite + 内嵌 Qdrant + 打包 CLI

技术栈核心：Go + Uber FX（DI）+ Twilight AI SDK（LLM）+ sqlc（SQL→Go）+ MCP go-sdk + Docker/containerd/Apple Virtualization（工作空间）。

## 学习价值定位

Memoh 比 `ai-projects/` 里的 MCP 单点项目大一个量级，是**端到端 Agent 系统**。学它的价值不在单点技巧，而在「子系统如何咬合」：

| 学什么 | 为什么值得学 |
|---|---|
| Agent 内核 + 工具系统 | 看一个**进程内 Agent**（非独立网关）如何用 ToolProvider 接口组织工具，且「工具用法住在工具里，不进静态 prompt」 |
| 长期记忆 | 多 provider 混合检索：Qdrant 向量 + BM25 稀疏 + LLM 抽取，是生产级 RAG 范本 |
| 对话流编排 | flow resolver + pipeline 双路径，看消息组装/记忆注入/历史裁剪如何解耦 |
| 容器工作空间 | gRPC over UDS 的 bridge 架构，多后端（Docker/containerd/Apple）抽象 |
| Prompt 工程 | 嵌入式 Markdown 模板 + 模式切换（chat/discuss/heartbeat/schedule/subagent） |
| 数据库双后端 | PostgreSQL + SQLite 双轨迁移 + sqlc 生成，工程纪律样本 |
| 桌面端 | Electron 管理本地 server 生命周期 + 内嵌 Qdrant + 打包 CLI |

## 子系统学习清单

按优先级和依赖关系排序。建议按编号顺序学，每学完一个子系统写一篇笔记（`01_xxx.md` …）。

### P0 — Agent 内核（必学）

- [ ] **`internal/agent/agent.go`** — `Stream()` / `Generate()` 入口，Twilight SDK 如何驱动一轮对话 → 验证：能画出一次 Stream 调用的数据流
- [ ] **`internal/agent/stream.go`** — 流式事件组装 → 验证：说清 SSE 事件如何拼回完整输出
- [ ] **`internal/agent/prompt.go`** — Prompt 组装（system / heartbeat / schedule / subagent / discuss）→ 验证：列出 prompt 的可组合片段
- [ ] **`internal/agent/sential.go`** — 循环检测（文本 + 工具循环）→ 验证：讲清阈值与触发逻辑
- [ ] **`internal/agent/guard_state.go`** — Guard 状态机 → 验证：状态在何时切换
- [ ] **`internal/agent/retry.go`** — 重试策略 → 验证：什么错误重试、什么不重试

### P0 — 工具系统（必学，核心设计）

- [ ] **重点理解**：为什么「工具用法住在工具里，不进静态 prompt」——`sdk.Tool.Description` + 可选 `tools.ToolUsage` 的 `Usage()` 方法，由 `assembleTools` 按需注入
- [ ] **`internal/agent/prompt_test.go`** — 守卫测试：prompt 模板不能命名条件注册的工具 → 验证：能复述这个设计解决了什么 bug（原 `speak`/`search_memory`/`schedule` bug）
- [ ] Browser Use / Computer Use（`browser.go` / `computer_a11y.go`）— 仅在 workspace display 开启时暴露 → 验证：说清 headed browser（CDP）与 headless Playwright 的区别

## 笔记约定

- **源码优先**：每条结论指回真实文件 + 行号（如 `internal/agent/agent.go:120`）。自己追问时跳源码。
- **先讲法后代码**：每篇笔记 1 分钟叙事开场（这个子系统解决什么问题、为什么这么设计），再贴支撑代码。
- **诚实范围**：没读的部分标「待学」，不臆测。
- **对照学习**：MCP 部分对照 `ai-projects/mcp/`；Agent 部分对照 Twilight AI SDK 文档。
- **每篇笔记配验证问题**：学完能脱稿回答验证问题，才算过。
