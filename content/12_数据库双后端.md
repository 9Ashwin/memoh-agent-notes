# 12 — 数据库双后端

> **1 分钟叙事**：Memoh 一套代码要同时跑在两种部署形态上——云端 server 模式多成员共享一个 PostgreSQL，桌面端单机模式嵌入式跑一个 SQLite。为了让业务层不关心底下是哪个库，项目用 `sqlc` 把 SQL 查询编译成两份强类型 Go 代码（`pgx/v5` 一份、`database/sql` 一份），再用一个 `store` 接口层把它们藏起来。迁移走「postgres 是权威源、sqlite 阶段性重新生成全量基线」的约定：postgres 累计到 0102 号迁移，sqlite 只有 0027 号，但两边最新语义对齐。代价是 schema 和 query 都得维护双份——这是「免 CGO + 单二进制 + 多成员」这个组合的必然取舍。

---

## 核心结论

1. **双轨制**：PostgreSQL（`pgx/v5`，连接池）面向 server；SQLite（`modernc.org/sqlite` 纯 Go 驱动 + `database/sql`）面向桌面端。选哪个由配置项 `database.driver` 决定（`internal/db/target.go:23-25`，`internal/config/config.go:175-182`）。
2. **sqlc 双份生成**：`sqlc.yaml` 声明两个 engine，共享相同 query 名，但输出到两个包 `internal/db/postgres/sqlc` 与 `internal/db/sqlite/sqlc`（`sqlc.yaml:2-23`）。`mise run sqlc-generate` 一条命令刷两份（`mise.toml:51-53`）。
3. **统一抽象在 store 层**：`internal/db/store/contracts.go` 定义 `BotStore`/`AccountStore` 等接口，两后端各自实现；事务签名统一成 `WithTx(pgx.Tx)`，sqlite 实现里忽略该参数（`internal/db/sqlite/store/queries.go:26`）。
4. **迁移不一一对应**：postgres 102 个迁移对（最新 `0102_acp_default_chat_runtime`），sqlite 27 个（最新 `0027_acp_default_chat_runtime`）。语义对齐、编号独立。
5. **query 必须双份**：占位符（`$1` vs `sqlc.arg`）、JSON 取值（`->>'k'` vs `json_extract`）、UUID 生成（`gen_random_uuid()` vs `randomblob()`）、类型标注（`::uuid`）这些方言差异无法用一份 SQL 表达。

## SQLite 0001_init：与 postgres 的差异

`db/sqlite/migrations/0001_init.up.sql` 第 2 行写明：`-- SQLite initial schema generated from the PostgreSQL baseline.`。它不是一次性的原始快照，而是**阶段性重新生成**的派生产物——证据是 sqlite 的 `bots` 表第 164 行已有 `acl_default_effect TEXT NOT NULL DEFAULT 'allow'`，而 postgres 0001 没有。说明 sqlite 0001 吸收了 postgres 后期某次增量迁移后才定格。

核心差异（逐条对照两份 schema）：

| 维度 | PostgreSQL | SQLite |
|---|---|---|
| 主键类型 | `UUID PRIMARY KEY DEFAULT gen_random_uuid()` (例 13) | `TEXT PRIMARY KEY`，INSERT 时手写 `lower(hex(randomblob(4)))...` 生成 UUID v4（见 sqlite query `acl.sql:74-78`） |
| 布尔 | `BOOLEAN NOT NULL DEFAULT true` | `INTEGER NOT NULL DEFAULT 1` |
| JSON | `JSONB NOT NULL DEFAULT '{}'::jsonb` | `TEXT NOT NULL DEFAULT '{}'` |
| 时间戳 | `TIMESTAMPTZ NOT NULL DEFAULT now()` | `TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP`（秒精度） |
| 大整数 | `BIGINT`（如 `received_at_ms`、`turn_message_seq`） | 多数降为 `INTEGER`（`turn_message_seq` 在 sqlite 611 行是 `INTEGER`），仅 `received_at_ms` 保留 `BIGINT`（496 行） |
| 枚举 | `CREATE TYPE user_role AS ENUM ('member','admin')` (5-9) | 直接 `role TEXT NOT NULL DEFAULT 'member'` (12)，无独立类型 |
| 数组 | `scopes_supported TEXT[] NOT NULL DEFAULT '{}'` (372) | `scopes_supported TEXT NOT NULL DEFAULT '{}'` (368)，退化成 JSON 文本 |
| 名字格式校验 | `name ~ '^[a-z0-9][a-z0-9-]{1,62}$'` 正则 (208) | `name GLOB '[a-z0-9]*' AND name NOT GLOB '*[^a-z0-9-]*' AND length(name) BETWEEN 2 AND 63` (205-209) |
| JSON 取值 | `metadata->>'conversation_name'` (acl.sql:49) | `json_extract(source_route.metadata, '$.conversation_name')` (sqlite acl.sql:48) |
| 跨表约束 | plpgsql 函数 + `CREATE TRIGGER ... EXECUTE FUNCTION` (721-748) | `BEFORE INSERT ... WHEN NOT EXISTS (...) BEGIN SELECT RAISE(ABORT, ...) END` (532-586, 757-815) |
| 复合外键 | `FOREIGN KEY (session_id, bot_id) REFERENCES bot_sessions(id, bot_id)` (593-594, 647-648) | 同样支持复合 FK (596-597, 685) |
| `NULLS NOT DISTINCT` | `UNIQUE NULLS NOT DISTINCT (...)` (247, 251) | SQLite 不支持该语法，靠业务层保证 |

**秒精度陷阱**：sqlite `bot_sessions.updated_at` 用 `CURRENT_TIMESTAMP` 只有秒精度，schema 注释（462-467 行）明确警告——`internal/db/sqlite/store/sessions_paged.go` 的 keyset 游标必须把边界时间戳截断到秒，否则同一秒内的行会被无限跳过或重复返回。这是 SQLite 双后端里最隐蔽的一致性坑。

## sqlc 生成流程（sqlc.yaml + queries 双份）

`sqlc.yaml` 声明两个独立 engine（第 2-23 行）：

```yaml
sql:
  - engine: "postgresql"
    schema: "db/postgres/migrations"
    queries: "db/postgres/queries"
    gen:
      go:
        package: "sqlc"
        out: "internal/db/postgres/sqlc"
        sql_package: "pgx/v5"           # 用 pgx 原生连接
        emit_json_tags: true
        overrides:
          - db_type: "user_role"
            go_type: "string"            # 枚举映射成 string
  - engine: "sqlite"
    schema: "db/sqlite/migrations"
    queries: "db/sqlite/queries"
    gen:
      go:
        package: "sqlc"
        out: "internal/db/sqlite/sqlc"
        sql_package: "database/sql"      # 用标准库
        emit_json_tags: true
```

执行入口是 `mise run sqlc-generate`（`mise.toml:51-53`，`run = "sqlc generate"`），它也是 `mise run setup` 的依赖之一（`mise.toml:449`）。生成产物落在两个独立包，**接口签名不同**（pgx 的 `DBTX` 与 `database/sql` 的 `DBTX` 不兼容），所以上层需要 store 层收口。

**query 为什么必须双份**——以 `acl.sql` 的 `CreateBotACLRule` 为例：

- postgres 版（`db/postgres/queries/acl.sql:64-93`）：`INSERT INTO bot_acl_rules (bot_id, ...) VALUES ($1, $2, sqlc.narg(description)::text, ...)`，靠 schema 的 `DEFAULT gen_random_uuid()` 生成主键，`RETURNING *`。
- sqlite 版（`db/sqlite/queries/acl.sql:64-94`）：`INSERT INTO bot_acl_rules (id, bot_id, ...) VALUES (lower(hex(randomblob(4))) || '-' || ..., sqlc.arg(bot_id), ...)`，**必须显式写 `id` 列并手搓 UUID v4**，`RETURNING` 列出全部字段。

再看 `sessions.sql` 的 `CreateSession`：
- postgres（前 30 行）用 `WITH input AS (SELECT sqlc.arg(bot_id)::uuid ...)` CTE 包装再 `INSERT ... SELECT`。
- sqlite（前 30 行）直接 `INSERT ... SELECT lower(hex(randomblob(4)))..., sqlc.arg(bot_id), ...`，并内联 `WHERE` 子句做 `default_head_turn_id` 的跨表校验（postgres 依赖触发器）。

占位符（`$1` vs `sqlc.arg`）、类型标注（`::uuid` vs 无）、UUID 生成、JSON 取值、`now()` vs `CURRENT_TIMESTAMP`——这些方言差异叠加起来，使得「一份 SQL 两端跑」不现实，query 双份是必然。

queries 目录两端文件名几乎一一对应（44 vs 44 个 `.sql`，目录列表一致），同名 query 的 `-- name:` 标签完全一致（如 `EvaluateBotACLRule :one`、`CreateSession :one`），保证生成出来的 Go 函数签名对齐，store 层才能用同一接口收口。

## driver 选择与切换逻辑

选择链路集中在 `internal/db/target.go`：

```go
const (
    DriverPostgres = "postgres"   // target.go:14
    DriverSQLite   = "sqlite"     // target.go:15
)

func DriverFromConfig(cfg config.Config) string {
    return strings.TrimSpace(strings.ToLower(cfg.Database.DriverOrDefault()))  // target.go:23-25
}
```

`DriverOrDefault`（`internal/config/config.go:175-182`）读 `database.driver` 配置项，空则回落到 `DefaultDatabaseDriver` 常量。从 `config_test.go:300/337/368` 看，测试覆盖了 sqlite 与 postgres 两种默认场景。

**连接路径分叉**在 `internal/db/db.go:12-21`：

```go
func Open(ctx context.Context, cfg config.Config) (*pgxpool.Pool, error) {
    switch driver := DriverFromConfig(cfg); driver {
    case DriverPostgres:
        return OpenPostgres(ctx, cfg.Postgres)   // db.go:15 → pgxpool.New (db.go:24)
    case DriverSQLite:
        return nil, nil                            // db.go:17 — pgx 路径直接放弃
    default:
        return nil, fmt.Errorf("unsupported database driver %q", driver)
    }
}
```

sqlite 不走 pgx，而是 `internal/db/sqlite.go:15-25` 的 `OpenSQLite` → `sql.Open(DriverSQLite, SQLiteFileDSN(cfg))`，驱动是 `modernc.org/sqlite`（sqlite.go:10 的空白导入，纯 Go，无 CGO）。DSN 拼接在 `target.go:38-56`，支持 `_journal_mode=WAL` 与 `_busy_timeout`，`sqlite.go:34-58` 再把这些参数转成 modernc 驱动认识的 `_pragma` 形式。

**迁移路径**在 `internal/db/migrate.go:144-163` 的 `newMigrateForTarget`：sqlite 分支用 `migratesqlite.WithInstance` 并显式设 `NoTxWrap: true`（153-155 行），注释（150-152 行）解释——多个 sqlite 迁移要在重建表时临时关 `foreign_keys`，而 `PRAGMA foreign_keys` 在外层事务里不生效，所以迁移文件自己管事务边界。postgres 分支走 `migrate.NewWithSourceInstance` 直接用 DSN。

**嵌入 FS 分叉**在 `internal/db/migrations_fs.go:10-19`：`MigrationsFSForConfig` 根据 driver 从同一 `embed.FS` 里取 `postgres/migrations` 或 `sqlite/migrations` 子树。

**store 层统一接口**：`internal/db/store/contracts.go` 定义 `AccountStore` (78)、`BotStore` (93)、`ModelStore` (111)、`BotACLStore` (257) 等接口；`internal/db/store/queries.go:413` 定义统一事务签名 `WithTx(tx pgx.Tx) Queries`。两后端各自实现：
- postgres：`internal/db/postgres/store/queries.go:26-30`，`WithTx(tx pgx.Tx)` 真正用 pgx 事务。
- sqlite：`internal/db/sqlite/store/queries.go:26`，`WithTx(_ pgx.Tx)` **显式忽略参数**，内部用 `q.store.db.BeginTx(ctx, nil)` 走 `database/sql` 事务（34 行）。

这是双后端统一的代价：接口签名里出现了一个对 sqlite 无意义的 `pgx.Tx` 参数，靠下划线吞掉。

## 「全量基线 + 增量 diff」约定

这是双后端最关键的工程约定，从迁移目录可见：

| | postgres | sqlite |
|---|---|---|
| 迁移对数 | 102（`*.up.sql` 计数） | 27 |
| 最新迁移 | `0102_acp_default_chat_runtime` | `0027_acp_default_chat_runtime` |
| 0002 迁移 | `channel_identity_avatar` | `container_workspace_backend` |
| 0003 迁移 | `route_conversation_type` | `acl_target_cleanup` |

两边 0001 之后**文件名与编号完全不同**，但最新迁移的语义功能对齐（`acp_default_chat_runtime`、`session_turn_graph`、`email_provider_user_scope` 都是近期同步点）。约定可概括为：

1. **postgres 是权威源**。新功能先在 postgres 写增量迁移（如 `0102_acp_default_chat_runtime.up.sql`）。
2. **sqlite 阶段性重新生成 0001 基线**。当 postgres 的增量积累到一定程度，会把增量「折叠」进 sqlite 的 `0001_init.up.sql`，使其成为新的全量快照（这就是为什么 sqlite 0001 里已有 `acl_default_effect`，而 postgres 0001 没有）。
3. **sqlite 的增量迁移独立编号**。sqlite 的 `0002_container_workspace_backend`、`0003_acl_target_cleanup` 这些是 sqlite 特有的演化（如重建表、清理旧数据），与 postgres 的 `0002_channel_identity_avatar` 没有对应关系。
4. **语义对齐靠人工保证**。两端最新迁移同名（`acp_default_chat_runtime`），但 postgres 是 0102、sqlite 是 0027，编号只对齐「最终状态」不对齐「路径」。

这套约定让 sqlite 端不必跟 postgres 的 102 步历史走一遍（很多 postgres 增量在 sqlite 全量基线里已自然包含），代价是 sqlite 的 0001 是「会变的基线」——每次重新生成后，旧 sqlite 实例的迁移历史可能与新基线对不上（需要 `force` 到某个版本或重建库）。

## 设计动机与取舍

**为什么双后端**：Memoh 有两种部署形态。server 模式（多成员、云上）需要并发写、丰富类型、JSONB 查询能力，PostgreSQL 是自然选择；desktop 模式（单机、嵌入式）要求零运维、单文件、易分发，SQLite 是唯一务实选项。两者都要支持同一套业务逻辑，所以有了双轨。

**为什么选 modernc.org/sqlite 而不是 mattn/go-sqlite3**：`modernc.org/sqlite` 是纯 Go 实现（`internal/db/sqlite.go:10` 注释明确写 `Register the pure-Go SQLite database driver`），免 CGO。桌面端要交叉编译到 macOS/Windows/Linux 多架构，CGO 会让构建链复杂化。代价是性能略低于 cgo 版本，但桌面端单用户场景可接受。

**为什么用 sqlc 而不是 ORM**：sqlc 把 SQL 编译成强类型 Go 代码，编译期捕获列名/类型错误，且生成代码可读、可调试。双后端场景下，两份生成代码各自贴合方言，比 ORM 的「方言抽象层」更透明。代价是 query 必须双份。

**为什么不用 schema diff 工具自动派生 sqlite**：从 sqlite 0001 注释看，派生是人工/脚本辅助的。自动 diff 工具（如 `pgdump` 转 sqlite）处理不了触发器、枚举、`NULLS NOT DISTINCT` 这些差异，且生成的 SQL 往往不可读。人工维护双份 schema 虽然笨重，但可控、可审。

**代价清单**：
- schema 双份（postgres 1065 行 + sqlite 1129 行的 0001，外加各自增量）。
- query 双份（44 个 `.sql` 文件 × 2，同名同语义但方言不同）。
- 生成代码双份（两个 `sqlc` 包，接口签名不兼容）。
- store 层多一层抽象（`contracts.go` + 两份实现）。
- 迁移历史不对齐（postgres 102 步 vs sqlite 27 步），排错时要先判断是哪端的迁移。
- sqlite 秒精度时间戳带来的游标一致性坑（`sessions_paged.go` 必须截断）。

## 验证问题

1. **说清双后端迁移的「全量基线 + 增量 diff」约定**：postgres 0001 是真正的初始基线，sqlite 0001 是从 postgres 派生、并阶段性重新生成的快照（吸收 postgres 增量后定格）。postgres 后续 101 个增量迁移在 sqlite 端不一一对应——sqlite 只保留自己需要的增量（如表重建、数据清理），最新语义靠同名迁移人工对齐（如 `0102` vs `0027` `acp_default_chat_runtime`）。验证方式：对比两端 `0001_init.up.sql` 的 `bots` 表，sqlite 多出 `acl_default_effect` 字段（sqlite:164），证明 sqlite 0001 晚于 postgres 某次 acl 相关增量。

2. **为何 query 也要双份，不能复用一份 SQL**：占位符（`$1` vs `sqlc.arg`）、类型标注（`::uuid`/`::text`）、UUID 生成（`gen_random_uuid()` vs `randomblob()`）、JSON 取值（`->>` vs `json_extract`）、时间函数（`now()` vs `CURRENT_TIMESTAMP`）这些方言差异无法用一份 SQL 同时满足两个 engine。sqlc 也要求每个 engine 单独配置 `queries` 目录。验证方式：对比 `db/postgres/queries/acl.sql:64-93` 与 `db/sqlite/queries/acl.sql:64-94` 的 `CreateBotACLRule`。

3. **driver 切换在代码里如何落地**：`config.Database.DriverOrDefault()`（`config.go:175-182`）→ `DriverFromConfig`（`target.go:23-25`）→ `db.Open`/`MigrationTargetFromConfig` 的 switch。postgres 走 `pgxpool.New`，sqlite 走 `sql.Open(modernc.org/sqlite)`，迁移走 `migrate.go:144-163` 的 `newMigrateForTarget`（sqlite 显式 `NoTxWrap: true`）。

4. **sqlite 的 `updated_at` 秒精度会引发什么问题**：`bot_sessions.updated_at` 在 sqlite 是 `CURRENT_TIMESTAMP`（秒精度），keyset 分页游标若不截断边界时间戳到秒，同一秒内的行会因为 `updated_at` 相等而被游标条件漏掉或重复返回。`internal/db/sqlite/store/sessions_paged.go` 必须显式截断，schema 注释（sqlite:462-468）已留下警告。

5. **store 层如何统一 `pgx.Tx` 与 `database/sql` 的事务**：`store/queries.go:413` 定义 `WithTx(tx pgx.Tx) Queries`，sqlite 实现（`sqlite/store/queries.go:26`）用 `_ pgx.Tx` 显式忽略参数，内部改用 `q.store.db.BeginTx(ctx, nil)`。这是接口对齐的妥协——sqlite 实现里有一个永远不用的参数。

- `sessions_paged.go` 的 keyset 游标具体如何截断秒精度，是否有对应的回归测试。
