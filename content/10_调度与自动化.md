# 10 — 调度与自动化

> **1 分钟叙事**：一个 Agent 平台如果只能被动等用户发消息，就只是个高级聊天框。Memoh 给每个 Bot 配了两条"自己动起来"的路径：**heartbeat** 是 Bot 的定期自检——每隔 N 分钟自己醒一次，看看有没有事要报，没事就回一句 `HEARTBEAT_OK` 继续睡；**schedule** 是用户预设的定时任务——用 cron 表达式约定"每天 9 点跑这条 command"。两者都走 cron 调度，都创建独立 session、都签发短期 JWT 回调 Resolver、都落执行日志，但语义完全不同：heartbeat 是"被动巡检、低噪声、三态"，schedule 是"主动执行、可限次、二态"。这套设计让 Bot 在没人盯的时候也能持续工作，同时把"巡检"和"执行"这两种心智模型彻底分开，不混在一个抽象里。

## 核心结论

1. **两条路径，两个 service，两张日志表，两种 session 模式**——heartbeat 与 schedule 在代码、数据、prompt 三层都保持分离，只在最末端的 `Resolver` 处汇合（`internal/conversation/flow/resolver_trigger.go`）。
2. **heartbeat 用 `@every %dm` 固定间隔**（`internal/heartbeat/service.go:245`），每 Bot 一个间隔，存在 `bots.heartbeat_interval`；**schedule 用完整 cron 表达式**（`internal/schedule/service.go:504`），支持秒级可选、Descriptor、且按 Bot 时区求 `Next()`。
3. **执行超时统一 5 分钟**（`heartbeatRunTimeout` / `scheduleRunTimeout`），JWT token 统一 10 分钟 TTL，避免长期凭证外泄。
4. **schedule 有 `max_calls` 自动停用**，且停用逻辑写在 SQL 的 `CASE WHEN` 里（`db/postgres/queries/schedule.sql` 经 `IncrementScheduleCalls`），原子防竞态；heartbeat 无此机制——它本就该一直跑。
5. **状态空间不同**：heartbeat log 三态 `ok/alert/error`（`0001_init.up.sql:890`），schedule log 二态 `ok/error`（`0001_init.up.sql:924`）。`alert` 是"有异常但未失败"的中间态，对应 heartbeat 输出非 `HEARTBEAT_OK` 但也没崩的情况（`resolver_trigger.go:129-133`）。
6. **两者都复用主 Agent 内核**：`TriggerSchedule` / `TriggerHeartbeat` 最终都调 `r.agent.Generate(ctx, cfg)`（`resolver_trigger.go:58,124`），只是 `SessionType` 不同、注入的 user prompt 不同。

## heartbeat：cron 心跳调度

入口在 `internal/heartbeat/service.go`。用 `github.com/robfig/cron/v3`，`NewService` 时 `c.Start()` 直接拉起 cron 引擎（`service.go:58`）。

**调度模型**：固定间隔，不是 cron 表达式。

```go
// service.go:243-260
spec := fmt.Sprintf("@every %dm", cfg.Interval)
entryID, err := s.cron.AddFunc(spec, job)
```

`cfg.Interval` 来自 `bots.heartbeat_interval`，`normalizeHeartbeatInterval` 对 `<=0` 回退到 `defaultHeartbeatIntervalMinutes = 1440`（24 小时，`service.go:29`）。注意 0016 原始默认是 30 分钟，0086 把默认改成 1440——"低频自检、别吵"是后来调整的方向。

**生命周期三个入口**：

- `Bootstrap(ctx)`（`service.go:62`）：启动时 `ListHeartbeatEnabledBots` 一次性把所有 `heartbeat_enabled=true` 的 Bot 注册进 cron。
- `Reschedule(ctx, botID)`（`service.go:86`）：配置变更后重排。先 `removeJob`，再 `GetBotByID` 检查 `HeartbeatEnabled && Status=="ready"`（`service.go:97`），通过才重新 `scheduleJob`。注意这里只看 `ready` 状态——Bot 还在初始化时不会触发心跳。
- `Stop(botID)`（`service.go:108`）：纯移除。

**单次执行 `runHeartbeat`**（`service.go:112-176`）的六步：

1. `CreateSession(ctx, botID, "heartbeat")` 起一个独立 session（`service.go:127`）。
2. `ListHeartbeatLogsByBot limit 1` 取上次心跳时间，格式化成 ISO 8601 传给 prompt（`service.go:137-142`）。首次为空串，prompt 里会显示 `never (first heartbeat)`（`prompt.go:189`）。
3. `CreateHeartbeatLog` 先写一条"进行中"日志（`service.go:144`），`started_at` 默认 `now()`，`completed_at` 此时为空。
4. `generateTriggerToken(ownerUserID)` 签 10 分钟 TTL JWT，加 `Bearer ` 前缀（`service.go:232-241`）。
5. `triggerer.TriggerHeartbeat(...)` 调出去——桥接到 `Resolver.TriggerHeartbeat`（见下文）。
6. 无论成功失败都 `completeLog` 回填 `status/result_text/error_message/usage/model_id`（`service.go:174,155,168`）。失败路径先回填再 return，不会漏掉日志。

**执行上下文超时**：`scheduleJob` 里用 `context.WithTimeout(context.WithoutCancel(ctx), heartbeatRunTimeout)`（`service.go:247`），5 分钟硬上限。`WithoutCancel` 是关键——父 ctx 取消时心跳不应被牵连，但心跳自己必须有截止。

## schedule：定时任务

入口在 `internal/schedule/service.go`，比 heartbeat 重得多（16KB vs 8KB），因为它带完整 CRUD。

**调度模型**：完整 cron 表达式。

```go
// service.go:44
parser := cron.NewParser(cron.SecondOptional | cron.Minute | cron.Hour | cron.Dom | cron.Month | cron.Dow | cron.Descriptor)
```

支持秒级可选、`@daily` 之类 Descriptor。Create/Update 时都会 `s.parser.Parse(req.Pattern)` 校验，非法直接报错（`service.go:88,176`）。

**时区处理是 schedule 的硬需求**——`"0 9 * * *"` 在 UTC 和 Asia/Shanghai 完全不同。Memoh 的做法（`service.go:488-513, 569-616`）：

1. `resolveBotLocation` 读 `bots.timezone`（0045 加的列），`time.LoadLocation` 失败回退到 service 默认（UTC 或运行时配置）。
2. 用 `locationSchedule` 包装器包住 `cron.Schedule`，重写 `Next(t)` 为 `s.inner.Next(t.In(s.loc))`（`service.go:613`）。这样无论 cron 引擎全局 location 是什么，每次求下次触发时间都把基准点先 `In(bot时区)`。
3. service 自己也 `cron.WithLocation(location)` 设全局默认（`service.go:49`），双保险。

heartbeat 不需要这套——`@every 30m` 是相对时间，与时区无关。这是两条路径分離的一个技术理由。

**`max_calls` 自动停用**——`runSchedule` 第一步就 `IncrementScheduleCalls`（`service.go:254`），SQL 在 `internal/db/postgres/sqlc/schedule.sql.go:92-102`：

```sql
UPDATE schedule
SET current_calls = current_calls + 1,
    enabled = CASE
      WHEN max_calls IS NOT NULL AND current_calls + 1 >= max_calls THEN false
      ELSE enabled
    END
WHERE id = $1
RETURNING ...
```

原子操作，不会出现"两个并发触发都过了检查然后多跑一次"。返回的 `updated.Enabled` 若为 false，`runSchedule` 立即 `removeJob`（`service.go:258-260`）。`max_calls = NULL` 表示无限次。

**手动触发**：`Trigger(ctx, scheduleID)`（`service.go:230`）会检查 `Enabled` 再 `runSchedule`，给"立即试跑一次"的入口。它与 cron 自动触发的区别只是不经过 cron 引擎，执行路径完全相同。

**Update 后 reschedule**：`Update` 末尾必调 `rescheduleJob`（`service.go:212`）——先 `removeJob`，若仍 `Enabled` 再 `scheduleJob`。改 pattern / 改 command / 改时区都会重新挂载，不会留下旧 cron entry。

## 数据模型（bot_heartbeat_logs / schedule / schedule_logs）

> 权威源是 `db/postgres/migrations/0001_init.up.sql`（被增量 migration 持续同步为"当前态快照"，这是 Memoh 双后端的全量基线约定）。

### bots 表的 heartbeat 字段

`0001_init.up.sql:181-184`（基线已是最终态）：

| 字段 | 类型 | 默认 | 来源 migration |
|---|---|---|---|
| `heartbeat_enabled` | BOOLEAN | false | 0016 |
| `heartbeat_interval` | INTEGER | 1440 | 0016（原 30）→ 0086（改 1440） |
| `heartbeat_prompt` | TEXT | '' | 0016 |
| `heartbeat_model_id` | UUID FK models | NULL | 0017 |
| `timezone` | TEXT | NULL | 0045（schedule 也用） |

`heartbeat_model_id` 让心跳用便宜模型、主对话用强模型——成本控制点。

### bot_heartbeat_logs

`0001_init.up.sql:886-897`（0016 创建初版无 `session_id`，0038 补上）：

- `status TEXT CHECK IN ('ok', 'alert', 'error')` —— **三态**。
- `session_id UUID FK bot_sessions ON DELETE SET NULL` —— 软关联，session 删了日志还在。
- `usage JSONB` —— LLM 调用的 token 用量。
- `model_id UUID FK models` —— 实际用的模型（可能与 `heartbeat_model_id` 不同，若前者为空则 fallback 到主模型）。
- 索引 `idx_heartbeat_logs_bot_started (bot_id, started_at DESC)` —— 按 Bot 倒序查最近日志，覆盖 `ListHeartbeatLogsByBot`。

### schedule

`0001_init.up.sql:826-841`：

- `pattern TEXT` —— cron 表达式，由 service 层 parser 校验。
- `command TEXT NOT NULL` —— 要执行的指令，作为 user message 注入。
- `max_calls INTEGER`（nullable）—— NULL = 无限。
- `current_calls INTEGER NOT NULL DEFAULT 0` —— 由 `IncrementScheduleCalls` 原子递增。
- `enabled BOOLEAN NOT NULL DEFAULT true`。
- 两个索引：`idx_schedule_bot_id`、`idx_schedule_enabled`——后者支撑 `ListEnabledSchedules` 快速过滤要挂载的任务。

### schedule_logs

`0001_init.up.sql:919-931`（0038 重建，初版在 0001 里就有）：

- `status TEXT CHECK IN ('ok', 'error')` —— **二态**，没有 `alert`。
- 其余字段与 `bot_heartbeat_logs` 同构：`schedule_id` / `bot_id` / `session_id` / `result_text` / `error_message` / `usage` / `model_id` / `started_at` / `completed_at`。
- 两个索引：按 schedule、按 bot，分别支撑 `ListLogsBySchedule` 和 `ListLogsByBot` 两条查询路径。

### bot_sessions 的类型约束

`0001_init.up.sql:454-455`：

- `type TEXT CHECK IN ('chat', 'heartbeat', 'schedule', 'subagent', 'discuss', 'acp_agent')`
- `session_mode TEXT CHECK IN ('chat', 'discuss', 'heartbeat', 'schedule', 'subagent')`

heartbeat/schedule 触发时 `CreateSession(ctx, botID, "heartbeat"/"schedule")` 写入的 `type` 必须命中这个 CHECK——DB 层兜底，防止 service 层传错字符串。

## heartbeat 模式 vs schedule 模式

模式 prompt 在 `internal/agent/prompts/mode_heartbeat.md` 与 `mode_schedule.md`（详见 [[05_prompt工程与模式切换]]），这里只比对设计差异：

| 维度 | heartbeat 模式 | schedule 模式 |
|---|---|---|
| **心智** | 被动巡检："看看有没有事" | 主动执行："跑这条命令" |
| **触发内容** | 系统生成的心跳 prompt（含 `interval`/`now`/`lastHeartbeat`/`HEARTBEAT.md` checklist） | 用户预设的 `command` 字段 |
| **输出契约** | 没事输出 `HEARTBEAT_OK`；有事才通知 | 执行 command，需要时通知，否则静默完成 |
| **噪声容忍** | "Prefer low-noise behavior"、"Do not send routine status updates" | "complete the work silently and output a short log summary" |
| **状态映射** | 文本命中 `HEARTBEAT_OK` → `ok`；否则 → `alert`；崩溃 → `error`（`resolver_trigger.go:129-150`） | 执行成功 → `ok`；失败 → `error` |
| **数据模型** | `bot_heartbeat_logs` 三态、无 `max_calls` | `schedule_logs` 二态、有 `max_calls` 自动停用 |
| **调度方式** | `@every %dm` 固定间隔，无时区 | 完整 cron 表达式，按 Bot 时区 |
| **配置位置** | 字段在 `bots` 表（每 Bot 一个心跳） | 独立 `schedule` 表（每 Bot 多个任务） |
| **模型选择** | `heartbeat_model_id` 可独立配置 | 用 Bot 主模型 |

两者的 prompt 都注入 `{{mainAgentSections}}`（`mode_heartbeat.md:18`、`mode_schedule.md:12`），即主 Agent 的工具/记忆/身份片段是共享的——差异只在"本次会话的指令"和"输出契约"。

## 设计动机与取舍

**为什么不把 heartbeat 和 schedule 合并成一个"定时任务"抽象？** 三条理由从代码里读得出：

1. **语义不同导致状态空间不同**。heartbeat 是自检，需要"有异常但没崩"的 `alert` 中间态（`resolver_trigger.go:129` 的判定逻辑）；schedule 是任务执行，要么成要么败。合并会让 `schedule_logs` 多一个用不到的 `alert`，或在 heartbeat 里硬塞 `max_calls` 这种没意义的字段。

2. **调度模型不同导致配置位置不同**。heartbeat 是"每 Bot 一个间隔"，天然是 `bots` 表的一个字段；schedule 是"每 Bot 多个任务"，必须是独立表。强行合并要么把 heartbeat 拆成一行 `schedule` 记录（丢掉与 Bot 配置的原子性），要么把 schedule 塞进 `bots` 表（无法多任务）。

3. **时区处理的复杂度只属于 schedule**。`locationSchedule` 包装器（`service.go:601-615`）是有成本的抽象，heartbeat 用 `@every` 根本不需要。让 heartbeat 复用 schedule 的调度路径会引入不必要的时区逻辑。

**为什么用 cron 引擎而不是自己写 ticker？** `robfig/cron/v3` 提供了 `@every`、cron 表达式、Descriptor、动态 Add/Remove、`WithParser`/`WithLocation` 全套能力，自己写 ticker 处理"每月最后一天"这种 cron 语义是重复造轮子。Memoh 只在它之上加了 `locationSchedule` 这一层薄包装解决"per-bot 时区"。

**为什么 JWT 短期 token 而不是长期凭证？** 调度服务签发、Resolver 验证，10 分钟 TTL 覆盖一次执行（`heartbeatTokenTTL`/`scheduleTokenTTL`，`service.go:23`/`service.go:244`）。即使 token 泄露也只能在 10 分钟内冒用，且 `sub` 锁定到 Bot owner——ACL 层还能再拦。

**为什么 5 分钟执行超时？** 防止 `agent.Generate()` 无限挂起占满 cron worker（`heartbeatRunTimeout`/`scheduleRunTimeout`，`service.go:27`/`service.go:248`）。cron 引擎本身的 job 是同步执行的，一个挂死会阻塞后续。

**为什么 `IncrementScheduleCalls` 用 SQL CASE 而不是先查后改？** 并发安全。两个 cron 触发同时命中时，`SELECT then UPDATE` 会双超 `max_calls`；`UPDATE ... SET enabled = CASE WHEN current_calls+1 >= max_calls THEN false ...` 是单条原子语句，DB 层保证不会多跑。

## 验证问题

1. **说清 heartbeat 模式与 schedule 模式的区别。**（00 学习计划原题）
   - 心智：被动巡检 vs 主动执行；调度：`@every` 固定间隔 vs cron 表达式（按时区）；配置：`bots` 表字段 vs 独立 `schedule` 表；状态：三态 `ok/alert/error` vs 二态 `ok/error`；停用：无 vs `max_calls` 自动停；输出契约：`HEARTBEAT_OK` 静默 vs 执行 command。详见上文表格。

2. **说清各自数据模型。**
   - `bot_heartbeat_logs`：三态 status、`session_id` 软关联、`usage` JSONB、`model_id`，索引 `(bot_id, started_at DESC)`。无 `max_calls`。
   - `schedule`：`pattern`(cron) + `command` + `max_calls`(nullable) + `current_calls` + `enabled`，索引按 `bot_id` 和 `enabled`。
   - `schedule_logs`：二态 status、`schedule_id` + `bot_id` 双外键、其余同构。两个索引分别支撑按 schedule 和按 bot 查询。

3. **heartbeat 的 `alert` 状态怎么产生？schedule 为什么没有？**
   - `resolver_trigger.go:129-133`：`isHeartbeatOK(text)` 判定文本是否含 `HEARTBEAT_OK`，不含则 `status = "alert"`。这是"Agent 跑完了但输出不是 OK 信号"的中间态——既非崩溃也非正常。schedule 执行 command 没有"OK 信号"约定，成即 ok、败即 error，不需要中间态。

4. **schedule 如何处理"每月最后一天 9 点"这种 cron 语义在不同时区下的差异？**
   - `resolveBotLocation`（`service.go:572`）读 `bots.timezone`，`locationSchedule.Next(t)`（`service.go:613`）把基准点 `t.In(bot时区)` 后再求 Next。即使 service 全局 location 是 UTC，"9 点"也按 Bot 时区的 9 点算。

5. **`max_calls` 达到上限时，正在跑的这次会执行吗？cron entry 会被移除吗？**
   - 会执行。`runSchedule` 先 `IncrementScheduleCalls` 拿到 `updated`（`service.go:254`），这次调用本身已经 +1 并返回；若 `updated.Enabled == false` 才 `removeJob`（`service.go:259`）。所以"达到上限的那次"会跑完，跑完后 cron entry 被移除，不再触发下次。这是"包含式"的——`current_calls + 1 >= max_calls` 时仍执行本次。
